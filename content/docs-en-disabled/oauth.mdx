## Overview

This document traces the complete OAuth authentication flow for Google and Facebook providers in Hogwarts. It covers provider configuration, the custom multi-tenant Prisma adapter, account linking, callback URL preservation, JWT/session population, and cross-subdomain cookie setup.

For the overall authentication architecture, redirect logic, and entry points, see [Authentication](/docs/authentication). For email/password credential flows (registration, login, 2FA, verification, password reset), see [Credentials](/docs/credentials).

**Providers**: Google (OIDC) and Facebook (OAuth 2.0 only)

## Social Component

**Source**: `src/components/auth/social.tsx`

Client component rendering Google and Facebook buttons in a 2-column grid. The `onClick(provider)` handler has two branches:

### Tenant/Subdomain Context

When the user is on a subdomain (e.g., `demo.localhost:3000`) or has a `?tenant=` URL param:

1. Extracts subdomain from hostname or `tenant` search param
2. Stores tenant info in `sessionStorage` (`oauth_tenant`, `oauth_callback_url`)
3. Calls `signIn(provider, { callbackUrl: dashboardUrl?tenant=... })`

```typescript
// Subdomain detection
const isProdSubdomain =
  currentHost.endsWith(".databayt.org") && currentHost !== "ed.databayt.org"
const isDevSubdomain =
  currentHost.includes(".localhost") && currentHost !== "localhost"
```

### Main Domain

When the user is on the main domain (`ed.databayt.org` or `localhost:3000`):

1. Resolves `callbackUrl` from search params or falls back to `DEFAULT_LOGIN_REDIRECT`
2. Triple-stores callback URL before redirect:
   - **Server httpOnly cookie** via `POST /api/auth/store-callback` (most reliable)
   - **sessionStorage** key `oauth_callback_intended` (lost on domain change)
   - **Client cookie** `oauth_callback_intended` (15min TTL, `SameSite=Lax`)
3. Adds base64-encoded `state` param with `{ callbackUrl, timestamp }`
4. Calls `signIn(provider, { callbackUrl, redirect: true, state })`

### Facebook `#_=_` Cleanup

Facebook appends `#_=_` to the callback URL. The `cleanUrlHash()` function runs on component mount via `useEffect` to strip this:

```typescript
if (window.location.hash === "#_=_") {
  const cleanUrl = window.location.href.replace(/#.*$/, "")
  window.history.replaceState({}, document.title, cleanUrl)
}
```

This hash is also stripped server-side in the `redirect` callback.

## Provider Configuration

**Source**: `src/auth.config.ts`

### Google (OIDC)

```typescript
Google({
  clientId: env.GOOGLE_CLIENT_ID || "",
  clientSecret: env.GOOGLE_CLIENT_SECRET || "",
  authorization: {
    params: {
      prompt: "consent",
      access_type: "offline",
      response_type: "code",
    },
  },
  profile(profile) {
    return {
      id: profile.sub,
      username: profile.name,
      email: profile.email,
      image: profile.picture,
      emailVerified: new Date(), // Always marked verified
    }
  },
})
```

- Uses t3-env validated `env.GOOGLE_CLIENT_ID` and `env.GOOGLE_CLIENT_SECRET`
- Forces `prompt: "consent"` and `access_type: "offline"` for refresh tokens
- Custom `profile()` maps: `sub` to `id`, `name` to `username`, `picture` to `image`
- `emailVerified` always set to `new Date()` (Google emails are pre-verified)

### Facebook (OAuth 2.0)

```typescript
Facebook({
  clientId: process.env.FACEBOOK_CLIENT_ID!,
  clientSecret: process.env.FACEBOOK_CLIENT_SECRET!,
})
```

- Uses `process.env` directly (NOT t3-env `env` object ŌĆö inconsistency with Google)
- Minimal configuration ŌĆö no custom `profile()` callback
- OAuth 2.0 only (Facebook does not support OIDC)

## Multi-Tenant Prisma Adapter

**Source**: `src/lib/multi-tenant-prisma-adapter.ts`

### The Problem

The User model has `@@unique([email, schoolId])` instead of `@unique email`. The standard Prisma adapter uses `findUnique({ where: { email } })` which fails because `email` alone is not a unique field.

### The Solution

`MultiTenantPrismaAdapter` wraps `PrismaAdapter` and overrides 4 methods:

| Method                                              | Override Behavior                                                                                  |
| --------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| `getUserByEmail(email)`                             | `findFirst({ where: { email, schoolId: null } })` ŌĆö finds OAuth users not yet assigned to a school |
| `createUser(data)`                                  | Creates with explicit `schoolId: null`, maps `name` to `username`                                  |
| `getUserByAccount({ provider, providerAccountId })` | Lookups via Account's `@@unique([provider, providerAccountId])` with user include                  |
| `linkAccount(account)`                              | Creates Account record linking provider to User                                                    |

### Key Design Decisions

- OAuth users start with `schoolId: null` ŌĆö they get assigned to a school during onboarding
- The `username` field is used instead of `name` (Prisma schema uses `username`)
- All methods return `AdapterUser` format: `{ id, email, emailVerified, name, image }`

## Account Linking

### Account Model

```prisma
model Account {
  id                String  @id @default(cuid())
  userId            String
  type              String
  provider          String
  providerAccountId String
  refresh_token     String? @db.Text
  access_token      String? @db.Text
  expires_at        Int?
  token_type        String?
  scope             String?
  id_token          String? @db.Text
  session_state     String?
  user              User    @relation(fields: [userId], references: [id], onDelete: Cascade)
  @@unique([provider, providerAccountId])
}
```

### First OAuth Sign-In

1. Adapter calls `getUserByAccount({ provider, providerAccountId })` ŌĆö not found
2. Adapter calls `getUserByEmail(email)` ŌĆö checks for existing user with `schoolId: null`
3. **If user exists**: `linkAccount()` links the OAuth provider to the existing user
4. **If user does not exist**: `createUser()` creates a new user, then `linkAccount()`

### Repeat Sign-In

1. `getUserByAccount()` finds existing Account record
2. Returns associated User ŌĆö no new records created

### OAuthAccountNotLinked Error

When a user registered with email/password tries to sign in with OAuth using the same email, NextAuth returns `OAuthAccountNotLinked`. This is a security feature ŌĆö the user must sign in with their original provider first.

### Dead Code Note

`getOrCreateOAuthUser()` in `src/components/auth/user.ts` is a utility function that exists but is NOT used by the actual OAuth flow. The adapter handles all user creation and linking.

## Callback URL Preservation

### The Problem

OAuth redirects the user to Google/Facebook, then back to `/api/auth/callback/{provider}`. The original destination URL (e.g., `/dashboard?tab=settings`) is lost during this redirect chain.

### Triple-Layer Storage

Before initiating the OAuth redirect, `social.tsx` stores the callback URL in three places:

| Layer | Storage                                                         | Reliability | Survives OAuth Redirect? |
| ----- | --------------------------------------------------------------- | ----------- | ------------------------ |
| 1     | Server httpOnly cookie via `POST /api/auth/store-callback`      | Highest     | Yes                      |
| 2     | `sessionStorage` key `oauth_callback_intended`                  | Medium      | No (different origin)    |
| 3     | Client cookie `oauth_callback_intended` (15min, `SameSite=Lax`) | Medium      | Yes                      |

### Store-Callback API

**Source**: `src/app/api/auth/store-callback/route.ts`

Sets an httpOnly cookie with the callback URL:

```typescript
cookieStore.set({
  name: "oauth_callback_intended",
  value: callbackUrl,
  httpOnly: true,
  sameSite: "lax",
  secure: process.env.NODE_ENV === "production",
  maxAge: 900, // 15 minutes
  path: "/",
})
```

### Resolution

In `src/auth.ts`, `resolveCallbackUrl()` reads the callback URL back (first match wins):

1. Server cookie `oauth_callback_intended` (most reliable)
2. URL `searchParams` (`callbackUrl` or `redirect`)
3. Regex match for `callbackUrl=...` in URL string
4. `baseUrl` searchParams

## JWT & Session for OAuth Users

**Source**: `src/auth.ts` (callbacks section)

### signIn Callback

Always returns `true` (allows sign-in). Performs provider-specific checks:

- **Facebook**: Logs error if no email in profile (missing `email` scope)
- **Google**: Logs warning if `email_verified` is false

### jwt Callback

Three triggers with different behaviors:

**`trigger: "signIn"` (initial sign-in)**:

- Receives `AdapterUser` with `id`, `email`, `name` ŌĆö but NOT `role` or `schoolId`
- Sets `token.id`, `token.provider`, `token.providerAccountId`
- Only sets `token.role` / `token.schoolId` if they exist on the user object (they usually don't for OAuth)
- Forces session update by setting `token.iat`, `token.exp`, `token.sessionToken`, `token.updatedAt`, `token.hash`

**No trigger (subsequent requests)**:

- Detects `!token.schoolId && token.id` ŌĆö refreshes from database
- Picks up `role` and `schoolId` once the User record is updated (e.g., after onboarding)

**`trigger: "update"` (manual session refresh)**:

- Loads `schoolId` and `role` from database
- Used after onboarding creates a school and assigns the user

### session Callback

Copies token data into session:

```typescript
session.user.id = token.id
session.user.role = token.role // May be undefined initially
session.user.schoolId = token.schoolId // May be undefined initially
```

Supports preview-role cookies for role impersonation.

**Key implication**: The first request after OAuth login may have an incomplete session (no `role` or `schoolId`). The JWT callback auto-refreshes from the database on the next request.

## Redirect Logic (Post-OAuth)

**Source**: `src/auth.ts` (`redirect` callback)

The redirect callback handles post-authentication routing. Full details in [Authentication](/docs/authentication). Brief priority order:

1. **Resolve callback URL** ŌĆö cookie, URL params, regex, baseUrl params
2. **Strip Facebook `#_=_` hash**
3. **Honor explicit relative paths** from `signIn({ redirectTo })` (skip root `/`, locale roots `/en`/`/ar`)
4. **Subdomain detection** ŌĆö `school.databayt.org` or `school.localhost:3000` ŌåÆ `/{locale}/dashboard`
5. **Main domain handling** ŌĆö locale paths, `?tenant=` param, school logout paths
6. **Smart redirect** ŌĆö decode JWT ŌåÆ look up user's school ŌåÆ redirect to school subdomain
7. **Fallback** ŌĆö `{baseUrl}/dashboard`

### Helper Functions

| Function                                    | Purpose                                  |
| ------------------------------------------- | ---------------------------------------- |
| `resolveCallbackUrl(url, baseUrl)`          | Multi-source callback URL resolution     |
| `validateCallbackUrl(callbackUrl, baseUrl)` | Same-origin security check               |
| `detectSubdomainFromHost(host)`             | Extract subdomain (null for main domain) |
| `getSmartRedirectUrl(url, baseUrl)`         | JWT ŌåÆ school lookup ŌåÆ subdomain URL      |
| `constructSchoolUrl(subdomain, path)`       | Environment-aware URL builder            |
| `extractLocaleFromUrl(url)`                 | Extract `ar` or `en` (default: `ar`)     |

## Cookie Configuration (Cross-Subdomain SSO)

**Source**: `src/auth.ts` (`cookies` section)

All auth cookies are configured for cross-subdomain sharing:

| Cookie             | Name                        | httpOnly | maxAge | Purpose           |
| ------------------ | --------------------------- | -------- | ------ | ----------------- |
| PKCE Code Verifier | `authjs.pkce.code_verifier` | Yes      | 15min  | OAuth PKCE flow   |
| Session Token      | `authjs.session-token`      | Yes      | 24h    | JWT session       |
| Callback URL       | `authjs.callback-url`       | No       | ŌĆö      | NextAuth callback |
| CSRF Token         | `authjs.csrf-token`         | Yes      | ŌĆö      | CSRF protection   |
| State              | `authjs.state`              | Yes      | ŌĆö      | OAuth state param |
| Nonce              | `authjs.nonce`              | Yes      | ŌĆö      | OIDC nonce        |

### Domain Configuration

```typescript
domain: process.env.NODE_ENV === "production" ? ".databayt.org" : undefined
```

- **Production**: `domain: ".databayt.org"` enables SSO across `ed.databayt.org`, `school1.databayt.org`, etc.
- **Development**: `domain: undefined` ŌĆö cookies scoped to `localhost` only (known limitation)
- **All cookies**: `sameSite: "lax"` (required for OAuth redirects), `secure: true` in production only

`trustHost: true` is set at the NextAuth config level ŌĆö required for Vercel proxy environments.

## Error Handling

**Source**: `src/components/auth/error-card.tsx`

When OAuth fails, NextAuth redirects to `/error?error={code}`. The `ErrorCard` component displays a user-friendly message:

| Error Code              | Message                                                | Trigger                                      |
| ----------------------- | ------------------------------------------------------ | -------------------------------------------- |
| `Configuration`         | There is a problem with the server configuration.      | Missing/invalid provider config              |
| `AccessDenied`          | You do not have permission to sign in.                 | Provider denied access                       |
| `OAuthSignin`           | Could not initiate sign in with OAuth provider.        | Failed to start OAuth flow                   |
| `OAuthCallback`         | Error completing OAuth sign in.                        | Callback processing failed                   |
| `OAuthCreateAccount`    | Could not create OAuth user in database.               | DB error during user creation                |
| `OAuthAccountNotLinked` | This email is already associated with another account. | Credentials user tries OAuth with same email |
| `Callback`              | Something went wrong with the authentication callback. | Generic callback error                       |
| `default`               | An unexpected error occurred.                          | Any other error                              |
