## Overview

The Timetable module handles school schedule management end-to-end: choosing a timetable structure during onboarding, configuring working days and periods, assigning weekly slots, detecting conflicts, managing substitutions, AI-powered generation, PDF export, and role-specific views. All operations are tenant-scoped by `schoolId`.

## Role-Based Experience

The production entry point is `content-production.tsx` which wraps `RoleRouter` in a `SessionProvider`. The `RoleRouter` calls `getPersonalizedTimetable()` to determine `viewType` and routes to the appropriate view.

### Data Flow Per Role

| Role       | View Component | Data Function                                  | What They See                                         | Known Bugs                                               |
| ---------- | -------------- | ---------------------------------------------- | ----------------------------------------------------- | -------------------------------------------------------- |
| DEVELOPER  | `AdminView`    | `getTimetableByRoom` / `getTimetableByTeacher` | Room/Teacher grid, editable slots, slot editor        | No class/grade filter mode                               |
| ADMIN      | `AdminView`    | Same as DEVELOPER                              | Same as DEVELOPER                                     | Same                                                     |
| TEACHER    | `TeacherView`  | `getTimetableByTeacher`                        | Personal schedule, workload stats, current/next class | No export button (despite `canExport` permission)        |
| STUDENT    | `StudentView`  | `getTimetableByStudentGrade`                   | Grade-level all-subjects view, PDF export, print      | Works correctly                                          |
| GUARDIAN   | `GuardianView` | `getChildTimetable`                            | Child selector + per-child schedule                   | **Bug**: `findFirst` returns only 1 class enrollment     |
| ACCOUNTANT | `AdminView`    | Same as ADMIN                                  | Full admin view with edit UI                          | **Bug**: `editable={true}` despite read-only permissions |
| STAFF      | `AdminView`    | Same as ADMIN                                  | Full admin view with edit UI                          | **Bug**: Same as ACCOUNTANT                              |

### How Routing Works

1. `getPersonalizedTimetable()` reads the user's role from session (`actions.ts:2400`)
2. Role → `viewType` mapping in a switch statement (`actions.ts:2422`)
3. ACCOUNTANT and STAFF **fall through to default** — `viewType` stays initialized as `"admin"` (`actions.ts:2415`)
4. `RoleRouter` renders the matching view component (`role-router.tsx:132`)
5. `AdminView` always renders `SimpleGrid` with `editable={true}` (`admin-view.tsx:353`)

### ACCOUNTANT/STAFF Bug Detail

The `permissions-config.ts` correctly defines ACCOUNTANT and STAFF as read-only (no `edit`, `create`, `delete` permissions). But `getPersonalizedTimetable()` has no cases for these roles, so they default to `viewType = "admin"`, and `AdminView` hardcodes `editable={true}`. The slot editor dialog opens on click and allows saves — the permission check only happens server-side in `upsertTimetableSlot()`.

## Grade-Based Filtering

### Student Grade Resolution Chain

The student view resolves a student's timetable through enrollment records:

```
User (session)
  → Student (db.student.findFirst where userId)
    → StudentClass (db.studentClass.findFirst where studentId)
      → classId
        → Timetable (db.timetable.findMany where classId IN [...])
```

Grade name comes from a separate chain:

```
Student → StudentYearLevel → YearLevel.levelName
```

### Where Grade Filtering Works

- **`getTimetableByStudentGrade()`**: Correctly uses `StudentClass` bridge table with `findMany` to get all enrolled class IDs, then queries timetable with `classId: { in: classIds }`. This is the function used by `StudentView`.

- **`getPersonalizedTimetable()` STUDENT branch**: Uses `StudentClass.findFirst` — returns only the most recent enrollment. Works for the simple case but misses multi-class enrollments.

### Where Grade Filtering Is Broken

- **`getWeeklyTimetable()` STUDENT branch** (`actions.ts:1377-1383`): References `Student.classId` — a field that **does not exist** on the Student model. Students are enrolled via the `StudentClass` bridge table. This always returns `undefined`, so no `classId` filter is applied and the student sees **all slots for the school**.

- **`getWeeklyTimetable()` TEACHER branch** (`actions.ts:1367`): Uses `session.user.id` (User model ID) directly as `teacherId` (Teacher model ID). These are different entities — the correct approach is to look up the Teacher record by `userId` first, as `getPersonalizedTimetable()` does.

- **`getChildTimetable()`** (`actions.ts:2670`): Uses `findFirst` on `StudentClass` — returns only 1 class enrollment per child. In this system, `Class = course section` (not homeroom), so a student enrolled in 6 subjects has 6 class enrollments. The guardian only sees the schedule for the most recent one.

- **Admin view**: Has no class/grade filter mode at all — only room and teacher selectors. To view a specific class schedule, admins must use the orphaned `/timetable/by-class` route.

## Structure Catalog

Schools don't invent their timetable structure - the ministry does. The catalog provides curated, real-world structures auto-recommended during onboarding via a scoring system.

### Design Decisions

- **TypeScript constants** (not Prisma) - the catalog is small, static, and has no relational data.
- **Scoring system** - `getRecommendedStructures()` scores each structure against the school profile and returns ranked results.

## Applying a Structure

The `applyTimetableStructure()` server action creates Period records from a structure definition and updates the school's week configuration.

### What It Does

1. Validates admin access and tenant context.
2. Looks up the structure by slug (supports legacy template names via `LEGACY_TEMPLATE_MAP`).
3. Optionally deletes existing periods for the year (`replaceExisting: true`).
4. Creates `Period` records for each entry in the structure (class, break, and lunch periods).
5. Upserts `SchoolWeekConfig` with the structure's working days and lunch position.
6. Logs the action for audit trail.

### Usage

```typescript
// From timetable settings (replacing existing periods)
await applyTimetableStructure({
  yearId: "clx...",
  structureSlug: "sd-gov-default",
  replaceExisting: true,
})

// Legacy template names still work
await applyTimetableStructure({
  yearId: "clx...",
  structureSlug: "standard_8", // mapped to sd-gov-default
})
```

## AI-Powered Generation

The generate page (`generate/content.tsx`) provides automated timetable creation.

### Algorithm (`generate/algorithm.ts`)

A 3-phase approach:

1. **Greedy Assignment**: Fast initial placement satisfying hard constraints
2. **Constraint Satisfaction**: Iterative refinement to resolve violations
3. **Optimization**: Score-based improvements (minimize gaps, balance workloads)

### Configuration Options

- **Hard constraints**: Enforce teacher expertise, room capacity, lunch breaks, prevent back-to-back periods
- **Workload limits**: Max periods per day, per week, max consecutive periods
- **Optimization preferences**: Configurable via UI toggles

### UI Flow

1. Select a term
2. Toggle constraint settings
3. Set workload limits
4. Click "Generate" to preview results with iteration count and quality score
5. Review generated slots
6. Click "Apply" to write to database

### Algorithm Limitations

The 3-phase algorithm has known gaps:

- **`preventBackToBack` detected but not enforced**: The code at `algorithm.ts:698-705` detects consecutive period conflicts but comments `// For now, allow it but flag` — no violation is returned and the slot is placed anyway.
- **No lunch break enforcement**: The algorithm doesn't reserve lunch periods. A teacher can be assigned straight through lunch.
- **No multi-period blocks**: Double periods (e.g., 2-hour lab sessions) aren't supported. Each slot is a single period.
- **No rotating schedules**: A/B week or day rotation patterns aren't modeled.
- **O(n^2) optimization phase**: The optimization loop compares all slots pairwise. For large schools (500+ slots), this can be slow.

### Server Actions

| Action                       | Purpose                       |
| ---------------------------- | ----------------------------- |
| `generateTimetablePreview()` | Run algorithm, return preview |
| `applyGeneratedTimetable()`  | Write generated results to DB |

## Conflict Detection

The conflicts page (`conflicts/content.tsx`) provides a dedicated UI for finding and resolving scheduling conflicts.

### Conflict Types

- **Teacher conflict**: Same teacher assigned to two classes at the same time
- **Room conflict**: Same room assigned to two classes at the same time

### Detection

Server-side detection via `detectTimetableConflicts()` uses SQL `GROUP BY` + `HAVING` for efficient O(conflicts) performance. Client-side `detectConflicts()` in `utils.ts` provides O(n^2) detection for preview/validation.

### Resolution Workflow

1. Select a term and click "Scan"
2. View teacher and room conflicts with affected classes
3. Click "Suggest" to find free slots for resolution
4. Click "Apply" to pre-fill the Slot Editor with the suggestion
5. Save the slot to resolve the conflict

The `conflicts-drawer.tsx` component also provides a drawer overlay from the main timetable view.

## Integration Points

The timetable module should connect with other school modules but currently operates in isolation for most.

### Attendance (Not Connected)

The attendance module records daily/period attendance independently. There is no pre-population of attendance records from timetable slots — teachers must manually select classes and students each period. The timetable knows which teacher is in which room at which period, but this information isn't used to auto-populate attendance forms.

### Exams (Not Connected)

Exams are scheduled independently with their own date/time fields. There's no check against the timetable for conflicts — an exam can be scheduled during an existing timetable slot without warning.

### Schedule Exceptions (Partial)

The `ScheduleException` model and CRUD actions exist (`getScheduleExceptions`, `createScheduleException`, `updateScheduleException`, `deleteScheduleException`). However, timetable query functions (`getTimetableByRoom`, `getTimetableByTeacher`, `getWeeklyTimetable`, etc.) **do not check schedule exceptions**. A holiday or modified schedule won't affect what the grid displays.

### Substitutions (Partial)

The substitution workflow is complete: absence recording, substitute finder, assignment, confirmation/decline. However, the timetable **views don't reflect substitutions** — if Teacher A is absent and Teacher B is substituting, the grid still shows Teacher A's name.

## Troubleshooting

- **No terms listed**: Ensure academic years/terms exist for the school.
- **Missing periods**: Apply a structure via settings or define periods manually under the school year.
- **No recommended structures**: The school's country/schoolType may not match any catalog entries. All structures are still available under "Other options".
- **Empty suggestions**: The teacher/room may be fully occupied for selected days; try another day or adjust schedule settings.
- **Legacy templates**: Old `standard_8`/`standard_6`/`half_day` names still work via the legacy mapping.
- **Active term required**: The role router requires an active term matching the current date range. If no term is active, the view won't load data.
- **Guardian sees empty or partial schedule**: This is known issue #1 — `getChildTimetable()` uses `findFirst` and returns only 1 class enrollment. Until fixed, guardians will see an incomplete schedule.
- **ACCOUNTANT/STAFF can edit slots**: This is known issue #5 — these roles are routed to AdminView with `editable={true}`. The server action `upsertTimetableSlot()` has its own permission check, but the UI is misleading. Until the routing is fixed, edits from these roles will fail server-side.
- **Student sees all school slots via API**: If using `getWeeklyTimetable()` directly (e.g., via `/api/timetable`), the Student branch references a non-existent `Student.classId` field, so no class filter is applied. The production `StudentView` component uses `getTimetableByStudentGrade()` instead, which works correctly.
- **Teacher sees empty timetable via API**: If using `getWeeklyTimetable()` directly, the Teacher branch uses `user.id` as `teacherId`. Since these are different IDs (User vs Teacher model), no slots match. The production `TeacherView` component uses `getTimetableByTeacher()` with the correct teacher ID lookup.
