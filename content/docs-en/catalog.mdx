---
title: Catalog
description: Universal shared backbone — one set of subjects, chapters, and lessons referenced by all schools via bridge tables.
---

The catalog is the single source of truth for curriculum content. All schools reference the same subjects via bridge tables; teacher-contributed content (videos, books, exams) flows through SaaS review and enriches the catalog for everyone.

## Why a shared catalog

| Approach           | Data for 1,000 schools × ~240 subjects                         | Sharing                                 |
| ------------------ | -------------------------------------------------------------- | --------------------------------------- |
| Per-school clone   | 220k subjects, 800k chapters, 4M lessons                       | None                                    |
| **Shared catalog** | **~240 subjects, ~800 chapters, ~4k lessons** + bridge records | Every contribution enriches all schools |

Onboarding a school takes seconds — `setupCatalogForSchool()` reads `country` + `curriculum` and creates `SubjectSelection` bridge rows.

## Three layers

| Layer                 | Models                                                                         | Scope                                                                            |
| --------------------- | ------------------------------------------------------------------------------ | -------------------------------------------------------------------------------- |
| Global catalog        | `Subject`, `Chapter`, `Lesson`                                                 | No `schoolId`                                                                    |
| Content (contributed) | `Video`, `Attachment`, `Exam`, `Question`, `Material`, `Assignment`, `Book`    | No `schoolId` — attribution is the `contributedSchoolId` string (see note below) |
| School bridge         | `SubjectSelection`, `BookSelection`, `ContentOverride`, `InstructorPreference` | Carries `schoolId`                                                               |

> **`schoolId` on content.** Of the contributed models, only `Video` / `VideoPurchase` carry an
> _optional_ `schoolId` (for source attribution and free-for-my-school gating). The rest
> (`Exam`, `Question`, `Material`, `Assignment`, `Book`, `Textbook`, `Document`, `Image`,
> `ExamTemplate`) have **no** `schoolId` — who contributed them is recorded in the plain
> `contributedSchoolId` string field, not a tenant FK.

**FK naming** — two conventions live side by side. The hierarchy spine (`Chapter.subjectId`,
`Lesson.chapterId`) and `Exam` / `Quiz` use **bare** FK names (`subjectId` / `chapterId` /
`lessonId`). Content attachments (`Question`, `Material`, `Assignment`, `Book`, `ExamTemplate`)
use **`catalog`-prefixed** FKs (`catalogSubjectId` / `catalogChapterId` / `catalogLessonId`).
Exceptions: `Document` and `Textbook` break the content convention and use bare FKs.

**Name collisions** — a catalog model and its school-scoped counterpart are _different_ models. When you read `Exam`, check the table to know which one:

| Catalog (global)                          | School-scoped (per-tenant)              |
| ----------------------------------------- | --------------------------------------- |
| `Exam` → `catalog_exams`                  | `SchoolExam` → `exams`                  |
| `Book` → `catalog_books`                  | `SchoolBook` → `books`                  |
| `Assignment` → `catalog_assignments`      | `SchoolAssignment` → `assignments`      |
| `Question` → `catalog_questions`          | `QuestionBank` → `question_bank`        |
| `ExamTemplate` → `catalog_exam_templates` | `SchoolExamTemplate` → `exam_templates` |

## Curriculums

A `Curriculum` row (`catalog_curricula`, unique on `country` + `code`) groups the subjects for one
country/framework. Ten exist today, but only two carry real chapters and lessons — the rest are
**subject-only shells** (tiles with no content yet).

**Default vs separate seeds.** `pnpm db:seed` (Phase 0) only loads US + the Sudan base set via
`seedCatalog`. The eight world curricula, the formal `Curriculum` records, and the full Sudan
content tree are explicit, separate runs:

```bash
pnpm db:seed:single world-curricula    # GB, IB, SA, EG, AE, QA, KW, JO (subjects only)
pnpm db:seed:single curriculum         # Curriculum records + curriculumId backfill
pnpm db:seed:single sync-sd-curriculum # Full Sudan tree (authoritative)
```

A new school picks its curriculum at onboarding via `inferCurriculum(country, schoolType)` in
`src/lib/catalog-setup.ts` (`international → us-k12`; `US → us-k12`, `GB → british`, and the MENA
countries `SD/SA/EG/AE/QA/KW/JO → national`; unknown → `us-k12`).

### What to add next

1. **Deepen IB and British first.** They already have subject shells but no chapters/lessons —
   the highest-demand international curricula. Add real content the same way Sudan was built
   (directory tree or JSON data file → a sync seed that creates `Chapter` / `Lesson` rows).
2. **Wire the prepared data.** `scripts/curriculum-data/cbse-curriculum.json` (India CBSE) and
   `french-curriculum.json` (France) already exist on disk but are not referenced by any seed or by
   `inferCurriculum` yet.

### How to add a curriculum

**Shallow (a new country, subjects only):**

1. Add a `CurriculumWithSubjects` block to `prisma/seeds/world-curricula.ts` — `curriculum` metadata
   (`country`, `code`, `slug`, `name`, `lang`, `gradeRange`, `structure`) plus `subjects[]`, each
   with `name` / `slug` / `department` / `concept` / `color` (via `colorFor(concept)`) / `grades[]`.
2. Push the block into `ALL_CURRICULA` and add a `sortOrder` band in `getSortBase()`.
3. Add the country → code entry to `inferCurriculum` (`catalog-setup.ts`) **and** its mirror in
   `src/lib/timetable-reference.ts` (they must stay in sync).
4. `pnpm db:seed:single world-curricula`. No new images needed — concept S3 paths are reused.

**Content-rich (real chapters + lessons):** additionally provide either

- a **directory tree** like `curriculum/sd/` (`g{N}/<subject>/chapters/<ch>/lessons/<lesson>` +
  `curriculum.json` + `structure.json`) consumed by a directory-driven sync (model
  `sync-sd-curriculum.ts`), **or**
- a **JSON data file** (model `scripts/sudan-data/sudan-curriculum.json` or
  `scripts/clickview-data/*.json`) consumed by a dedicated seed (model `sd-catalog.ts` /
  `us-catalog.ts`) that creates `Chapter` and `Lesson` rows and updates the denormalized counts.

## Auto-assignment

`setupCatalogForSchool(schoolId, opts)` creates bridge rows via 4-step progressive fallback (the internal query is `findSubjects()`):

| Step | Filter                                 |
| ---- | -------------------------------------- |
| 1    | `country + curriculum + schoolType`    |
| 2    | `country + curriculum`                 |
| 3    | `country="*" + curriculum` (worldwide) |
| 4    | `US + us-k12` (baseline fallback)      |

Grade-specific subjects produce a 1:1 mapping — "Math Grade 3" maps directly to the school's Grade 3.

```ts
import { setupCatalogForSchool } from "@/lib/catalog-setup"

await setupCatalogForSchool(schoolId, { country: "US", curriculum: "us-k12" })
await setupCatalogForSchool(schoolId, { country: "AE", curriculum: "ib" })
await setupCatalogForSchool(schoolId, { country: "GB", curriculum: "british" })
```

> The subject _selection_ works for any curriculum, but the academic _structure_ it builds (level/grade names, streams) is currently hardcoded to the Sudanese model — see [Known gaps & roadmap](#known-gaps--roadmap).

## Content review flow

```
Teacher/School creates content
  → approvalStatus: PENDING, visibility: PRIVATE
SaaS admin reviews
  → APPROVED → indexed, visibility set by creator
       PRIVATE  — only creator's school
       SCHOOL   — schools using that subject (free)
       PUBLIC   — anyone (free or paid)
  → REJECTED → feedback sent to creator
```

## Data flows

**Operational** — drives timetabling, library, exams:

```
Subject → SubjectSelection → Class → Timetable
Subject → Book → BookSelection → SchoolBook (library)
Subject → Exam → SchoolExam
Subject → Question → QuestionBank
```

**LMS** — drives e-learning + certificates:

```
Subject → Enrollment → LessonProgress → SubjectCertificate
Lesson → Video (multi-teacher)
Lesson → Attachment / Question
```

The `syncStudentClassToEnrollment()` helper (`src/lib/enrollment-sync.ts`) bridges classroom enrollment to LMS access via `Class.subjectId` → `Subject`. It's non-blocking, idempotent, and skips gracefully when the catalog FK is missing.

## Shared concept & CDN assets

Catalog images are **shared by concept, not cloned per subject**. Every `Subject` carries an
optional `concept` string (`"math"`, `"languages"`, …) that resolves to one set of S3 assets reused
across all grades and all curricula. Arabic in Sudan and World Languages in the US both map to the
same `languages` concept and serve the same images — so ~1,300 subjects share ~two dozen source
images instead of one image each. The full concept list, cross-curriculum map, and chapter rotation
pools live in [Concepts](/docs/concept); this section covers how the keys turn into URLs.

### The four S3 key fields on `Subject`

| Field       | DB column      | Kind   | Resolves to                                                                         |
| ----------- | -------------- | ------ | ----------------------------------------------------------------------------------- |
| `thumbnail` | `thumbnailKey` | prefix | `{key}-{sm\|md\|lg\|original}.webp` appended at read time                           |
| `banner`    | `bannerUrl`    | prefix | same `-{size}.webp` suffixing, wide landscape art                                   |
| `cover`     | `cover`        | full   | textbook cover — passed verbatim, grade-agnostic `catalog/concepts/{concept}/cover` |
| `pdf`       | `pdf`          | full   | textbook PDF, unique per subject (only populated for Sudan today)                   |

`thumbnail` / `banner` are **prefixes** (the size + `.webp` is appended on read); `cover` / `pdf`
are **full keys** passed straight to the CDN with no suffixing.

### Pipeline (server) → URL (client)

| File                           | Environment | Purpose                                                                                          |
| ------------------------------ | ----------- | ------------------------------------------------------------------------------------------------ |
| `src/lib/catalog-image.ts`     | Server only | Sharp → 4 WebP variants → S3 upload (`processAndUploadCatalogImage`, `deleteCatalogImage`)       |
| `src/lib/catalog-image-url.ts` | Client-safe | `getCatalogImageUrl(key, size)`, `getCatalogImageSrcSet(key)`                                    |
| `src/lib/cloudfront-url.ts`    | Client-safe | Pure env/string helpers — the real fallback chain (`getCloudFrontUrl`, `isCloudFrontConfigured`) |
| `src/lib/cloudfront.ts`        | Server only | AWS SDK — signed URLs for paid video, cache invalidation                                         |

The split is deliberate: `catalog-image.ts` and `cloudfront.ts` pull in Sharp / `@aws-sdk/*` and are
server-only; `catalog-image-url.ts` and `cloudfront-url.ts` are SDK-free so they can ship in client
bundles.

| Variant          | Width  | Quality  | Use             |
| ---------------- | ------ | -------- | --------------- |
| `-sm.webp`       | 200px  | 95       | Grid thumbnails |
| `-md.webp`       | 600px  | 97       | Card images     |
| `-lg.webp`       | 1200px | 100      | Hero banners    |
| `-original.webp` | Full   | Lossless | Full resolution |

All variants upload as `image/webp` with `Cache-Control: public, max-age=31536000, immutable`.

### URL resolution & CDN config

`getCloudFrontUrl(key)` resolves in order: **(1)** `CLOUDFRONT_DOMAIN` → `https://{domain}/{key}`;
**(2)** else `AWS_S3_BUCKET` set → `https://{bucket}.s3.{AWS_REGION}.amazonaws.com/{key}`; **(3)**
else the bare key is returned unchanged. `isCloudFrontConfigured()` is true if **either**
`CLOUDFRONT_DOMAIN` or `AWS_S3_BUCKET` is set. For thumbnails, `getCatalogImageUrl` returns `null`
when nothing is configured, and the `CatalogImage` atom renders a color/"No image" placeholder.

> A separate `NEXT_PUBLIC_CDN_DOMAIN` powers the generic `src/lib/asset-url.ts` — don't confuse it
> with `CLOUDFRONT_DOMAIN`, which is the one the catalog concept pipeline uses. Note also that URL
> helpers never check S3 object existence, so a key set without a matching upload surfaces as a
> broken `<img>` (404), not the placeholder — see [Known gaps](#known-gaps--roadmap).

## Lesson progress and certificates

## Query patterns

```ts
// All subjects for a school + grade
await db.subjectSelection.findMany({
  where: { schoolId, gradeId, isActive: true },
  include: {
    subject: {
      include: {
        chapters: {
          where: { status: "PUBLISHED" },
          orderBy: { sequenceOrder: "asc" },
        },
      },
    },
  },
})

// Lesson with ranked videos
await db.lesson.findUnique({
  where: { id: lessonId },
  include: {
    videos: {
      where: {
        approvalStatus: "APPROVED",
        OR: [{ visibility: "PUBLIC" }, { schoolId }, { isFeatured: true }],
      },
      orderBy: [
        { isFeatured: "desc" },
        { viewCount: "desc" },
        { averageRating: "desc" },
      ],
    },
  },
})

// Catalog questions for a chapter test
await db.question.findMany({
  where: {
    catalogChapterId: chapterId,
    approvalStatus: "APPROVED",
    visibility: "PUBLIC",
    status: "PUBLISHED",
  },
  orderBy: [{ qualityScore: "desc" }, { usageCount: "desc" }],
})
```

### Coverage reporting

`src/lib/catalog/coverage.ts` (pure reads, no auth/side-effects) answers "what's actually seeded per
curriculum" — `listCurriculaWithCoverage()` returns every curriculum with its subject count and grade
spread; `getCurriculumCoverage(curriculumId)` drills into per-grade / per-department distribution and
chapters-with-lessons vs empty. It is the read-side counterpart to the provisioning engine and feeds
the planned catalog browse + bridge UI (it has no production consumer yet — see
[Known gaps](#known-gaps--roadmap)).

## Seed pipeline

The default `catalog` seed loads US K-12 plus a Sudan base set. The full Sudan curriculum and eight more world curricula ship as **separate** seeds — they are not in the default path.

| Seed (`pnpm db:seed:single …`)                                                    | Contents                                                                       |
| --------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| `catalog`                                                                         | `seedUsCatalog` (US K-12, ~240 subjects) + `seedSudanCatalog` (Sudan base set) |
| `sync-sd-curriculum`                                                              | Full Sudan national curriculum from `curriculum/sd/` (97 subjects)             |
| `world-curricula`                                                                 | 8 curricula — GB, IB, SA, EG, AE, QA, KW, JO — grade-specific subjects         |
| `curriculum`                                                                      | `Curriculum` records (SD-national, US-k12) + backfill `Subject.curriculumId`   |
| `concept-images` / `clickview-images`                                             | Shared concept thumbnails → Sharp → S3/CloudFront                              |
| `catalog-books` / `catalog-content` / `catalog-videos` / `catalog-exam-templates` | Books, materials, videos, exam templates                                       |
| `academic-catalog`                                                                | Academic structure + bridge rows for the demo school                           |

```bash
pnpm db:seed:single catalog            # US K-12 + Sudan base set
pnpm db:seed:single world-curricula    # GB, IB, SA, EG, AE, QA, KW, JO
pnpm db:seed:single sync-sd-curriculum # Full Sudan curriculum
pnpm db:seed:single concept-images     # Thumbnails → S3/CDN
```

For exact subject/chapter/lesson counts and the seed architecture, see [Curriculum engineering](/docs/curriculum-engineering) and [Sudan curriculum](/docs/sudan-curriculum).

## Known gaps & roadmap

The catalog's _content_ layer is multi-curriculum, but the _provisioning_ layer
(`src/lib/catalog-setup.ts`, ~1,716 lines) still bakes in Sudanese assumptions. Status as of
this writing:

### P0 — Blockers

- **Academic structure is hardcoded to the Sudanese model.** `LEVEL_CONFIG` (6+3+3),
  `GRADE_NAMES` (Arabic), `HIGH_SCHOOL_STREAMS` (Science/Arts), and `SCIENCE/ARTS_ONLY_PATTERNS`
  (`catalog-setup.ts` lines 10–122) are applied to **every** school. A US/UK/IB school gets
  Arabic Sudanese grade names and forced Science/Arts streams on grades 10–12. Fix: split
  `catalog-setup.ts` into a shared engine + per-curriculum `CurriculumConfig`
  (`src/lib/catalog/curricula/{sd,us,gb,ib}.ts`). Do **not** split the `catalog_subjects` table.
- **British / IB are not provisionable end-to-end.** The eight world curricula are subjects-only
  (no chapters/lessons) and the `world-curricula` seed is **not** in the default `catalog` seed, so
  `findSubjects` falls through to the US baseline until it is run explicitly.
- **GB curriculum-code mismatch.** `inferCurriculum` returns `british` for `GB`, but
  `world-curricula.ts` creates GB subjects with code `national` (slug `gb-national`). A GB school
  looks for `british` subjects and finds none — it only works via the progressive fallback. Pick one
  code (`national`) and use it in both places.
- **IB is unreachable via inference.** IB uses `country = "*"` and no country maps to `ib` in
  `inferCurriculum`, so IB can only be selected by passing `options.curriculum: "ib"` explicitly.

### P1

- **Partial-failure recovery is incomplete.** Provisioning runs fire-and-forget via Next.js
  `after()` (`onboarding/legal/actions.ts`). Only `ensureSubjectSelections` self-heals — and only
  for subject selections, not levels/grades/streams/timetable/sections/fees.
- **Dead / unwired exports.** `setupLibraryForSchool` (library books are never provisioned at
  onboarding), `teardownCatalogForSchool`, `getRankedVideos`, and `recordVideoView` have **no**
  production callers (test-only). The last two are stream-domain logic that doesn't belong here.
  (`publishSchool` referenced in older docs actually lives in
  `onboarding/congratulations/actions.ts`, not `catalog-setup.ts`.) Wire them in or remove them.
- **`findSubjects` has no visibility guard** (filters on `status: PUBLISHED` only) —
  defense-in-depth gap: a published-but-non-public catalog subject could be provisioned anywhere.
- **Single-responsibility drift.** `catalog-setup.ts` also owns timetable generation and section
  auto-provisioning (plus the dead stream helpers above) — split these out.

### P2

- **`getDefaultWeeklyPeriods` is dead code** — production resolves weekly periods via
  `getReferenceWeeklyPeriods` (`src/lib/timetable-reference.ts`); the `catalog-setup.ts` copy
  survives only in the `_testing` export. Remove it.
- `gpa` in `SCORE_RANGE_DEFAULTS` is stripped before insert (no `gpa` column on `ScoreRange`).
- `legalData` is collected during onboarding but discarded (legal-step TODO).
- Academic levels/grades/streams are inserted in a sequential `tx.create` loop — batch with
  `createMany` (subject selections already use `createMany`).
- **`coverage.ts` has no consumer.** The per-curriculum coverage module (PR #348) is ready but the
  catalog browse + bridge UI it feeds is not built yet.
- **CDN asset gaps.** No seed uploads the `cover` object, so `Subject.cover` keys can 404; banners
  depend on `banner-copy.ts` (ClickView source); `pdf` is only populated for Sudan; and URL helpers
  never check S3 existence, so missing assets surface as broken images, not the placeholder.
- `inferCurriculum` is duplicated in `timetable-reference.ts` — the two copies must be kept in sync.
- TOC extraction quality and denormalized count-update performance — tracked in
  [Curriculum engineering → Improvement areas](/docs/curriculum-engineering).

### Direction

- Drive level/grade names from `School.preferredLanguage` / `Curriculum.lang`; gate Science/Arts
  streams to `national` only.
- Make provisioning a resumable, idempotent pipeline with per-step status persisted on `School`;
  generalize `ensureSubjectSelections` → `ensureProvisioned`.
