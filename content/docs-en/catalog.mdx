The catalog is the platform-wide store of curriculum content. One copy of every subject, chapter, and lesson lives in shared tables with no `schoolId`; every school references that same content through small bridge rows instead of cloning it. Teacher- and school-contributed content (videos, materials, questions, books) flows through SaaS review and enriches the catalog for everyone.

## How it works

### Loading content

Content is authored outside the database and pulled in by seeds. `catalog/us.ts` reads `scripts/clickview-data/master-inventory.json`; `catalog/sd-base.ts` reads `scripts/sudan-data/sudan-curriculum.json`; `catalog/sd.ts` reads the richer on-disk tree under `curriculum/sd/`. Each seed upserts rows into `catalog_subjects`, `catalog_chapters`, and `catalog_lessons` and is safe to re-run. The default `catalog` seed loads US K-12 and the Sudan base set; the full Sudan sync and the eight world curricula in `catalog/world.ts` are separate, explicit runs — see [Completing the catalog](#completing-the-catalog) and [Seed pipeline](#seed-pipeline).

### Images

Thumbnails are not stored per subject. Each subject points at one of the shared **concepts**, and `catalog/concepts.ts` processes the source files in `public/subjects/concepts/` into S3 under `catalog/concepts/…`, served through CloudFront. At read time `image-url.ts` builds the public URL. See [Concepts](/docs/concept) and [Image pipeline](#image-pipeline).

### Connecting a school

A school never copies the catalog. When onboarding finishes, `completeOnboarding` (`src/components/onboarding/legal/actions.ts`) calls `setupCatalogForSchool` (`src/components/catalog/setup.ts`): it infers the school's curriculum from country + type, finds the matching subjects through a four-step fallback, builds the academic levels/grades/streams, and writes one `SubjectSelection` bridge row per adopted subject and grade. Those bridge rows tie a tenant to the shared content; everything school-facing — classes, timetable, library, exams — reads through them. If bridge rows go missing, `ensureSubjectSelections` repairs them lazily on first read. See [Auto-assignment](#auto-assignment).

### Reading it at runtime

Schools query the bridge, not the catalog directly — `db.subjectSelection.findMany({ where: { schoolId, gradeId }, include: { subject: { include: { chapters } } } })` returns the subjects a school adopted for a grade, joined to the shared content, carrying the optional per-school custom name and weekly-period settings. See [Query patterns](#query-patterns).

### A student learning

From a `SubjectSelection` a student gets an `Enrollment`. As lessons are watched, `LessonProgress` records the position per lesson, and when every lesson in a subject is complete a `SubjectCertificate` is issued. `syncStudentClassToEnrollment` bridges classroom membership to LMS access so a student enrolled in a class automatically gets the matching subject. See [Lesson progress and certificates](#lesson-progress-and-certificates).

### Contributing back

A teacher or school can add content — videos, materials, questions. It starts private and pending, a platform admin reviews it in the SaaS catalog (`src/components/saas-dashboard/catalog/`), and on approval it is given a visibility — private, school-only, public, or paid — and becomes part of the shared catalog for everyone. See [Content review flow](#content-review-flow).

## Why a shared catalog

| Approach           | Data for 1,000 schools × ~240 subjects                         | Sharing                                 |
| ------------------ | -------------------------------------------------------------- | --------------------------------------- |
| Per-school clone   | 220k subjects, 800k chapters, 4M lessons                       | None                                    |
| **Shared catalog** | **~240 subjects, ~800 chapters, ~4k lessons** + bridge records | Every contribution enriches all schools |

Onboarding a school takes seconds — `setupCatalogForSchool()` reads `country` + `curriculum` and creates `SubjectSelection` bridge rows.

## Auto-assignment

`setupCatalogForSchool(schoolId, opts)` creates bridge rows via 4-step progressive fallback (the internal query is `findSubjects()`):

| Step | Filter                                 |
| ---- | -------------------------------------- |
| 1    | `country + curriculum + schoolType`    |
| 2    | `country + curriculum`                 |
| 3    | `country="*" + curriculum` (worldwide) |
| 4    | `US + us-k12` (baseline fallback)      |

Grade-specific subjects produce a 1:1 mapping — "Math Grade 3" maps directly to the school's Grade 3.

```ts
import { setupCatalogForSchool } from "@/components/catalog/setup"

await setupCatalogForSchool(schoolId, { country: "US", curriculum: "us-k12" })
await setupCatalogForSchool(schoolId, { country: "AE", curriculum: "ib" })
await setupCatalogForSchool(schoolId, { country: "GB", curriculum: "british" })
```

> The subject _selection_ works for any curriculum, but the academic _structure_ it builds (level/grade names, streams) is currently hardcoded to the Sudanese model — see [Known gaps & roadmap](#known-gaps--roadmap).

## Content review flow

```
Teacher/School creates content
  → approvalStatus: PENDING, visibility: PRIVATE
SaaS admin reviews
  → APPROVED → indexed, visibility set by creator
       PRIVATE  — only creator's school
       SCHOOL   — schools using that subject (free)
       PUBLIC   — anyone (free or paid)
  → REJECTED → feedback sent to creator
```

## Data flows

**Operational** — drives timetabling, library, exams:

```
Subject → SubjectSelection → Class → Timetable
Subject → Book → BookSelection → SchoolBook (library)
Subject → Exam → SchoolExam
Subject → Question → QuestionBank
```

**LMS** — drives e-learning + certificates:

```
Subject → Enrollment → LessonProgress → SubjectCertificate
Lesson → Video (multi-teacher)
Lesson → Attachment / Question
```

The `syncStudentClassToEnrollment()` helper (`src/lib/enrollment-sync.ts`) bridges classroom enrollment to LMS access via `Class.subjectId` → `Subject`. It's non-blocking, idempotent, and skips gracefully when the catalog FK is missing.

| `lang`                                                           | `String`            | Default `"ar"`                                            |
| `type`                                                           | `MaterialType`      | `TEXTBOOK`, `SYLLABUS`, `WORKSHEET`, `PRESENTATION`, etc. |

### `Assignment`

Reusable homework/project templates (table `catalog_assignments`). Same shape as `Material` plus:

| Field                     | Type       | Notes                                                 |
| ------------------------- | ---------- | ----------------------------------------------------- |
| `instructions` / `rubric` | `String?`  | Long-form text                                        |
| `totalPoints`             | `Decimal?` |                                                       |
| `estimatedTime`           | `Int?`     | Minutes                                               |
| `assignmentType`          | `String?`  | `homework`, `project`, `lab`, `essay`, `presentation` |

### `ExamQuestion`

Junction table linking `Exam` ↔ `Question` with ordering (table `catalog_exam_questions`). Replaces the legacy `Exam.questions` JSON field.

| Field                                | Notes                                |
| ------------------------------------ | ------------------------------------ |
| `catalogExamId`, `catalogQuestionId` | FKs (cascade delete)                 |
| `order`                              | Question order within exam           |
| `points`                             | `Decimal` — per-exam points override |

Indexes: `@@unique([catalogExamId, order])`, `@@unique([catalogExamId, catalogQuestionId])`.

### `Video`, `Attachment`

Full schemas in `prisma/models/video.prisma` (`lesson_videos`, `lesson_attachments`). Key fields on `Video`: `videoUrl`, `provider` (`youtube`/`vimeo`/`self-hosted`), `visibility`, `price`, `approvalStatus`, plus engagement (`viewCount`, `likeCount`, `averageRating`, `isFeatured`). Paid videos are gated through `VideoPurchase`.

## Image pipeline

Server-side via Sharp, served from CloudFront CDN. Pure client/server split.

| Variant          | Width  | Quality  | Use             |
| ---------------- | ------ | -------- | --------------- |
| `-sm.webp`       | 200px  | 95       | Grid thumbnails |
| `-md.webp`       | 600px  | 97       | Card images     |
| `-lg.webp`       | 1200px | 100      | Hero banners    |
| `-original.webp` | Full   | Lossless | Full resolution |

| File                                  | Environment | Purpose                                              |
| ------------------------------------- | ----------- | ---------------------------------------------------- |
| `src/components/catalog/image.ts`     | Server only | `processAndUploadCatalogImage`, `deleteCatalogImage` |
| `src/components/catalog/image-url.ts` | Client-safe | `getCatalogImageUrl`                                 |

`Subject` carries four S3 key fields — `thumbnail` (`@map("thumbnailKey")`), `banner` (`@map("bannerUrl")`), `cover` (textbook cover), `pdf` (textbook PDF). URL resolution priority:

1. CloudFront CDN URL from `thumbnail` → `{CDN}/{thumbnailKey}-{size}.webp`
2. Concept-based static fallback (shared concept thumbnails per [/docs/concept](/docs/concept))
3. `null` — UI falls back to color background

## Lesson progress and certificates

`LessonProgress` tracks per-lesson state: `watchedSeconds` (resume), `isCompleted`, `watchCount`, `lessonVideoId`. Linked to both `Enrollment` and `Lesson`.

When `markLessonComplete` (`src/components/stream/dashboard/lesson/catalog-actions.ts`) is called:

1. Upsert `LessonProgress` with `isCompleted: true`
2. Query all published lessons in the subject
3. If all complete → set `Enrollment.status = COMPLETED`
4. Generate `SubjectCertificate` with unique number
5. Send completion email asynchronously

## Query patterns

```ts
// All subjects for a school + grade
await db.subjectSelection.findMany({
  where: { schoolId, gradeId, isActive: true },
  include: {
    subject: {
      include: {
        chapters: {
          where: { status: "PUBLISHED" },
          orderBy: { sequenceOrder: "asc" },
        },
      },
    },
  },
})

// Lesson with ranked videos
await db.lesson.findUnique({
  where: { id: lessonId },
  include: {
    videos: {
      where: {
        approvalStatus: "APPROVED",
        OR: [{ visibility: "PUBLIC" }, { schoolId }, { isFeatured: true }],
      },
      orderBy: [
        { isFeatured: "desc" },
        { viewCount: "desc" },
        { averageRating: "desc" },
      ],
    },
  },
})

// Catalog questions for a chapter test
await db.question.findMany({
  where: {
    catalogChapterId: chapterId,
    approvalStatus: "APPROVED",
    visibility: "PUBLIC",
    status: "PUBLISHED",
  },
  orderBy: [{ qualityScore: "desc" }, { usageCount: "desc" }],
})
```

## Seed pipeline

The default `catalog` seed loads US K-12 plus a Sudan base set. The full Sudan curriculum and eight more world curricula ship as **separate** seeds — they are not in the default path.

| Seed (`pnpm db:seed:single …`)                    | Contents                                                                       |
| ------------------------------------------------- | ------------------------------------------------------------------------------ |
| `catalog`                                         | `seedUsCatalog` (US K-12, ~240 subjects) + `seedSudanCatalog` (Sudan base set) |
| `sd`                                              | Full Sudan national curriculum from `curriculum/sd/` (125 subjects)            |
| `world`                                           | 8 curricula — GB, IB, SA, EG, AE, QA, KW, JO — grade-specific subjects         |
| `meta`                                            | `Curriculum` records (SD-national, US-k12) + backfill `Subject.curriculumId`   |
| `concepts` / `banners` / `clickview-lessons`      | Concept thumbnails, banners & lesson images → Sharp → S3/CloudFront            |
| `books` / `content` / `videos` / `exam-templates` | Books, materials, videos, exam templates                                       |
| `demo`                                            | Academic structure + bridge rows for the demo school                           |

```bash
pnpm db:seed:single catalog   # US K-12 + Sudan base set
pnpm db:seed:single world     # GB, IB, SA, EG, AE, QA, KW, JO
pnpm db:seed:single sd        # Full Sudan curriculum
pnpm db:seed:single concepts  # Thumbnails → S3/CDN
```

For exact subject/chapter/lesson counts and the seed architecture, see [Curriculum engineering](/docs/curriculum-engineering) and [Sudan curriculum](/docs/sudan-curriculum).

## Completing the catalog

Today the catalog is **two curricula deep and ten curricula wide**: US and Sudan have full Subject → Chapter → Lesson content; the other eight exist as subjects only. This is the plan to close that gap and finish wiring every curriculum to the shared concept images.

### Where each curriculum stands

| Curriculum                     | Subjects | Chapters | Lessons | Source                                         | Seed                       |
| ------------------------------ | -------- | -------- | ------- | ---------------------------------------------- | -------------------------- |
| US (`us-k12`)                  | ~240     | ~750     | ~3,993  | `scripts/clickview-data/master-inventory.json` | `us` (default)             |
| Sudan (`national`)             | 125      | 883      | 2,940   | `curriculum/sd/`                               | `sd-base` (default) + `sd` |
| GB, IB, SA, EG, AE, QA, KW, JO | yes      | none     | none    | inline in `catalog/world.ts`                   | `world` (manual)           |

Two rich trees are authored on disk but no seed reads them: `curriculum/uk` (British — 184 subjects / 363 chapters / 730 lessons) and `curriculum/us` (American — 128 / 266 / 535), both in the same shape as `curriculum/sd/`.

### Two things are missing

1. **Depth** — eight curricula have subjects but no chapters or lessons.
2. **Orchestration** — the default `pnpm db:seed` loads only US and the Sudan base set; `world`, the full `sd` sync, `concepts`, and `meta` are manual single-seed runs, so a fresh database has no world curricula and no concept images until each is run by hand.

### Completing the content

The fastest unlock is already on disk. `sd.ts` (`syncSdCurriculum`) ingests the `curriculum/sd/` tree shape; generalizing it into one `syncCurriculumTree(dir, country, curriculum)` lets it read `curriculum/uk` and `curriculum/us` immediately — giving GB real depth and a real American tree to reconcile against ClickView.

For the curricula with no source yet (IB, SA, EG, AE, QA, KW, JO), content must be authored before it can be ingested. Three existing pipelines produce the on-disk shape `syncCurriculumTree` consumes:

- `curriculum/_build_tools/engine.py` — the authoring engine that generated the `uk` and `us` trees (`structure.json` + `qbank`/`exams` per node); `make_curriculum_json.py` defines per-curriculum subject lists with local overlays.
- `scripts/assemble-sudan-curriculum.ts` — merges per-subject table-of-contents files into one curriculum JSON (the template for "extract a TOC, assemble, seed").
- The content layer — teachers and schools contribute chapters and lessons through the SaaS review flow, filling depth organically.

Reconcile the **dual US**: the database `us-k12` (ClickView) and the disk `curriculum/us` (American with local overlays) are two different datasets with different slugs and subjects, cross-referenced nowhere. Pick one as canonical, or merge them under one `country`/`curriculum` key, before ingesting the disk tree.

Then make completion the default: order the seeds into one pipeline — `meta` → `us` → `sd-base`/`sd` → `world` → `concepts` → `banners` — so a fresh database has every curriculum and its images. See [Curriculum engineering](/docs/curriculum-engineering) for the unified `seedCurriculumFromJSON` refactor, the canonical curriculum JSON schema, and the bulk-SQL count update.

### Connecting concept images

Every subject in every curriculum already resolves to one of the shared concepts — `world.ts` assigns them inline; US and Sudan through `NAME_TO_CONCEPT`. What remains:

- **Chapter and lesson images** are wired only for US and Sudan, because only those have chapters and lessons. They come for free for the eight world curricula once depth lands: `us.ts` and `sd.ts` rotate chapters through a `SUBJECT_CONCEPT_POOL`, and the generalized sync inherits that.
- **One source of truth for the maps.** `NAME_TO_CONCEPT` lives in three places (`concepts.ts`, `us.ts`, and `sd.ts` as `SUBJECT_CONCEPT_MAP`) and the rotation pool in two; the US copy has already drifted (missing the Arabic keys). Hoist them into the shared core (`src/components/catalog/`) so every seed reads one map.
- **Finish the assets.** `concepts.ts` uploads only the `thumbnail` variant; `banners.ts` covers 10 of 23 concepts, so the other 13 `banner` keys resolve to missing objects, and no `cover` object is uploaded. Extend the concept seed to produce `banner` (and `cover`, if used) for all concepts, or drop the unused keys. The `arabic` source image is currently orphaned — no subject maps to it.

## Known gaps & roadmap

The catalog's _content_ layer is multi-curriculum (see [Completing the catalog](#completing-the-catalog)), but the _provisioning_ layer (`src/components/catalog/setup.ts`, ~1,716 lines) still bakes in Sudanese assumptions. Status as of this writing:

### P0 — Blockers

- **Academic structure is hardcoded to the Sudanese model.** `LEVEL_CONFIG` (6+3+3),
  `GRADE_NAMES` (Arabic), `HIGH_SCHOOL_STREAMS` (Science/Arts), and `SCIENCE/ARTS_ONLY_PATTERNS`
  (`setup.ts` lines 10–122) are applied to **every** school. A US/UK/IB school gets
  Arabic Sudanese grade names and forced Science/Arts streams on grades 10–12. Fix: split
  `setup.ts` into a shared engine + per-curriculum `CurriculumConfig`
  (`src/components/catalog/curricula/{sd,us,gb,ib}.ts`). Do **not** split the `catalog_subjects` table.
- **British / IB are not provisionable end-to-end.** `inferCurriculum` maps `GB → british` and
  international → `ib`, but the `world` seed is **not** in the default `catalog` seed,
  so `findSubjects` may fall through to the US baseline until it is run explicitly.

### P1

- **Partial-failure recovery is incomplete.** Provisioning runs fire-and-forget via Next.js
  `after()` (`onboarding/legal/actions.ts`). Only `ensureSubjectSelections` self-heals — and only
  for subject selections, not levels/grades/timetable/sections/fees.
- **`findSubjects` has no visibility guard** (filters on `status: PUBLISHED` only) —
  defense-in-depth gap.
- **Single-responsibility drift.** `setup.ts` also owns timetable generation and section
  auto-provisioning — split these out.

### P2

- Duplicated weekly-period logic (`getDefaultWeeklyPeriods` vs `src/lib/timetable-reference.ts`).
- `gpa` in `SCORE_RANGE_DEFAULTS` is stripped before insert (no `gpa` column on `ScoreRange`).
- `legalData` is collected during onboarding but discarded (legal-step TODO).
- Academic-structure rows are inserted in a sequential `tx.create` loop — batch with `createMany`.
- TOC extraction quality and denormalized count-update performance — tracked in
  [Curriculum engineering → Improvement areas](/docs/curriculum-engineering).

### Direction

- Drive level/grade names from `School.preferredLanguage` / `Curriculum.lang`; gate Science/Arts
  streams to `national` only.
- Make provisioning a resumable, idempotent pipeline with per-step status persisted on `School`;
  generalize `ensureSubjectSelections` → `ensureProvisioned`.
