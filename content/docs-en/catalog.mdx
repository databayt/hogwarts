The catalog is the platform's shared curriculum store. One copy of every subject,
chapter, and lesson lives in global tables with **no `schoolId`**; every school
references that same content through small bridge rows instead of cloning it.
Teacher- and school-contributed content (videos, materials, questions, books)
flows through SaaS review and enriches the catalog for everyone.

This page is the single hub for the catalog. The full CDN/asset reference lives
separately at [CDN & Assets](/docs/cdn).

## Overview

The whole system rests on **one mental model**: there are **global catalog
tables** (shared, no tenant) and **`schoolId`-scoped bridge rows** that connect a
tenant to that shared content. A school never copies the catalog — it points at it.

<CatalogStructure />

- **Global catalog** (`catalog_*` tables, no `schoolId`) — the curriculum
  hierarchy (`Subject` → `Chapter` → `Lesson`) plus the content attached to it
  (`Exam`, `Question`, `Material`, `Assignment`, `Video`, `Book`, `Quiz`,
  `Document`, `Image`, `Textbook`) and `Curriculum` records.
- **Bridge rows** (`school_*` tables, carry `schoolId`) — `SubjectSelection`,
  `ContentOverride`, `InstructorPreference`, `BookSelection`. These tie a tenant
  to the shared content; everything school-facing reads through them.
- **Why** — at 1,000 schools the shared model stores ~240 subjects instead of
  220k, and every contribution enriches all schools.

| Approach           | Data for 1,000 schools × ~240 subjects                         | Sharing                                 |
| ------------------ | -------------------------------------------------------------- | --------------------------------------- |
| Per-school clone   | 220k subjects, 800k chapters, 4M lessons                       | None                                    |
| **Shared catalog** | **~240 subjects, ~800 chapters, ~4k lessons** + bridge records | Every contribution enriches all schools |

## Architecture

Catalog models use **bare names** mapped to `catalog_*` tables; bridge models map
to `school_*`. The Prisma client accessor is the camelCase of the model name
(`db.subject`, `db.subjectSelection`).

### Global catalog models

| Model          | Table (`@@map`)          | Accessor          | Holds                                                                                |
| -------------- | ------------------------ | ----------------- | ------------------------------------------------------------------------------------ |
| `Subject`      | `catalog_subjects`       | `db.subject`      | Root of the hierarchy; `slug` (unique), `curriculum`, `concept`, denormalized counts |
| `Chapter`      | `catalog_chapters`       | `db.chapter`      | `subjectId` FK, `sequenceOrder`                                                      |
| `Lesson`       | `catalog_lessons`        | `db.lesson`       | `chapterId` FK, `sequenceOrder`, `durationMinutes`                                   |
| `Video`        | `lesson_videos`          | `db.video`        | `videoUrl`, `provider`, `visibility`, `price`, approval                              |
| `Exam`         | `catalog_exams`          | `db.exam`         | Catalog exam (≠ school `SchoolExam`)                                                 |
| `ExamQuestion` | `catalog_exam_questions` | `db.examQuestion` | `Exam` ↔ `Question` junction with `order`/`points`                                   |
| `Question`     | `catalog_questions`      | `db.question`     | Catalog question bank (≠ school `QuestionBank`)                                      |
| `Material`     | `catalog_materials`      | `db.material`     | Textbooks/syllabi/worksheets at any level                                            |

### Bridge models (carry `schoolId`)

| Model                  | Table (`@@map`)                 | Accessor                  | Purpose                                                                |
| ---------------------- | ------------------------------- | ------------------------- | ---------------------------------------------------------------------- |
| `SubjectSelection`     | `school_subject_selections`     | `db.subjectSelection`     | One row per adopted subject+grade; `isActive` toggles a subject on/off |

| `BookSelection`        | `school_book_selections`        | `db.bookSelection`        | Bridges a catalog `Book` into a school library                         |

### School mirrors (catalog FK → `catalogXxxId`)

A catalog model and its school-scoped counterpart are **different** models. When
you read `Exam`, check the table to know which one. The school mirror links back to
the catalog through optional `catalogSubjectId` / `catalogChapterId` /
`catalogLessonId` FKs.

| Catalog (global)                          | School-scoped (per-tenant)              |
| ----------------------------------------- | --------------------------------------- |
| `Exam` → `catalog_exams`                  | `SchoolExam` → `exams`                  |
| `Book` → `catalog_books`                  | `SchoolBook` → `books`                  |
| `Assignment` → `catalog_assignments`      | `SchoolAssignment` → `assignments`      |
| `Question` → `catalog_questions`          | `QuestionBank` → `question_bank`        |
| `ExamTemplate` → `catalog_exam_templates` | `SchoolExamTemplate` → `exam_templates` |

### Content-model detail

`Material` (table `catalog_materials`) — books, syllabi, worksheets at any level:

| Field                                                         | Type           | Notes                                               |
| ------------------------------------------------------------- | -------------- | --------------------------------------------------- |
| `catalogSubjectId` / `catalogChapterId` / `catalogLessonId`   | `String?`      | One of these is set (FK into the catalog hierarchy) |
| `title` / `lang`                                              | `String`       | `lang` defaults `"ar"`                              |
| `type`                                                        | `MaterialType` | `TEXTBOOK`, `SYLLABUS`, `WORKSHEET`, …              |
| `fileUrl` / `fileKey` / `fileSize` / `mimeType` / `pageCount` | mixed          | S3-backed                                           |
| `externalUrl`                                                 | `String?`      | For non-uploaded resources                          |
| `contributedBy` / `contributedSchoolId`                       | `String?`      | Attribution                                         |
| `approvalStatus` / `visibility`                               | enums          | Review + gating                                     |

`Assignment` (table `catalog_assignments`) — same shape as `Material` plus
`instructions`/`rubric` (`String?`), `totalPoints` (`Decimal?`), `estimatedTime`
(`Int?`, minutes), `assignmentType` (`homework`/`project`/`lab`/`essay`/`presentation`).

`ExamQuestion` (table `catalog_exam_questions`) — junction linking `Exam` ↔
`Question` with `order` and a per-exam `points` override. Indexes:
`@@unique([catalogExamId, order])`, `@@unique([catalogExamId, catalogQuestionId])`.
Replaces the legacy `Exam.questions` JSON field.

`Video`, `Attachment` — full schemas in `prisma/models/video.prisma`
(`lesson_videos`, `lesson_attachments`). Key `Video` fields: `videoUrl`,
`provider` (`youtube`/`vimeo`/`self-hosted`), `visibility`, `price`,
`approvalStatus`, plus engagement (`viewCount`, `likeCount`, `averageRating`,
`isFeatured`). Paid videos are gated through `VideoPurchase`.

## Provisioning

A school never copies the catalog. When onboarding finishes, `completeOnboarding`
(`src/components/onboarding/legal/actions.ts`) calls `setupCatalogForSchool`
(`src/components/catalog/setup.ts`): it **infers** the curriculum, **finds** the
matching subjects through a progressive fallback, **builds** the academic
levels/grades/streams from the per-curriculum config, and writes one
`SubjectSelection` bridge row per adopted subject and grade.

```ts
import { setupCatalogForSchool } from "@/components/catalog/setup"

await setupCatalogForSchool(schoolId, { country: "US", curriculum: "US" })
await setupCatalogForSchool(schoolId, { country: "AE", curriculum: "IB-DP" })
await setupCatalogForSchool(schoolId, { country: "GB", curriculum: "GB" })
```

### `inferCurriculum` + `findSubjects` fallback

A school row stores **no** curriculum code — `inferCurriculum()` derives it from
country + type at runtime. `findSubjects()` then resolves subjects through a
4-step progressive fallback, so a school always gets _something_:

| Step | Filter                                 |
| ---- | -------------------------------------- |
| 1    | `country + curriculum + schoolType`    |
| 2    | `country + curriculum`                 |
| 3    | `country="*" + curriculum` (worldwide) |
| 4    | `US` (baseline fallback)               |

Grade-specific subjects produce a 1:1 mapping — "Math Grade 3" maps directly to
the school's Grade 3.

### `ensureSubjectSelections` — lazy self-heal

Provisioning runs fire-and-forget via Next.js `after()`. If bridge rows go
missing, `ensureSubjectSelections` repairs them lazily on the first read, so a
partial provisioning failure self-corrects for subject selections.

### Per-curriculum academic structure (the fixed gap)

The old `setup.ts` hardcoded the **Sudanese** model — Arabic 6+3+3 levels, Arabic
grade names, forced Science/Arts streams on grades 10–12 — and applied it to
**every** school, so a US/UK/IB school got Arabic grade names and streams it
shouldn't have. **This is now fixed.** Provisioning resolves the school's
curriculum code to a `CurriculumAcademicConfig` in
`src/components/catalog/academic-config.ts` (`CURRICULUM_ACADEMIC_CONFIG`,
selected via `getAcademicConfig`). The SD entry preserves the original constants
byte-for-byte, so Sudanese schools provision exactly as before.

| Curriculum code(s)                       | Structure                        | Grade naming   | Streams                      |
| ---------------------------------------- | -------------------------------- | -------------- | ---------------------------- |
| `SD`, `SA`, `EG`, `AE`, `QA`, `KW`, `JO` | Arabic 6+3+3                     | `الصف الأول …` | Science / Arts from grade 10 |
| `US`, `CAIE-IGCSE`, `IB-DP`              | English 5+3+4                    | `Grade N`      | None                         |
| `GB`                                     | Key stages (6+3+3 map)           | `Year N`       | None                         |
| `CBSE`                                   | 5+3+4 (primary/middle/secondary) | `Class N`      | None                         |
| _(unknown)_                              | Generic English 6+3+3            | `Grade N`      | None (`GENERIC_FALLBACK`)    |

> Grade **slugs** are intentionally not configurable — they stay `grade-${n}`
> everywhere because the `YearLevel` mapping in `setup.ts` matches on that slug
> shape. Only the display name, level boundaries, and stream policy vary per
> curriculum.

## Proposals & Approval

A school can **propose** new catalog content and, on approval, opt in to it. This
is the request flow shipped this session — note the approval publishes to the
**catalog only**; there is **no auto-bridge** (the school adds it explicitly).

```
School proposes a subject/chapter/lesson
  → Proposal { status: SUBMITTED }      (requireProposer: ADMIN / DEVELOPER / TEACHER)
Operator queue at /catalog/proposals      (sidebar badge shows SUBMITTED count)
  → approveProposal → creates Subject/Chapter/Lesson (status PUBLISHED) in the catalog
                      → proposal PUBLISHED, catalogEntityId set
                      → NO auto-bridge (opt-in)
  → notifies proposer + school ADMINs     (in_app + email, school-language aware)
School catalog picker pins the new subject ("Requested by your school — now ready")
  → one-time reminder dialog + Add CTA
  → pin clears once the subject is added (SubjectSelection exists)
  → rejectProposal → notifies proposer + ADMINs with the rejection reason
```

### Proposing (school side)

`submitSubjectProposal` / `submitChapterProposal` / `submitLessonProposal`
(`src/components/school-dashboard/listings/subjects/catalog/proposal-actions.ts`)
write a `Proposal` row (`catalog_proposals`) with `status: SUBMITTED`. Authorized
by `requireProposer` — **ADMIN, DEVELOPER, or TEACHER** with a `schoolId`. The
`Proposal.data` JSON carries the entity fields; chapter/lesson proposals reference
`parentSubjectId` / `parentChapterId`. A DRAFT or REJECTED proposal can be edited
and resubmitted (`updateProposal`).

### Operator queue

`/catalog/proposals` (DEVELOPER only) lists proposals via `getProposalsForReview`,
defaulting the filter to `SUBMITTED`. The catalog layout computes badge counts —
`db.proposal.count({ where: { status: "SUBMITTED" } })` — and renders the nav
label as `Proposals (N)` when pending.

### Approval → catalog only

`approveProposal` runs the whole flow in one transaction (optimistic status check
inside the tx, slug-collision retry, Arabic names get a random slug suffix). It
creates the `Subject`/`Chapter`/`Lesson` with **`status: PUBLISHED`**, marks the
proposal `PUBLISHED` with `catalogEntityId`, and **does not** create a
`SubjectSelection` — the school opts in. Notifications fire **after** the
transaction (a notification failure never rolls back the approval):

- School language resolved via `resolveSchoolLang(schoolId)` (bilingual title/body).
- Reuses `NotificationType` **`document_shared`** for approval, **`system_alert`**
  for rejection.
- `metadata` carries `{ entityType: "proposal", kind: "proposal_approved" | "proposal_rejected", catalogEntityId, url: "/subjects/catalog" }`.
- Delivered to the **proposer** (skipped if the proposer is an ADMIN — the
  role-audience dispatch already covers them, avoiding duplicates) **and** all
  school **ADMIN**s, on `in_app` + `email`.
- `rejectProposal` requires a reason and includes it in the notification.

### Pinned in the school picker

The school catalog picker (`subject-picker.tsx`) takes `pinnedSubjectIds` —
approved subjects this school requested but hasn't added yet. They render in a
pinned **"Requested by your school — now ready"** section with an **Add** CTA, and
a one-time `CatalogReadyReminder` dialog announces them (dismissal stored in
`localStorage` under `catalog-ready-dismissed:${schoolId}`). The pin filters out
any subject that already has a selection, so it **clears automatically once the
subject is added**.

## Visibility & Paid Content

Every contributed item carries a `ContentVisibility` plus optional
`price`/`currency`. The contribution lifecycle:

```
Teacher/School creates content
  → approvalStatus: PENDING, visibility: PRIVATE, status: DRAFT
SaaS admin reviews (/catalog/approvals)
  → APPROVED → status set to PUBLISHED, visibility set by reviewer
       PRIVATE  — only creator's school
       SCHOOL   — schools using that subject (free)
       PUBLIC   — anyone (free)
       PAID     — anyone, gated by purchase (priceable types only)
  → REJECTED → reason sent to creator
```

### Approval now publishes (the fixed gap)

Approved school contributions are created with `status: DRAFT`, but school browse
paths only surface `PUBLISHED + APPROVED` rows — so previously an approved
contribution stayed **DRAFT-invisible**. `approveContent`
(`saas-dashboard/catalog/approval-actions.ts`) now also sets
**`status: PUBLISHED`** for `Question`, `Material`, and `Assignment` (Video has no
`ContentStatus`; Book is published + auto-selected into the contributor's library).

### Paid videos (wired)

Videos are fully Stripe-gated: a `VideoPurchase` row records a purchase, the URL is
served through **signed CloudFront**, and an unpurchased `PAID` video returns a
**null URL** so the player can't fetch it. Setting a video PAID at approval
requires a positive price and a 3-letter currency.

### Paid exams / questions (deferred)

`Question` and `Exam` carry `price`/`currency` and accept `PAID` visibility, but
the **purchase flow is not built** — the fields exist for a future paid
exam/question store.

### Operator flag management

`updateContentFlags` (+ the `content-flags-dialog`) lets a DEVELOPER set
visibility / status / price per content type, independent of approve/reject. The
dialog enforces a per-type whitelist:

| Field                | Allowed on                                                |
| -------------------- | --------------------------------------------------------- |
| `visibility`         | every type                                                |
| `status`             | every type **except** Video (no ContentStatus)            |
| `price` / `currency` | **Question, Video, Exam** only                            |
| `PAID`               | priceable types only — requires price + 3-letter currency |

### School-side controls

- **`SubjectSelection.isActive`** — turns an adopted subject on/off for the school.
- **`ContentOverride.isHidden`** — hides a chapter/lesson/video. Enforced on
  both school surfaces: the stream LMS (`get-course.ts` drops hidden nodes;
  `admin-get-course.ts` tags everything with `_isHidden` instead) **and** the
  subject detail page (`subjects/[slug]/page.tsx` filters chapters/lessons and
  the video rail for non-admin roles, while ADMIN/DEVELOPER see everything plus
  the mounted **Customize** panel — `SchoolCatalogCustomization`, which was
  previously orphaned).
- **Global search** now filters to **PUBLISHED-only** subjects
  (`atom/generic-command-menu/server/global-search.ts`) — a prior leak of
  unpublished content is fixed.

## Curriculum Engineering

The catalog hierarchy is three global tables, all scope-free; schools reference
them via the bridge. `slug` is the primary upsert key — it encodes country +
grade + subject (`sd-g5-math`, `us-g5-math`), so no compound unique on
`(country, curriculum, grade, name)` is needed.

```prisma
model Subject {                                    // table catalog_subjects
  country      String        @default("SD")        // ISO 3166-1 alpha-2
  curriculum   String        @default("SD")        // canonical code: bare ISO (US, SD, GB) or BODY-PROGRAMME (IB-DP, CAIE-IGCSE, CBSE)
  curriculumId String?                             // optional FK → Curriculum
  concept      String?                             // shared image key
  thumbnail    String?       @map("thumbnailKey")  // S3 key for WebP variants
  grades       Int[]         @default([])          // [5] = grade 5 only
  levels       SchoolLevel[]                       // [ELEMENTARY]
  slug         String        @unique               // sd-g5-math, us-g5-math
  totalChapters Int                                // denormalized counts
  totalLessons  Int
}
```

### On-disk content trees — `curriculum/`

The `curriculum/` directory is the **content source of truth** for the deep
curricula. The build tool (`curriculum/_build_tools/engine.py`) writes, and the
seed engine reads, this shape:

```
curriculum/<dir>/
  g{N}/                       # grade
    <subject>/                # e.g. math, science
      structure.json          # { subject, chapters: [{ name, slug, lessons: [...] }] }
      textbook.pdf            # optional — merged free official PDF
```

`scripts/gen-curriculum-structure.ts` turns a compact chapter-list spec into a
`structure.json` tree; `scripts/upload-textbooks-all.ts` pushes any tree's
textbook PDFs to S3. Legacy formats (`us-inventory.json`,
`sudan-curriculum.json`) keep their bespoke seeds — only net-new curricula go
through the tree engine.

### `syncCurriculumTree` (the tree engine)

The per-seed duplication (upsert, count update, chapter/lesson loop, concept
assignment) is factored into one helper, `syncCurriculumTree`
(`prisma/seeds/catalog/engine.ts`). A new deep curriculum is a ~20-line source
file:

```ts
// prisma/seeds/catalog/gb.ts
import { syncCurriculumTree } from "./engine"

export async function seedGbCurriculum(prisma: PrismaClient) {
  return syncCurriculumTree(prisma, {
    dir: "uk", // curriculum/uk/
    country: "GB",
    code: "GB", // canonical curriculum code
    slugPrefix: "gb", // gb-g{N}-{subject}
    name: "British (England)",
    lang: "en",
    gradeRange: [1, 12],
    sortBase: 20000,
  })
}
```

It reads any `curriculum/<dir>/g{N}/<subject>/structure.json` tree, upserts
`Subject`/`Chapter`/`Lesson` with the canonical code/country, attaches the
textbook PDF S3 key when present, and rotates the shared concept thumbnail across
chapters. It is idempotent (subject upsert by slug; chapters/lessons replaced).
Subjects carry only the curriculum **code** — `registry.ts` owns the `Curriculum`
row and backfills `curriculumId`. The four live callers are `gb.ts`, `cbse.ts`,
`caie-igcse.ts`, `ib.ts`. The six subjects-only Arab nationals
(`sa.ts`…`jo.ts`) share `seedSubjectsOnly`.

### Generated content

`catalog/content.ts` reads the published hierarchy and **generates synthetic**
rows in bulk: exams (per subject/chapter/lesson), a question bank (~10/subject,
6/chapter, 4/lesson), materials (metadata only — `fileUrl` is null), and one
assignment per lesson. A subjects-only curriculum gets only per-subject rows. The
exception is `catalog/exam-templates.ts`, which writes real hand-authored
blueprints + templates (idempotent, every subject). The `Quiz` model is not
seeded. **As of this session, `content.ts` is idempotent** — see
[Operations](#operations).

### Adding a national curriculum

Sudan predates the generic pipeline (TOC-driven seed). A new national curriculum
(e.g. Egypt `EG`) goes through the tree engine:

1. Author `curriculum/eg/g{N}/<subject>/structure.json`.
2. Add a one-line `prisma/seeds/catalog/eg.ts` calling `syncCurriculumTree` with
   `country: "EG", code: "EG"`, and register it in `prisma/seeds/single.ts`.
3. Add `EG → "EG"` to `inferCurriculum()` and `COUNTRY_CURRICULUM`
   (`src/lib/timetable-reference.ts`).
4. Add Arabic names to `SUBJECT_CONCEPT_BY_NAME` in `concepts-data.ts`.
5. (Optional) add an `EG` entry to `CURRICULUM_ACADEMIC_CONFIG` if it differs from
   the shared Arabic 6+3+3 default.
6. Run `pnpm db:seed:single eg` then `pnpm db:seed:single concepts`.

## Concepts

A **concept** is a curriculum-agnostic identifier that maps every subject to
shared visual assets. Arabic (Sudan) and World Languages (US) share the same
`languages` concept and the same S3 images. The artwork was originally referenced
from an external library during prototyping; those assets are now **owned,
redrawn, and shared across every curriculum** — see [Origins](#origins).

```
Subject (any curriculum)
  └── concept: "math"
        ├── thumbnail: catalog/concepts/g{grade}-math/thumbnail-{size}.webp
        ├── banner:    catalog/concepts/g{grade}-math/banner-{size}.webp
        └── cover:     catalog/concepts/math/cover
```

### The 23 concepts

`arts`, `biology`, `career-tech`, `celebrations`, `chemistry`, `civics`,
`computer-science`, `earth-science`, `economics`, `english`, `geography`,
`health`, `history`, `languages`, `life-skills`, `math`, `pe`, `physics`,
`psychology`, `religion`, `science`, `sociology`, `teacher-pd`.

Source images live in `public/subjects/concepts/{concept}.jpg`. The `concepts`
seed processes each into 4 WebP variants (sm/md/lg/original) and uploads to S3 for
all 12 grades. All 12 curricula map their subjects to one of these 23 concepts;
the `Image` model (`catalog_images`) stores the per-variant records.

### One source of truth — `concepts-data.ts`

Everything concept-related lives in `src/components/catalog/concepts-data.ts`: the
23 concepts and colors, the exact-match maps (`SUBJECT_CONCEPT_BY_SLUG`,
`SUBJECT_CONCEPT_BY_NAME`), the rotation pools (`CONCEPT_POOL`), and a
`nearestConcept()` keyword heuristic (EN + AR) so **no subject is ever
image-less** (a flat color is the last resort). Every seed (`us.ts`, `sd.ts`,
`engine.ts`, `concepts.ts`) imports from it — no per-seed copies remain.

### Chapter rotation pools

Chapters within a subject rotate through a pool of 5 related concepts for visual
variety (cycled by `sequenceOrder % poolSize`): e.g. `math` → math, science,
computer-science, physics, economics; `science` → science, biology, chemistry,
physics, earth-science. The deep seeds and `syncCurriculumTree` inherit this; for
subjects-only curricula, chapter/lesson images come for free once depth lands.

### Variants

| Variant          | Width  | Quality  | Use             |
| ---------------- | ------ | -------- | --------------- |
| `-sm.webp`       | 200px  | 95       | Grid thumbnails |
| `-md.webp`       | 600px  | 97       | Card images     |
| `-lg.webp`       | 1200px | 100      | Hero banners    |
| `-original.webp` | Full   | Lossless | Full resolution |

`image.ts` (server-only — `processAndUploadCatalogImage`, `deleteCatalogImage`)
runs Sharp; `image-url.ts` (client-safe — `getCatalogImageUrl`) builds
`{key}-{size}.webp` at read time. URL resolution: CloudFront from `thumbnailKey` →
concept-based static fallback → `null` (color background).

## CDN & Assets

Catalog assets follow a strict key convention and are served from S3 via
CloudFront. The full platform CDN reference (bucket, `asset()` helper, the 5 flat
UI-asset directories) is at **[CDN & Assets](/docs/cdn)** — this section covers
only the catalog-specific asset story.

`Subject` carries four S3 key fields — `thumbnail` (`@map("thumbnailKey")`),
`banner` (`@map("bannerUrl")`), `cover` (textbook cover), `pdf` (textbook PDF).
The naming convention distinguishes a **`*Key`** (raw S3 key, variants appended at
read time) from a **`*Url`** (a fully-formed URL, e.g. external book covers). S3
layout:

```
catalog/concepts/g{n}-{concept}/thumbnail-{sm,md,lg,original}.webp
catalog/concepts/g{n}-{concept}/banner-{sm,md,lg,original}.webp
catalog/concepts/{concept}/cover
catalog/textbooks/{slug}/textbook.pdf
catalog/subjects/{slug}/…/thumbnail-{size}.webp     # US lesson/chapter covers
```

### Honest asset-gap table

| Asset                       | Produced by               | Status                                                                                          |
| --------------------------- | ------------------------- | ----------------------------------------------------------------------------------------------- |
| Subject thumbnails          | `concepts.ts`             | **Complete** — all 12 curricula, all 12 grades                                                  |
| Banners                     | `banners.ts`              | **Partial** — only ~10 of 23 concepts; other `banner` keys 404                                  |
| Textbook covers             | —                         | **Missing** — `cover` key set on every subject, object never uploaded (card falls back to icon) |
| Textbook PDFs               | `upload-textbooks-all.ts` | **Sudan only** — 118 `textbook.pdf`; no other curriculum                                        |
| Lesson / chapter thumbnails | `lesson-covers.ts`        | **US only** — pulled from a CDN at seed time                                                    |
| Videos                      | `videos.ts` / contributed | **Placeholder** rows (S3 keys with no backing `.mp4`) + externally-hosted contributed links     |
| Books                       | `books.ts`                | **External** — 51 books on `covers.openlibrary.org` URLs (not on our CDN)                       |
| Materials                   | `content.ts`              | **Metadata-only** — `fileUrl` is null                                                           |

## Operations

### Seed commands

`seedFullCatalog` (the `full` seed) runs everything in canonical order — all 12
curricula → registry → concept images. The default `catalog` seed is the fast
subset: US + the full Sudan, registered. Every other curriculum is its own
explicit run. **Always use `pnpm db:seed:single <name>`** — never the full seed
(it is manual-only; see the rule below).

| Seed (`pnpm db:seed:single …`)                    | Contents                                                                    |
| ------------------------------------------------- | --------------------------------------------------------------------------- |
| `full`                                            | `seedFullCatalog` — all 12 curricula → registry → concept images (fresh DB) |
| `catalog`                                         | Fast default: US (~240) + full Sudan + registry                             |
| `us` / `sd`                                       | US (inventory) / full Sudan from `curriculum/sd/`                           |
| `gb` / `cbse` / `caie-igcse` / `ib`               | Deep curricula via `syncCurriculumTree` from `curriculum/<dir>/`            |
| `sa` / `eg` / `ae` / `qa` / `kw` / `jo`           | The 6 subjects-only nationals (`seedSubjectsOnly`)                          |
| `registry`                                        | All 12 `Curriculum` records + backfill `Subject.curriculumId`               |
| `concepts` / `banners`                            | Concept thumbnails & banners → Sharp → S3/CloudFront                        |
| `lesson-covers`                                   | US lesson/chapter thumbnails (CDN-dependent)                                |
| `books` / `content` / `videos` / `exam-templates` | Books, generated content, videos, exam blueprints/templates                 |
| `demo`                                            | Academic structure + bridge rows for the demo school                        |

```bash
pnpm db:seed:single --list        # list available seeds
pnpm db:seed:single catalog       # fast default: US + full Sudan + registry
pnpm db:seed:single gb            # England — deep, from curriculum/uk
pnpm db:seed:single sa            # Saudi Arabia — subjects-only national
pnpm db:seed:single registry      # (re)link all 12 Curriculum records
pnpm db:seed:single concepts      # thumbnails → S3/CDN
pnpm tsx scripts/upload-textbooks-all.ts   # Sudan textbook PDFs → S3
```

### Idempotency

Most seeds are upsert-safe (image seeds re-upload to immutable keys, also safe).
**The content seed is now idempotent too** (fixed this session): `content.ts`
computes a deterministic `seedKey` from each row's scope + title, pre-loads the
existing keys, and drops candidates that already exist — so re-running it no longer
duplicates exams/questions/materials/assignments. The `Quiz` model is still not
seeded.

### Deploy-pending live-DB sync

Canonical codes are **live**, but the deep-curricula breadth is shipped in code
and not yet in the live DB. At the next catalog deploy, in order:

1. Delete the stale subjects-only rows superseded by the deep trees:
   `gb-national-g*` (old world GB, ~156 rows) and `ib-diploma-g*` (old world IB,
   ~114 rows).
2. Re-run the catalog seeds in order: `gb`, `cbse`, `caie-igcse`, `ib`, then
   `concepts` / `banners`. (Cambridge IGCSE and CBSE have **no rows in the live DB
   yet**.)

### NEVER run the full seed

Per project rule, **never run `pnpm db:seed`** — the full seed is manual-only.
Always use `pnpm db:seed:single <name>`. Destructive DB operations (re-running
migration SQL, `db push --accept-data-loss`, `migrate reset`) are forbidden
without explicit approval.

dependency.
