The notifications module delivers messages across four channels (in-app, email, push, SMS) with per-user preferences, RBAC, batch operations, delivery tracking, and a polling fallback when Socket.IO is unavailable. Every domain dispatcher routes through `dispatchNotification()` so preference checking and expiration are uniform.

## Routes

| Route                        | Roles | Purpose                                   |
| ---------------------------- | ----- | ----------------------------------------- |
| `/notifications`             | all   | Notification center with list and filters |
| `/notifications/preferences` | all   | Per-type per-channel preference toggles   |

## Recipient map

19 wired notification types across 14 domains. The full mapping is the source of truth for "who gets what" — read it from the canonical table in `src/components/school-dashboard/notifications/recipient-map.ts`. Highlights:

- `attendance_alert` — five trigger points (mark absent, threshold cron, excuse submit / decide, teacher unmarked cron). Recipients vary: guardians, admins, teachers.
- `grade_posted`, `report_ready` — students + guardians; triggered from grades or exams actions.
- `fee_due` / `fee_paid` / `fee_overdue` — students + guardians; cron drives overdue.
- `system_alert` — catch-all for membership lifecycle, payroll, exam scheduling, substitutions, application status. Recipients depend on the trigger.
- `account_created` — onboarding submissions go to admins; membership approvals and enrollments go to the user.
- `message`, `message_mention` — conversation participants and mentioned users.

Four types are defined in the enum but **not yet dispatched**: `attendance_marked`, `password_reset`, `login_alert`, `document_shared`. Wire them in their respective actions when the use case lands.

## Architecture

`src/lib/dispatch-notification.ts` is the single entry point. Two functions:

```typescript
// single user — used by domain dispatchers
await dispatchNotification({
  userId,
  schoolId,
  type,
  priority,
  title,
  body,
  metadata,
  actorId,
})

// bulk by audience — school / class / role scoped
await dispatchNotificationsToAudience({
  schoolId,
  type,
  priority,
  title,
  body,
  scope: { type: "school" }, // or { type: "class", classId } or { type: "role", role }
})
```

`dispatchNotification()` checks `NotificationPreference`, picks channels, sets `expiresAt` from config, then writes the row. `dispatchNotificationsToAudience()` resolves users in scope, filters by preference, and uses `createMany({ skipDuplicates: true })`.

Domain dispatchers across 17 integration points all call `dispatchNotification()`. No domain code touches `db.notification.create` directly.

Real-time delivery uses Socket.IO with rooms keyed by `user:${userId}`. The client singleton is `src/lib/websocket/socket-service.ts`. **The Socket.IO server is external to this repo** — without it, the client falls back to 30-second polling.

## Preferences

`NotificationPreference` is unique on `[userId, type, channel]`. Fields: `enabled`, `quietHoursStart`/`quietHoursEnd` (0–23), `digestEnabled`, `digestFrequency` (`daily` | `weekly`).

The preferences page at `/notifications/preferences` renders a toggle matrix — 23 rows (types) × 4 columns (channels) plus quiet-hours and digest controls. Dispatch checks the user's preference for the matching `(type, channel)` row; missing preference falls back to role defaults.

## Schema

`Notification` is the row. Indexes are tuned for the main queries: `[schoolId, userId, read, createdAt]` (default), `[schoolId, type]` (filter), `[schoolId, createdAt]` (analytics), per-channel sent flags for queue draining, and `[expiresAt]` for cleanup.

```prisma
model Notification {
  id        String                 @id @default(cuid())
  schoolId  String
  userId    String
  type      NotificationType
  priority  NotificationPriority   @default(normal)
  title     String
  body      String                 @db.Text
  metadata  Json?
  actorId   String?
  read      Boolean                @default(false)
  readAt    DateTime?
  channels  NotificationChannel[]
  expiresAt DateTime?

  emailSent Boolean? emailSentAt DateTime? emailError String?
  pushSent  Boolean? pushSentAt  DateTime? pushError  String?
  smsSent   Boolean? smsSentAt   DateTime? smsError   String?

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@index([schoolId, userId, read, createdAt])
  @@index([schoolId, type])
  @@index([schoolId, createdAt])
  @@index([schoolId, emailSent])
  @@index([schoolId, pushSent])
  @@index([schoolId, smsSent])
  @@index([expiresAt])
}
```

Supporting models — `NotificationTemplate`, `NotificationBatch`, `NotificationSubscription`, `NotificationSummary` — are schema only (no rendering pipeline, no batch processor, no digest cron). Source: `prisma/models/notifications.prisma`.

Enums: `NotificationType` (23 values), `NotificationPriority` (`low` | `normal` | `high` | `urgent`), `NotificationChannel` (`in_app` | `email` | `push` | `sms`).

Three hooks expose the data, all dual-channel (Socket.IO when available, 30-second polling fallback), with optimistic updates:

```typescript
const {
  isConnected,
  unreadCount,
  recentNotifications,
  markAsRead,
  markAllAsRead,
} = useNotifications({
  autoConnect: true,
  showToast: true,
  pollInterval: 30000,
})

useNotificationBell() // count + 5 recent, no toast
useNotificationCenter() // all recent + toasts
```
