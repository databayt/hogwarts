## Overview

The Notifications module provides multi-channel notification delivery (in-app, email, push, SMS) with per-user preferences, RBAC authorization, batch operations, delivery tracking, and real-time updates via Socket.IO. It spans 7 Prisma models, 19 component files, 23 notification types, and 4 delivery channels.

### Key Features

| Feature                    | Status          | Description                                                 |
| -------------------------- | --------------- | ----------------------------------------------------------- |
| In-App Notifications       | **Production**  | Bell icon with badge, notification center, mark as read     |
| Email Delivery             | **Production**  | Resend SDK, HTML with RTL support, delivery logging         |
| Notification Preferences   | **Production**  | Per-user per-type per-channel toggles, quiet hours          |
| Batch Notifications        | **Production**  | Bulk create by role, class, or user IDs                     |
| Delivery Tracking          | **Production**  | Per-channel status (sent/failed/bounced/delivered/opened)   |
| RBAC Authorization         | **Production**  | Role-based create, read, batch permissions                  |
| Notification Subscriptions | **Production**  | Subscribe to specific entities (assignment, class)          |
| Expiration & Cleanup       | **Production**  | Type-based expiration (1-90 days), cron cleanup             |
| Grade Notifications        | **Production**  | Dispatched from grades actions                              |
| Exam Notifications         | **Production**  | Dispatched from exam actions (scheduled, reminder, results) |
| Attendance Notifications   | **Production**  | Absence alerts, excuse flow, teacher reminders, policy cron |
| Fee Notifications          | **Production**  | Fee assigned, paid, overdue (cron)                          |
| Assignment Notifications   | **Production**  | Created, due in 24h (cron)                                  |
| Membership Notifications   | **Production**  | Role change, suspend, remove, approve, reject               |
| Messaging Notifications    | **Production**  | New message, @mention, added to conversation                |
| Onboarding Notifications   | **Production**  | Application submitted → admin alert                         |
| Event Reminders            | **Production**  | Cron-based reminders for events in next 24h                 |
| Polling Fallback           | **Production**  | 30s interval when WebSocket unavailable                     |
| Real-Time (Socket.IO)      | **Wired**       | Client singleton exists, no server — no real-time delivery  |
| Push Notifications         | **Disabled**    | Service worker exists, no VAPID keys or subscription mgmt   |
| SMS Delivery               | **Disabled**    | Twilio SDK coded, `SMS_ENABLED=false` by default            |
| Template Rendering         | **Schema Only** | NotificationTemplate model exists, no rendering pipeline    |
| Digest Generation          | **Schema Only** | NotificationSummary model exists, no cron generates digests |
| Scheduled Batches          | **Schema Only** | NotificationBatch.scheduledFor exists, no processor         |

### Dispatch Pattern

The dispatch layer is the central entry point for creating notifications. It runs at the system level (no user session required) and handles preference checking, channel selection, and expiration.

```typescript
// src/lib/dispatch-notification.ts

// Single user dispatch — used by domain dispatchers
await dispatchNotification({
  userId: student.guardianId,
  schoolId: school.id,
  type: "attendance_alert",
  priority: "high",
  title: "Absence Alert",
  body: `${student.name} was marked absent today`,
  metadata: {
    studentId: student.id,
    date: today,
    link: "/attendance/records", // deep linking
  },
  actorId: teacher.id,
})
// → Checks NotificationPreference for this user+type
// → Creates Notification with channels: ["in_app"]
// → Sets expiresAt: now + 7 days (from config)

// Bulk dispatch — used for school-wide or class-wide
await dispatchNotificationsToAudience({
  schoolId: school.id,
  type: "announcement",
  priority: "normal",
  title: "School Event Tomorrow",
  body: "Annual sports day starts at 8 AM",
  scope: { type: "school" }, // or { type: "class", classId } or { type: "role", role: "GUARDIAN" }
})
// → Resolves all users in scope
// → Filters out users with disabled preferences
// → createMany with skipDuplicates
```

## Best Practices

### For Developers

1. Always use `dispatchNotification()` or `dispatchNotificationsToAudience()` -- never create Notification records directly
2. Include `metadata.link` for deep linking to the relevant page
3. Set appropriate priority -- `urgent` triggers special UI treatment (pulsing badge)
4. Use audience dispatch for class-wide or school-wide notifications
5. Add new notification types to the `NotificationType` enum, `config.ts` icons/expiration, and both dictionary files

### For Integrating New Features

When adding notification dispatch to a new feature:

```typescript
// 1. Import the dispatcher
import { dispatchNotification } from "@/lib/dispatch-notification"

// 2. Call after the business action succeeds
export async function markAttendance(input: MarkAttendanceInput) {
  // ... mark attendance logic ...

  // 3. Dispatch for absent students
  for (const record of absentRecords) {
    const guardians = await getGuardiansForStudent(record.studentId)
    for (const guardian of guardians) {
      await dispatchNotification({
        userId: guardian.id,
        schoolId,
        type: "attendance_alert",
        priority: "high",
        title: "Absence Alert",
        body: `${student.name} was marked absent`,
        metadata: { studentId: record.studentId, link: "/attendance/records" },
        actorId: session.user.id,
      })
    }
  }
}
```
