title: Content
description: Canonical patterns for writing content.tsx in the mirror pattern.
---

`content.tsx` is the file that the route's `page.tsx` delegates to. It owns data fetching (server) or hook orchestration (client) and composes children. Pages stay thin — auth + tenant + render `<XxxContent />`. Real logic lives in `content.tsx`.

## Two flavours

- **Server content (~111 files).** Async function. Fetches via `queries.ts` or `db`. No `"use client"`. Passes pre-computed data to client sub-components (`table.tsx`, `form.tsx`).
- **Client content (~114 files).** `"use client"`. Uses hooks, state, browser APIs. Fetches via server actions from `actions.ts`.

Decision tree:

| Needs                       | Use                                               |
| --------------------------- | ------------------------------------------------- |
| Hooks, events, browser APIs | Client content                                    |
| Data fetching + composition | Server content                                    |
| Both                        | Server content that renders client sub-components |

## Eight rules

1. **`export default [async] function XxxContent()`.** Named function, default export. Name matches the feature.
2. **`import type` for type-only imports.** No runtime cost.
3. **Stay under ~150 lines.** Anything more should split into sub-components.
4. **Typed dictionary props.** Use `Dictionary["school"]["grades"]`, never `any`. Type the slice.
5. **All user-facing text comes from the dictionary.** No hardcoded English. No `isArabic ? "..." : "..."` ternaries.
6. **Use `lang: Locale`, not `locale`.** Stay consistent with `[lang]` route segment.
7. **Server content extracts queries to `queries.ts` and uses try/catch.** Don't inline Prisma in content.
8. **Client content fetches via server actions from `actions.ts`.** Don't import `db` from a client component (TypeScript blocks it; the rule is about discipline before the error).

## Templates

| Template           | Reference                                           |
| ------------------ | --------------------------------------------------- |
| Server listing     | `school-dashboard/listings/students/content.tsx`    |
| Server dashboard   | `school-dashboard/dashboard/content.tsx`            |
| Client interactive | `school-dashboard/attendance/manual/content.tsx`    |
| Server composition | `school-dashboard/profile/content.tsx`              |

### Server listing — minimal shape

```tsx
import type { Locale } from "@/components/internationalization/dictionaries"
import type { Dictionary } from "@/components/internationalization/types"

import { getStudentList } from "./queries"
import { StudentsTable } from "./table"

interface Props {
  searchParams: Promise<Record<string, string | string[] | undefined>>
  dictionary: Dictionary["school"]["students"]
  lang: Locale
}

export default async function StudentsContent({
  searchParams,
  dictionary,
  lang,
}: Props) {
  const params = await searchParams
  const students = await getStudentList(params)
  return <StudentsTable data={students} dictionary={dictionary} lang={lang} />
}
```

## Anti-patterns

- **Monolith content** — anything over 300 lines should split. The 893-line `exams/content.tsx`, the 739-line `attendance/analytics`, the 612-line `stream/courses` are the canonical bad examples.
- **`dictionary: any`** — type the slice. 18+ files do this; fix when you touch them.
- **Dictionary accepted but ignored** — 8+ files accept `dictionary` and reference zero keys. Either use it or don't accept it.
- **Inline Prisma queries** — 50+ lines of query logic in `content.tsx` belongs in `queries.ts`.
- **Inline `isArabic` ternaries** — use the dictionary slice.
- **`db` imports in `page.tsx`** — pages stay thin. Move to `content.tsx` (server) or `queries.ts`.
- **Wrong `"use client"`** — only present when client logic is needed.
- **Inline SVGs** — extract icon paths to a separate file (or use the icon registry).
- **Duplicate type definitions** — share types via `types.ts`. Don't redeclare `School` in two content files.
- **Sequential `await` for permission checks** — parallelise with `Promise.all`.
- **`console.log` left in** — never ship debug output.

## Relationship to siblings

- `page.tsx` — auth + tenant + `<XxxContent />`. Nothing else.
- `queries.ts` — server-side database queries that `content.tsx` (server) consumes.
- `actions.ts` — server actions that `content.tsx` (client) consumes.
- `form.tsx` / `table.tsx` — client leaves rendered by content.
- `types.ts` — shared types for the feature.

## Naming

- Filename: `content.tsx`. Always.
- Function: `<Feature>Content` in PascalCase.
- Default export. Named exports for sub-components only.
