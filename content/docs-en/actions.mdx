---
title: Actions
description: Canonical patterns for writing actions.ts server actions in feature directories.
---

`actions.ts` files run on the server and mutate state. They authenticate, resolve tenant, check permissions, validate input with Zod, execute the database write, then revalidate the cache. Read-only queries belong in `queries.ts`, not here.

## Categories

| Category              | Shape                                                               | Reference                                          |
| --------------------- | ------------------------------------------------------------------- | -------------------------------------------------- |
| CRUD                  | Create, read, update, delete on a Prisma model. 200–1,300 lines.    | `listings/grades/actions.ts` (661)                 |
| Form step             | Validate a single multi-step-form step, no DB write. 15–50 lines.   | `school-marketing/application/personal/actions.ts` |
| Integration           | External API calls (Stripe, AI, email). Wraps third-party SDKs.     | `billing/actions.ts` (1,056)                       |
| SaaS / platform admin | DEVELOPER-only operations on tenants and platform-wide records.     | `saas-dashboard/tenants/actions.ts`                |
| Auth                  | Pre-tenant flows; uses singular `action.ts`. No `getTenantContext`. | `auth/login/action.ts`                             |
| File / upload         | Signed URL minting, S3 confirms. Uses `auth()` + `schoolId` claim.  | `file/upload/actions.ts`                           |

## Eight rules

1. **`"use server"` at the top.** Required for every action file.
2. **The 5-step flow.** Auth → tenant → permission → validate → execute + revalidate. See the template below.
3. **Import `ActionResponse` from `@/lib/action-response`.** Don't redefine it. The shared shape is `{ success: true, data } | { success: false, error?, errorCode? }`.
4. **Validate with Zod.** Schemas live in `validation.ts`. Don't accept raw `FormData` without parsing.
5. **Mutations only.** Read functions belong in `queries.ts`. Never mix.
6. **Wrap in `try/catch` with structured error returns.** Surface `errorCode` for translation; never throw to the client.
7. **Include `schoolId` in every query.** Missing `schoolId` = tenant data leak.
8. **Stay under ~300 lines.** Beyond that, split into an `actions/` subdirectory by capability (`crud.ts`, `notifications.ts`, `bulk.ts`).

```typescript
"use server"

import { revalidatePath } from "next/cache"
import { auth } from "@/auth"

import type { ActionResponse } from "@/lib/action-response"
import { db } from "@/lib/db"
import { getTenantContext } from "@/lib/tenant-context"

import { canCreate } from "./authorization"
import { subjectCreateSchema, type SubjectCreateInput } from "./validation"

export async function createSubject(
  input: SubjectCreateInput
): Promise<ActionResponse<{ id: string }>> {
  try {
    // 1. authenticate
    const session = await auth()
    if (!session?.user?.id) return { success: false, errorCode: "UNAUTHORIZED" }

    // 2. tenant
    const { schoolId } = await getTenantContext()
    if (!schoolId) return { success: false, errorCode: "MISSING_TENANT" }

    // 3. permission
    if (!canCreate(session.user.role)) {
      return { success: false, errorCode: "FORBIDDEN" }
    }

    // 4. validate
    const parsed = subjectCreateSchema.safeParse(input)
    if (!parsed.success) {
      return { success: false, errorCode: "INVALID_INPUT" }
    }

    // 5. execute and revalidate
    const subject = await db.subject.create({
      data: { ...parsed.data, schoolId },
      select: { id: true },
    })
    revalidatePath("/subjects")
    return { success: true, data: { id: subject.id } }
  } catch (error) {
    console.error("createSubject error", error)
    return { success: false, errorCode: "SERVER_ERROR" }
  }
}
```

## Templates

Pick the closest reference and adapt:

| Template    | File                                                              |
| ----------- | ----------------------------------------------------------------- |
| CRUD        | `src/components/school-dashboard/listings/grades/actions.ts`      |
| Form step   | `src/components/school-marketing/application/personal/actions.ts` |
| SaaS admin  | `src/components/saas-dashboard/tenants/actions.ts`                |
| Integration | `src/components/billing/actions.ts`                               |

The CRUD template typically exports `create`, `update`, `delete`, plus optional `bulkCreate`, `bulkDelete`. Each one runs the 5-step flow. Form-step actions skip auth/tenant when the step runs before login (e.g. application form pre-OTP).

## Anti-patterns

- **Redefining `ActionResponse` locally** — 27+ files do this. Import from `@/lib/action-response` instead.
- **Monolith files** — anything over 300 lines should split. The 5,692-line `timetable/actions.ts` is the canonical bad example; replace with an `actions/` subdirectory.
- **Duplicate function names across files** — `createPeriod`, `getInvoices`, `setActiveTerm` defined in 2–4 places. Pick a home, re-export if needed.
- **`console.log` of auth data** — never log emails, tokens, or session content. The `login/action.ts` and verification flows had 21 such logs.
- **Raw `FormData.get() as string`** — always parse with Zod first.
- **`any` types** — 150+ in the codebase. Use the inferred Zod type or a Prisma payload type.
- **Read functions** — they belong in `queries.ts`. Move them.
- **Mixing read + write** — split the file.

## Naming

`actions.ts` (plural) for feature directories; `action.ts` (singular) only for the auth flows that pre-date the convention. Don't introduce new singular files.

## Relationship to other files

- Imports schemas + types **from** `validation.ts`.
- Imports permission helpers **from** `authorization.ts`.
- Doesn't import from `form.tsx` or `table.tsx`. Actions don't know about UI.
- Action files in features **never** import from another feature's actions; cross-feature coordination happens in `src/lib/` or via shared queries.

## Error codes

Return error codes, not English strings. The dictionary maps them on the client via `useI18nMessages(dictionary).error`. Common codes:

| Code             | Meaning                              |
| ---------------- | ------------------------------------ |
| `UNAUTHORIZED`   | `auth()` failed                      |
| `MISSING_TENANT` | No `schoolId` on the session         |
| `FORBIDDEN`      | Permission check failed              |
| `INVALID_INPUT`  | Zod validation failed                |
| `NOT_FOUND`      | Record doesn't exist or other tenant |
| `CONFLICT`       | Unique constraint violation          |
| `SERVER_ERROR`   | Unexpected exception                 |

Return shape:

```typescript
type ActionResponse<T> =
  | { success: true; data: T }
  | { success: false; errorCode: ErrorCode; error?: string }
```

The `error` field is for developer diagnostics only — never display it to users.
