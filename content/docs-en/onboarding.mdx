title: Onboarding
description: Multi-step wizard for school registration, configuration, and business setup. Mirror pattern, 16 steps in 3 groups.

The onboarding system guides new schools through registration, configuration, and business setup. It follows the mirror pattern and spans ~166 files across 16 steps in 3 groups, with `WizardLayout` driving navigation. Completion takes 10–30 minutes for typical schools.

## Steps

### Group 1 — Basic information

| Step         | Path            | Required | Database fields                         |
| ------------ | --------------- | -------- | --------------------------------------- |
| About school | `/about-school` | No       | static welcome                          |
| Title        | `/title`        | Yes      | `School.name`                           |
| Description  | `/description`  | Yes      | `schoolType`, `schoolLevel`, `planType` |
| Location     | `/location`     | Yes      | `address`, `city`, `state`, `country`   |
| Stand out    | `/stand-out`    | No       | static                                  |

### Group 2 — School setup

| Step         | Path            | Required | Database fields                            |
| ------------ | --------------- | -------- | ------------------------------------------ |
| Capacity     | `/capacity`     | Yes      | `maxStudents`, `maxTeachers`, `maxClasses` |
| Branding     | `/branding`     | No       | `primaryColor`, `borderRadius`, `shadow`   |
| Import       | `/import`       | No       | bulk insert (CSV / Excel)                  |
| Finish setup | `/finish-setup` | No       | static review                              |

The `schedule` step is defined in `OnboardingStep` and has its own component directory but is **not** in the active navigation. It lives at `order: 5.5` in `config.ts` for future activation.

### Group 3 — Business and legal

| Step       | Path          | Required | Database fields                                               |
| ---------- | ------------- | -------- | ------------------------------------------------------------- |
| Join       | `/join`       | Yes      | `User`                                                        |
| Visibility | `/visibility` | No       | `School.isPublished`                                          |
| Price      | `/price`      | Yes      | `tuitionFee`, `registrationFee`, `applicationFee`, `currency` |
| Discount   | `/discount`   | No       | `Discount` model                                              |
| Legal      | `/legal`      | Yes      | `LegalConsent`                                                |
| Subdomain  | `/subdomain`  | No       | `School.domain`                                               |

After all steps the user lands on `/congratulations`, which performs a smart redirect (school subdomain dashboard or main domain depending on session state).

The active step order is `ONBOARDING_CONFIG` in `src/components/form/footer.tsx`. The authoritative metadata (labels, validation, group, order) is `ONBOARDING_STEPS` in `src/components/onboarding/config.ts`.

## Auth integration

The flow is a four-stage handshake:

1. **Start** — User visits `/get-started` or `/onboarding` while unauthenticated. `start-onboarding` action persists intent in a cookie and redirects to `/login`.
2. **Authenticate** — Standard login. After auth, the callback URL routes the user back to `/onboarding`.
3. **Provision** — `initializeSchoolSetup()` calls `ensureUserSchool()` from `src/lib/school-access.ts`, which runs an atomic Prisma `$transaction` to create the `School` row, the `SchoolMember` row binding the user as ADMIN, and to seed default `YearLevel`s. Idempotent: if a school already exists for this user, returns it.
4. **Refresh and redirect** — `_sessionRefreshRequired: true` instructs the client to refetch `/api/auth/session`, then `_redirect` sends the user to the next step.

Ownership is enforced by `requireSchoolOwnership(id)` on every step's load. Cross-tenant access raises `TenantError`.

## Multi-tenant integration

- **Subdomain routing** — `/onboarding` and child routes live on the main domain (`ed.databayt.org`), not on a subdomain. Subdomain assignment happens at the Subdomain step.
- **Database scoping** — every read uses `requireSchoolOwnership(id)` first, then queries scoped by `schoolId`.
- **Session schoolId** — the JWT carries `schoolId` after step 1; subsequent steps trust it after re-verification.

## Validation

Each step has a Zod schema co-located with its component. Schemas use the i18n factory pattern so the same schema produces Arabic or English error messages depending on the dictionary slice passed in.

```typescript
import { createTitleSchema } from "@/components/onboarding/title/validation"

const schema = createTitleSchema(dictionary)
const result = schema.safeParse(input)
```

The root `validation.ts` exports `validateStep(step, data, dictionary)` and `getRequiredFieldsForStep(step)` for shared step-aware validation.

## Auto-provisioning

When `completeOnboarding()` runs at the legal step, the school is marked active and a 5-step provisioning chain creates the academic infrastructure. Each step is wrapped in `try/catch` so one failure doesn't block the rest.

Source: `src/components/onboarding/legal/actions.ts` → `src/lib/catalog-setup.ts`.

| Step                            | Creates                                                               |
| ------------------------------- | --------------------------------------------------------------------- |
| `setupDefaultsForSchool()`      | YearLevels (8 / 6 / 14 by school level), 6 Departments, 9 ScoreRanges |
| `setupCatalogForSchool()`       | Subject / chapter / lesson bridges from the global catalog            |
| `setupTermsForSchool()`         | Default terms for the academic year                                   |
| `setupRolesForSchool()`         | Permission templates for the 8 roles                                  |
| `setupNotificationsForSchool()` | Default notification preferences for ADMIN role                       |

`setupCatalogForSchool` resolves the curriculum via `inferCurriculum()`:

| Priority | Strategy                             | Example                            |
| -------- | ------------------------------------ | ---------------------------------- |
| 1        | International school type → `us-k12` | any country + `international`      |
| 2        | Country exact match                  | `US` → `us-k12`, `SD` → `national` |
| 3        | Unmapped country → `us-k12`          | `JP`, `KE`, …                      |

## API routes

| Path                              | Method | Purpose                                                                |
| --------------------------------- | ------ | ---------------------------------------------------------------------- |
| `/api/onboarding/create-school`   | POST   | Idempotent atomic school creation; handles P2002 unique violation race |
| `/api/onboarding/validate-access` | POST   | Pre-flight access check; auto-provisions; returns redirect on mismatch |
| `/api/onboarding/extract`         | POST   | AI document extraction (10 MB max; JPEG / PNG / WebP / PDF)            |

## Server actions

All in `src/components/onboarding/actions.ts`:

| Action                               | Purpose                                                               |
| ------------------------------------ | --------------------------------------------------------------------- |
| `initializeSchoolSetup()`            | Creates or retrieves school for the current user; returns `_redirect` |
| `getListing(id)`                     | Fetch school with ownership verification                              |
| `updateListing(id, data)`            | Update fields for a step                                              |
| `getSchoolSetupStatus(schoolId)`     | Completion percentage based on 6 checks                               |
| `reserveSubdomainForSchool(id, sub)` | Reserve a subdomain (delegates to `src/lib/subdomain-actions.ts`)     |
| `getCurrentUserSchool()`             | Returns the current user's `schoolId`                                 |
| `getUserSchools()`                   | Lists schools owned by or associated with the current user            |
| `proceedToTitle(schoolId)`           | Validates ownership and redirects to the about-school step            |

## Troubleshooting

| Symptom                             | Cause                                          | Fix                                                             |
| ----------------------------------- | ---------------------------------------------- | --------------------------------------------------------------- |
| Build hangs at "Environments: .env" | TypeScript errors in onboarding components     | Run `pnpm tsc --noEmit`                                         |
| Session missing `schoolId`          | Session not refreshed after atomic transaction | Fetch `/api/auth/session` after `_sessionRefreshRequired: true` |
| Cross-tenant access denied          | User trying to access a school they don't own  | `requireSchoolOwnership(id)` throws `TenantError`               |
| Subdomain already taken             | Another school reserved it                     | Action returns `errorCode: "DOMAIN_TAKEN"`                      |
| Form data not persisting            | Missing `revalidatePath()` after mutation      | All update actions call `revalidatePath("/onboarding")`         |
| Dictionary property not found       | Missing translation key                        | Use optional chaining: `dictionary?.onboarding?.title?.label`   |
