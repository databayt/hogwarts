## Overview

The Profile system provides a GitHub-inspired profile experience for all school users. Each user gets a profile page with a contribution graph, pinned items, role-specific tabs, and an activity timeline. The system uses an **optional catch-all route** to handle both own-profile and view-by-ID in a single page file.

### Key Concepts

- **GitHub-style layout**: Left sidebar (avatar, bio, stats) + right content area (tabs, contribution graph, pinned items)
- **Single catch-all route**: `/profile` (own) and `/profile/<user-id>` (other) both handled by `profile/[[...id]]/page.tsx`
- **Permission-based filtering**: What you see depends on your role and relationship to the profile owner
- **Real contribution data**: Contribution graph queries Prisma for attendance, assignments, and activities

## Route Structure

| Route                | Page File                    | Behavior                     |
| -------------------- | ---------------------------- | ---------------------------- |
| `/profile`           | `profile/[[...id]]/page.tsx` | Own profile (`isOwner=true`) |
| `/profile/<user-id>` | `profile/[[...id]]/page.tsx` | View by ID (`isOwner=false`) |

The school route lives under `src/app/[lang]/s/[subdomain]/(school-dashboard)/profile/[[...id]]/`.

**Removed routes**: The old directory routes (`/profile/students`, `/profile/teachers`, `/profile/parents`, `/profile/staff`) and the separate `/profile/[id]` route were deleted during the catch-all refactor.

## Data Model

### User as Identity Hub

The `User` model serves as the identity hub with 1:1 optional relations to role-specific models:

```
User
├── Student?      (1:1, via userId)
├── Teacher?      (1:1, via userId)
├── Guardian?     (1:1, via userId)
└── StaffMember?  (1:1, via userId)
```

### GitHub-Style Fields on User

The `User` model includes fields inspired by GitHub profiles:

| Field           | Type      | Purpose                         |
| --------------- | --------- | ------------------------------- |
| `bio`           | `String?` | Short biography (max 500 chars) |
| `website`       | `String?` | Personal website URL            |
| `timezone`      | `String?` | User timezone                   |
| `statusEmoji`   | `String?` | Status emoji (max 10 chars)     |
| `statusMessage` | `String?` | Status message (max 100 chars)  |
| `pronouns`      | `String?` | User pronouns (max 50 chars)    |
| `socialLinks`   | `Json?`   | GitHub, Twitter, LinkedIn URLs  |

### UserActivity Model

Tracks user activities for the contribution graph. 16 activity types:

| Category       | Types                                                                           |
| -------------- | ------------------------------------------------------------------------------- |
| Academic       | `ASSIGNMENT_SUBMITTED`, `GRADE_RECEIVED`, `COURSE_ENROLLED`, `COURSE_COMPLETED` |
| Achievement    | `ACHIEVEMENT_EARNED`, `CERTIFICATE_EARNED`                                      |
| Library        | `LIBRARY_CHECKOUT`, `LIBRARY_RETURN`                                            |
| Events         | `EVENT_ATTENDED`, `CLUB_JOINED`                                                 |
| Projects       | `PROJECT_CREATED`, `PROJECT_UPDATED`                                            |
| Administrative | `ATTENDANCE_MARKED`, `EXAM_COMPLETED`, `PROFILE_UPDATED`, `OTHER`               |

### PinnedItem Model

Users can pin up to 6 items to their profile. 10 item types:

`COURSE`, `SUBJECT`, `PROJECT`, `ACHIEVEMENT`, `CERTIFICATE`, `CLASS`, `CHILD`, `DEPARTMENT`, `PUBLICATION`, `TASK`

Each pinned item has `title`, `description`, `metadata` (JSON), `isPublic` (boolean), and `order` (int).

## Permission System

### 5 Permission Levels

```
OWNER > ADMIN > STAFF > RELATED > PUBLIC
```

### How `getPermissionLevel()` Resolves Access

```typescript
// detail/permissions.ts
function getPermissionLevel(context: ProfileContext): PermissionLevel {
  if (!viewerId)                           → "PUBLIC"
  if (viewerId === profileUserId)          → "OWNER"
  if (role is DEVELOPER or ADMIN)          → "ADMIN"
  if (role is TEACHER, STAFF, ACCOUNTANT)  → "STAFF"
  if (same schoolId)                       → "RELATED"
  default                                  → "PUBLIC"
}
```

### Data Visibility by Level

| Level       | Sees                                                                                                          |
| ----------- | ------------------------------------------------------------------------------------------------------------- |
| **OWNER**   | Everything (full profile, private pinned items, all settings)                                                 |
| **ADMIN**   | Everything (same as owner for administrative purposes)                                                        |
| **STAFF**   | Name, email, photo, school data, year levels, classes, guardians, medical info, phone numbers, qualifications |
| **RELATED** | Name, photo, student ID, enrollment date, year levels, teacher email, classes                                 |
| **PUBLIC**  | Name and photo only                                                                                           |

### Field-Level Filtering

`filterProfileData()` applies per-level filtering. Each role model (student, teacher, guardian) has its own filter function that strips sensitive fields based on the viewer's permission level.

## Server Actions

### Main Actions (`actions.ts`)

| Action                                    | Purpose                                                                    |
| ----------------------------------------- | -------------------------------------------------------------------------- |
| `getStudentProfile(id?)`                  | Fetch student with classes, year levels, guardians                         |
| `getTeacherProfile(id?)`                  | Fetch teacher with departments, classes, students                          |
| `getParentProfile(id?)`                   | Fetch guardian with children and their classes                             |

| Action                   | Purpose                                                      |

The contribution graph queries real Prisma data per role:

```typescript

//   0 contributions → level 0

//   3-4             → level 2

```

### Server Action Pattern

All actions follow the standard pattern:

```typescript
export async function getProfileBasicData(userId: string, lang?: string) {
  const session = await auth()
  if (!session?.user?.id) return { success: false, error: "Not authenticated" }

  const { schoolId } = await getTenantContext()
  if (!schoolId) return { success: false, error: "School context not found" }

  const user = await db.user.findFirst({
    where: { id: userId, schoolId },
    select: {
      /* role-specific fields */
    },
  })

  // Build flat data, translate if needed
  return { success: true, data }
}
```

## Achievements System

### Achievement Badges

The profile sidebar displays role-relevant achievement badges using GitHub-style badge PNGs from `/github/\{badge-name\}.png`. Each badge opens a popover (similar to GitHub's badge detail card) showing the achievement title, description, earned date, and context. Per-badge background images are used for the popover header.

### Achievement Type

```typescript
// types.ts
interface Achievement {
  id: string
  title: string
  description: string
  icon: string // Badge name, maps to /github/{icon}.png
  earnedAt?: Date | string
  level?: "bronze" | "silver" | "gold" | "platinum"
  context?: string // History detail (e.g., "Term 1, 2025-2026 · Grade 10 Mathematics")
}
```

### Badge Names

The `icon` field maps to GitHub achievement badge PNGs: `starstruck`, `galaxy-brain`, `pull-shark`, `yolo`, `pair-extraordinaire`, `quickdraw`, `public-sponsor`.

### AchievementPopover Component

Located in `sidebar.tsx`. Renders a `Popover` (from shadcn/ui) with:

- **Trigger**: `size-16` (64px) badge PNG via `<img>`, cursor-pointer, no hover scale effect
- **Content** (`w-[360px]`, `p-0`, `overflow-hidden`, `side="top"`):
  - Header: Per-badge gradient background PNG from `achievementDetailBg` map (e.g., `/github/starstruck-detail.png`), badge image centered at `size-[140px]` (140px)
  - Title + description: `<h3>` with `text-base font-bold`, description in `text-sm text-muted-foreground`
  - Separator: `<hr className="mx-4 mt-3">`
  - History section: `OcticonTrophy` icon (muted) + "Unlocked on \{date\}" text, bullet + context string

### Known Issues

- **Directory routes removed**: `/profile/students`, `/profile/teachers`, `/profile/parents`, `/profile/staff` routes and their ~60 associated component files were deleted during the catch-all refactor (see [Cleanup History](#cleanup-history))
- **Parent contribution payments query disabled**: In `actions.ts`, the parent payment query uses `take: 0`, effectively returning no results
- **Hardcoded tab badge counts**: All tab counts in `ROLE_TABS` are static numbers, not queried from the database
- **No SWR hooks**: The old `hooks.ts` with SWR-based hooks was deleted (referenced non-existent `/api/profile/*` routes); only server actions remain
- **Avatar upload placeholder**: `uploadProfileAvatar()` has no actual file upload implementation
- **Settings update no-op**: `updateProfileSettings()` has an empty `data` object in the Prisma update
