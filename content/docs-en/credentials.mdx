## Overview

This document traces every email/password (credentials) flow in Hogwarts: registration, email verification, login with 2FA branching, password reset, and the supporting token/email infrastructure.

For OAuth providers (Google, Facebook), multi-tenant redirect logic, session management, and cross-subdomain SSO, see [Authentication](/docs/authentication).

## Registration (Join)

**Source files**: `src/components/auth/join/form.tsx`, `src/components/auth/join/action.ts`

### How it works

1. Client submits `{ email, password, username }` to `register()` server action
2. Validates against `RegisterSchema` (email format, password min 6, username min 1)
3. `getUserByEmail(email)` checks for existing user via `findMany` (prefers user with password)
4. If duplicate: returns `"Email already in use!"`
5. `bcrypt.hash(password, 10)` and `db.user.create()` with default `role: USER`, no `schoolId`
6. `generateVerificationToken(email)` creates a UUID token with 24h expiry
7. `sendVerificationEmail()` sends link: `/{locale}/new-verification?token={uuid}`
8. Returns `"Confirmation email sent!"` (success)

### Error messages

| Message                      | Type    | Trigger                     |
| ---------------------------- | ------- | --------------------------- |
| `"Invalid fields!"`          | error   | Schema validation failed    |
| `"Email already in use!"`    | error   | User with this email exists |
| `"Confirmation email sent!"` | success | Registration completed      |

## Email Verification

**Source files**: `src/components/auth/verification/form.tsx`, `src/components/auth/verification/action.ts`, `src/components/auth/verification/verificiation-token.ts`

### How it works

1. User clicks email link, lands on `/new-verification?token={uuid}`
2. `NewVerificationForm` calls `newVerification(token)` on mount via `useCallback`
3. Looks up token with `getVerificationTokenByToken(token)`
4. If token not found: checks if user already verified (fallback), otherwise `"Token does not exist!"`
5. Checks `new Date(existingToken.expires) < new Date()` for expiry
6. `getUserByEmail(existingToken.email)` to find the user
7. If already verified: returns `"Email verified!"` (idempotent)
8. Updates `user.emailVerified = new Date()` and also sets `user.email = existingToken.email`
9. Deletes the used token
10. Returns `"Email verified!"` - page shows success + "Back to login" link (no auto-redirect)

### Error messages

| Message                     | Type    | Trigger                                 |
| --------------------------- | ------- | --------------------------------------- |
| `"Token does not exist!"`   | error   | Token not found in DB                   |
| `"Token has expired!"`      | error   | Token past expiry date                  |
| `"Email does not exist!"`   | error   | No user for token's email               |
| `"Email verified!"`         | success | First-time or repeat verification       |
| `"Email already verified!"` | success | Token missing but user already verified |

## Login

The login flow is a 10-step process with a 2FA branch.

**Source files**: `src/components/auth/login/form.tsx`, `src/components/auth/login/action.ts`, `src/auth.config.ts`

### Step-by-step

1. **Validate** - `LoginSchema.safeParse({ email, password, code? })`
2. **Brute force check** - `isBruteForceBlocked(email)` counts failures in last 15 min, blocks at >= 5
3. **User lookup** - `getUserByEmail(email)` uses `findMany` + prefers user with password (multi-tenant design)
4. **Exists check** - If no user, no email, or no password: logs `USER_NOT_FOUND`, returns `"Email does not exist!"`
5. **Email verified?** - If `!emailVerified`, regenerates verification token, sends email, returns `"Confirmation email sent!"` (blocks login)
6. **2FA branch** (if `isTwoFactorEnabled`):
   - **No code submitted**: generates 6-digit token (5min TTL), sends via email, returns `{ twoFactor: true }` — client shows code input
   - **Code submitted**: validates token exists, matches, not expired. On success: deletes token, deletes old `TwoFactorConfirmation`, creates new one. On failure: returns `"Invalid code!"` or `"Code expired!"`
7. **Build redirect URL** - Smart redirect logic based on context, callbackUrl, role, and subdomain (see [Authentication](/docs/authentication) for full redirect logic)
8. **Sign in** - `signIn("credentials", { email, password, redirectTo })` which calls the Credentials provider authorize function
9. **Credentials authorize** (`auth.config.ts`): `LoginSchema` parse, `getUserByEmail`, `bcrypt.compare` — returns user or null
10. **Error handling** - `AuthError` type `CredentialsSignin` → `"Invalid credentials!"`, NEXT_REDIRECT re-thrown (expected)

### Error messages

| Message                                                       | Type    | Trigger                                |
| ------------------------------------------------------------- | ------- | -------------------------------------- |
| `"Invalid fields!"`                                           | error   | Schema validation failed               |
| `"Too many failed attempts. Please try again in 15 minutes."` | error   | Brute force block (5+ failures)        |
| `"Email does not exist!"`                                     | error   | No user, no email, or no password      |
| `"Confirmation email sent!"`                                  | success | Unverified email — resent verification |
| `"Invalid code!"`                                             | error   | 2FA token missing or mismatch          |
| `"Code expired!"`                                             | error   | 2FA token past 5min expiry             |
| `"Invalid credentials!"`                                      | error   | bcrypt.compare failed                  |
| `"Something went wrong!"`                                     | error   | Other AuthError types                  |
| `"An unexpected error occurred. Please try again."`           | error   | Non-AuthError exception                |

## Password Reset

Two-step flow: request reset email, then set new password.

### Step 1: Request Reset

**Source files**: `src/components/auth/reset/form.tsx`, `src/components/auth/reset/action.ts`

1. Client submits `{ email }` to `reset()` server action
2. Validates against `ResetSchema` (email format)
3. `getUserByEmail(email)` — if not found: `"Email not found!"`
4. `generatePasswordResetToken(email)` creates UUID token with 1h expiry
5. `sendPasswordResetEmail()` sends link: `/{locale}/new-password?token={uuid}`
6. Returns `"Reset email sent!"`

| Message               | Type    | Trigger                                         |
| --------------------- | ------- | ----------------------------------------------- |
| `"Invalid emaiL!"`    | error   | Schema validation failed (note: typo in source) |
| `"Email not found!"`  | error   | No user with this email                         |
| `"Reset email sent!"` | success | Token generated and email sent                  |

### Step 2: Set New Password

**Source files**: `src/components/auth/password/form.tsx`, `src/components/auth/password/action.ts`, `src/components/auth/password/token.ts`

1. Client submits `{ password }` + token from URL to `newPassword()` server action
2. If no token: `"Missing token!"`
3. Validates against `NewPasswordSchema` (password min 6)
4. `getPasswordResetTokenByToken(token)` — if not found: `"Invalid token!"`
5. Checks `new Date(existingToken.expires) < new Date()` — if expired: `"Token has expired!"`
6. `getUserByEmail(existingToken.email)` — if not found: `"Email does not exist!"`
7. `bcrypt.hash(password, 10)` and `db.user.update({ data: { password: hashedPassword } })`
8. Deletes the used token
9. Returns `"Password updated!"`

| Message                   | Type    | Trigger                           |
| ------------------------- | ------- | --------------------------------- |
| `"Missing token!"`        | error   | No token in URL                   |
| `"Invalid fields!"`       | error   | Password schema validation failed |
| `"Invalid token!"`        | error   | Token not found in DB             |
| `"Token has expired!"`    | error   | Token past 1h expiry              |
| `"Email does not exist!"` | error   | No user for token's email         |
| `"Password updated!"`     | success | Password changed successfully     |

## Two-Factor Authentication

2FA is integrated into the login flow, not a standalone feature. When `user.isTwoFactorEnabled` is true, login requires an additional 6-digit code.

### Token generation

```typescript
// src/components/auth/tokens.ts
const token = crypto.randomInt(100_000, 1_000_000).toString() // 6-digit numeric
const expires = new Date(new Date().getTime() + 5 * 60 * 1000) // 5 minutes
```

### Lifecycle

1. Login detects `isTwoFactorEnabled` and no code → generates token, sends email, returns `{ twoFactor: true }`
2. Client shows code input field
3. User enters code, resubmits login form
4. Login validates code against `TwoFactorToken` record (exact match + not expired)
5. Deletes the `TwoFactorToken`
6. Deletes existing `TwoFactorConfirmation` (if any)
7. Creates new `TwoFactorConfirmation` for the user
8. Proceeds with normal `signIn("credentials")`

### Email format

The 2FA email contains the code inline (no link):

| Locale | Subject      | Body                                 |
| ------ | ------------ | ------------------------------------ |
| en     | `2FA Code`   | `Your 2FA code: {token}`             |
| ar     | `رمز التحقق` | `رمز التحقق الخاص بك: {token}` (RTL) |

**Source files**: `src/components/auth/tokens.ts`, `src/components/auth/verification/2f-token.ts`, `src/components/auth/verification/2f-confirmation.ts`

## Token System

All tokens follow a **delete-before-create** pattern: if an existing token exists for the email, it is deleted before generating a new one. This ensures only one active token per email at any time.

| Token              | Format          | TTL  | Generator                    | DB Model             |
| ------------------ | --------------- | ---- | ---------------------------- | -------------------- |
| VerificationToken  | UUID v4         | 24h  | `generateVerificationToken`  | `VerificationToken`  |
| PasswordResetToken | UUID v4         | 1h   | `generatePasswordResetToken` | `PasswordResetToken` |
| TwoFactorToken     | 6-digit numeric | 5min | `generateTwoFactorToken`     | `TwoFactorToken`     |

**Source file**: `src/components/auth/tokens.ts`

## Brute Force Protection

**Source file**: `src/lib/audit-log.ts`

### `isBruteForceBlocked(email, ip?)`

Counts `LoginAttempt` records where `success: false` in the last 15 minutes. Blocks at >= 5 failures.

```typescript
const recentFailures = await db.loginAttempt.count({
  where: {
    email,
    success: false,
    timestamp: { gte: fifteenMinutesAgo },
  },
})
return recentFailures >= 5
```

### `logLoginAttempt(params)`

Records every login attempt (success or failure) with: `email`, `success`, `failureReason`, `ip`, `userAgent`, `schoolId`.

**Note**: The `ip` parameter is accepted by `isBruteForceBlocked` but NOT used in the query — blocking is email-only. The `ip` field is only stored in `LoginAttempt` records for auditing.

## Validation Schemas

Two tiers exist: **legacy** schemas (server-side, English messages) and **i18n factory** functions (client-side, dictionary-translated messages).

### Legacy schemas (used in server actions)

| Schema              | Fields                                                            | Constraints                                           |
| ------------------- | ----------------------------------------------------------------- | ----------------------------------------------------- |
| `LoginSchema`       | email, password, code?                                            | email format, password min 1                          |
| `RegisterSchema`    | email, password, username                                         | email format, password min 6, username min 1          |
| `ResetSchema`       | email                                                             | email format                                          |
| `NewPasswordSchema` | password                                                          | min 6 chars                                           |
| `SettingsSchema`    | name?, email?, role, password?, newPassword?, isTwoFactorEnabled? | Coupled: newPassword requires password and vice versa |

### i18n factory functions (used in client forms)

| Factory                               | Creates                                            |
| ------------------------------------- | -------------------------------------------------- |
| `createLoginSchema(dictionary)`       | Same as LoginSchema with translated messages       |
| `createRegisterSchema(dictionary)`    | Same as RegisterSchema with translated messages    |
| `createResetSchema(dictionary)`       | Same as ResetSchema with translated messages       |
| `createNewPasswordSchema(dictionary)` | Same as NewPasswordSchema with translated messages |
| `createSettingsSchema(dictionary)`    | Same as SettingsSchema with translated messages    |

**Note**: `password/form.tsx` uses the legacy `NewPasswordSchema` directly, not the i18n factory.

**Source file**: `src/components/auth/validation.ts`

## Prisma Models

### User (credential-relevant fields)

```prisma
model User {
  id                    String    @id @default(cuid())
  email                 String?
  password              String?
  emailVerified         DateTime?
  isTwoFactorEnabled    Boolean   @default(false)
  role                  UserRole  @default(USER)
  schoolId              String?
  twoFactorConfirmation TwoFactorConfirmation?
  @@unique([email, schoolId])
}
```

### Token models

```prisma
model VerificationToken {
  id      String   @id @default(cuid())
  email   String
  token   String   @unique
  expires DateTime
  @@unique([email, token])
}

model PasswordResetToken {
  id      String   @id @default(cuid())
  email   String
  token   String   @unique
  expires DateTime
  @@unique([email, token])
}

model TwoFactorToken {
  id      String   @id @default(cuid())
  email   String
  token   String   @unique
  expires DateTime
  @@unique([email, token])
}

model TwoFactorConfirmation {
  id     String @id @default(cuid())
  userId String @unique
  user   User   @relation(fields: [userId], references: [id], onDelete: Cascade)
}
```

### LoginAttempt

```prisma
model LoginAttempt {
  id            String   @id @default(cuid())
  schoolId      String?
  email         String
  ip            String?
  userAgent     String?
  success       Boolean
  failureReason String?
  timestamp     DateTime @default(now())
  @@index([email, timestamp])
  @@map("login_attempts")
}
