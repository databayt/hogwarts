The catalog hierarchy is three global tables. All are scope-free (no `schoolId`); schools reference them via bridge tables.

## Schema

```
CatalogSubject (337 rows)
  ├── id, slug (unique), name, lang
  ├── country, curriculum          ← multi-curriculum routing
  ├── concept                      ← cross-curriculum image key
  ├── thumbnailKey                 ← S3/CDN path (unified image system)
  ├── grades[], levels[]
  ├── department, color, status
  ├── totalChapters, totalLessons  ← denormalized counts
  └── CatalogChapter (1,156 rows)
        ├── id, slug, name, lang
        ├── subjectId (FK)
        ├── sequenceOrder, grades[], levels[]
        ├── totalLessons, totalContent
        └── CatalogLesson (5,042 rows)
              ├── id, slug, name, lang
              ├── chapterId (FK)
              └── sequenceOrder, grades[], levels[]
```

Key fields for multi-curriculum routing:

```prisma
model CatalogSubject {
  country      String        @default("SD")        // ISO 3166-1 alpha-2
  curriculum   String        @default("national")  // us-k12, national, british, ib
  concept      String?                             // shared image key
  thumbnailKey String?                             // S3 key for WebP variants (unified)
  grades       Int[]         @default([])          // [5] = grade 5 only
  levels       SchoolLevel[]                       // [ELEMENTARY]
  slug         String        @unique               // sd-g5-math, us-g5-math
}
```

`slug` is the primary upsert key. It encodes country + grade + subject; no compound unique on `(country, curriculum, grade, name)` is needed.

## Runtime flow

### School onboarding

```
School.country = "SD"
  → inferCurriculum("SD") → "national"
  → findCatalogSubjects("SD", "national")
       Step 1  exact   (country + curriculum + schoolType)
       Step 2  broad   (country + curriculum)            ← SD hits here (97 subjects)
       Step 3  worldwide (country = "*")
       Step 4  fallback (US + us-k12)
  → filter by school.schoolLevel
       primary    → grades 1-6  (ELEMENTARY)
       secondary  → grades 7-12 (MIDDLE + HIGH)
       both       → grades 1-12
  → create SchoolSubjectSelection bridge rows
  → create YearLevel, Department, Stream, ScoreRange defaults
```

### Subject display

```
SchoolSubjectSelection.findMany({ schoolId })
  → join CatalogSubject (thumbnailKey, concept)
  → getCatalogImageUrl(thumbnailKey, "sm")
       1. thumbnailKey  → CDN: catalog/concepts/math/thumbnail-sm.webp
       2. concept       → static SUBJECT_IMAGE_MAP["math"] fallback
```

## Seed architecture

Two independent seeds, both idempotent (upsert).

| Seed                   | Source                  | Subjects | Chapters | Lessons | Slug                | Country | Curriculum | Lang |
| ---------------------- | ----------------------- | -------: | -------: | ------: | ------------------- | ------- | ---------- | ---- |
| `clickview-catalog.ts` | `master-inventory.json` |      240 |      750 |   3,993 | `us-g{N}-{subject}` | US      | `us-k12`   | en   |
| `sudan-catalog.ts`     | `sudan-curriculum.json` |       97 |      406 |   1,049 | `sd-g{N}-{subject}` | SD      | `national` | ar   |

A third seed `concept-images.ts` runs after both:

1. Reads `public/subjects/concepts/*.jpg` (24 images).
2. Sharp → 4 WebP variants → S3 upload.
3. Backfills `concept` field on subjects via `NAME_TO_CONCEPT`.
4. Sets `thumbnailKey` on subjects that don't have one.

## Scale analysis

| Metric        |              US |                SD |    Total |
| ------------- | --------------: | ----------------: | -------: |
| Subjects      |             240 |                97 |      337 |
| Chapters      |             750 |               406 |    1,156 |
| Lessons       |           3,993 |             1,049 |    5,042 |
| S3 thumbnails | 240 per-subject | 19 shared concept | 259 keys |
| WebP files    |   960 (240 × 4) |       96 (24 × 4) |    1,056 |

Adding Egypt (~30 subjects × 12 grades = ~360):

| Changes                           | Stays the same                                        |
| --------------------------------- | ----------------------------------------------------- |
| +360 `CatalogSubject` rows        | Schema                                                |
| +~1,500 chapters, +~5,000 lessons | No migration                                          |
| +0-5 new concept images           | Existing 24 cover most                                |
| New seed `egypt-catalog.ts`       | `catalog-setup.ts` already supports `EG → "national"` |

At 10 curricula × ~300 subjects each (~3,000 total), PostgreSQL handles it trivially. `findCatalogSubjects` is one indexed `findMany`; UI lists go via the `SchoolSubjectSelection` bridge; denormalized counts avoid N+1.

## Improvement areas

### Unified seed pattern

Two seeds share duplicated logic (upsert, denormalized count update, chapter/lesson loop). Extract `seedCurriculumFromJSON()`:

```ts
// prisma/seeds/shared/seed-curriculum.ts
interface CurriculumInput {
  country: string
  curriculum: string
  lang: string
  sortIdxStart: number
  dataPath: string
}

export async function seedCurriculumFromJSON(
  prisma: PrismaClient,
  input: CurriculumInput
): Promise<{ subjects: number; chapters: number; lessons: number }>
```

Prerequisite: normalize `master-inventory.json` to the same shape as `sudan-curriculum.json`. Current shapes differ.

### Canonical JSON schema

Define `curriculum.schema.json` as the contract:

```json
{
  "type": "object",
  "required": ["country", "curriculum", "lang", "subjects"],
  "properties": {
    "subjects": {
      "type": "array",
      "items": {
        "required": ["name", "slug", "department", "concept", "gradeChapters"],
        "properties": {
          "gradeChapters": {
            "type": "object",
            "additionalProperties": {
              "required": ["level", "chapters"],
              "properties": {
                "level": { "enum": ["elementary", "middle", "secondary"] }
              }
            }
          }
        }
      }
    }
  }
}
```

All future curricula write to this shape. ClickView normalizer converts its format once.

### Slug convention (unified)

All curricula use the same pattern: `{cc}-g{N}-{subject}`. Unified March 2026; the US K-12 seed migrates legacy `us-{subject}-grade-{N}` slugs automatically.

### Image layer unification (complete)

Both curricula now share the unified image system: `thumbnailKey` → concept static fallback. The legacy `imageKey` field has been dropped from `CatalogSubject`. The 240 US per-subject keys collapsed into 23 shared concept keys, matching the SD topology. See [/docs/concept](/docs/concept) for the canonical concept system.

### TOC extraction quality

Many Sudan PDFs (especially grades 6-8 and 10-12 secondary) produced garbled or empty `pdftotext` output. Subjects with chapters but 0 lessons:

| Grade range | With lessons | With 0 lessons |
| ----------- | -----------: | -------------: |
| 1-5         |     31 (89%) |        4 (11%) |
| 6-9         |      8 (40%) |       12 (60%) |
| 10-12       |     14 (29%) |       34 (71%) |

Priorities: OCR pass with Tesseract Arabic on scanned PDFs (grade 6) → manual entry for 10-12 — community contribution via the content layer.

### Denormalized count update performance

Today: ~97×3 + 406×2 ≈ 1,103 queries. Sudan seed took 822s, mostly on this loop. Replace with two bulk SQL statements:

```sql
UPDATE catalog_chapters SET
  "totalLessons" = sub.count,
  "totalContent" = sub.count
FROM (SELECT "chapterId", COUNT(*) AS count FROM catalog_lessons GROUP BY "chapterId") sub
WHERE catalog_chapters.id = sub."chapterId";

UPDATE catalog_subjects SET
  "totalChapters" = ch.count,
  "totalLessons"  = ls.count,
  "totalContent"  = ls.count
FROM
  (SELECT "subjectId", COUNT(*) AS count FROM catalog_chapters GROUP BY "subjectId") ch,
  (SELECT c."subjectId", COUNT(*) AS count FROM catalog_lessons l
   JOIN catalog_chapters c ON l."chapterId" = c.id GROUP BY c."subjectId") ls
WHERE catalog_subjects.id = ch."subjectId"
  AND catalog_subjects.id = ls."subjectId";
```

Expected: 822s → ~5s.
