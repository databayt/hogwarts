The admin admission flow runs from application review through enrollment confirmation. Confirming enrollment fires a single 30-second transaction that creates the Student, Guardians, FeeAssignments, and UserInvoices in one atomic step.

## Pipeline

```
Review application      → UNDER_REVIEW
Score & rank            → Merit list
Select applicant        → SELECTED (admissionOffered=true)
Confirm enrollment      → ADMITTED
                          → Student (ACTIVE)
                          → User (role=STUDENT)
                          → Guardians + StudentGuardian
                          → StudentYearLevel + AcademicGrade
                          → FeeAssignments + UserInvoices
                          → StudentDocuments
Place in section        → Classroom roster
Edit in wizard          → Update details
```

## Status flow

| Status                | Trigger                      | What happens                                      |
| --------------------- | ---------------------------- | ------------------------------------------------- |
| `DRAFT`               | Session created              | Form data being collected                         |
| `SUBMITTED`           | Applicant submits            | Application number generated, email sent          |
| `UNDER_REVIEW`        | Admin updates                | `reviewedAt` + `reviewedBy` set                   |
| `SHORTLISTED`         | Admin updates                | Applicant notified                                |
| `ENTRANCE_SCHEDULED`  | Admin schedules              | Entrance exam date set                            |
| `INTERVIEW_SCHEDULED` | Admin schedules              | Interview date set                                |
| `SELECTED`            | Admin selects                | Offer created (configurable expiry), payment link |
| `WAITLISTED`          | Admin waitlists              | Applicant notified                                |
| `REJECTED`            | Admin rejects                | Applicant notified                                |
| `ADMITTED`            | Admin confirms enrollment    | Student + Guardians + Fees + Invoices created     |
| `WITHDRAWN`           | Applicant or admin withdraws | Application closed                                |

## Enrollment transaction

`confirmEnrollment()` runs a single 30-second `db.$transaction`:

1. Set application status `ADMITTED` with enrollment number `ENR-{timestamp}-{random}`.
2. Create `User` (role `STUDENT`) if applicant was a guest; promote `USER` → `STUDENT` otherwise.
3. Create `Student` with application data pre-populated; match `applyingForClass` → `YearLevel` (exact name → case-insensitive → grade number → `levelOrder`); create `StudentYearLevel` and set `academicGradeId`.
4. Auto-assign active `FeeStructures` for the campaign's academic year → `FeeAssignment` rows (`PENDING`).
5. Create `Guardian` + `StudentGuardian` + `GuardianPhoneNumber` from father / mother data; copy application documents to `StudentDocument`.
6. Generate one `UserInvoice` per `FeeAssignment` via `createInvoiceFromEnrollment()` (non-fatal): number `ENR-{unique}`, status `UNPAID`, due 30 days; one `UserInvoiceItem` per fee structure.

Post-transaction (best-effort): notify student + guardians (in-app + email), dispatch `fee_due` notifications, return `studentId`. The student is created with `wizardStep: null` and `status: ACTIVE`.

| Application                                       | Student                            | Notes                              |
| ------------------------------------------------- | ---------------------------------- | ---------------------------------- |
| firstName / middleName / lastName                 | same                               | Direct copy                        |
| dateOfBirth, gender, nationality                  | same                               | Direct copy                        |
| email                                             | email                              | Direct copy                        |
| phone / alternatePhone                            | mobileNumber / alternatePhone      | Direct copy                        |
| address, city, state, postalCode, country         | currentAddress, …                  | Direct copy                        |
| photoUrl                                          | profilePhotoUrl                    | Profile picture                    |
| previousSchool / previousClass                    | previousSchoolName / previousGrade | Direct copy                        |
| previousMarks + previousPercentage + achievements | previousAcademicRecord             | Combined string                    |

| documents (JSON)                                  | StudentDocument records            | Each document creates a row        |
| applyingForClass                                  | academicGradeId                    | Via YearLevel → AcademicGrade      |

## Guardian auto-creation

For each parent (father, mother):

1. Split name into `firstName` + `lastName` (last space).
2. Upsert `GuardianType` (`father`, `mother`).
3. Create `Guardian` (or find by email).
4. Create `StudentGuardian` join row with occupation, `isPrimary`.
5. Create `GuardianPhoneNumber` if phone provided.

Father is primary; if no father, mother becomes primary.

## Section placement

After enrollment, place the student in a section:

- Enrollment response includes `suggestedSectionId` if exactly one section has capacity for the student's grade.
- Lists sections with available capacity for that grade.
- `placeStudentInSection()` assigns + auto-enrols in grade classes via `enrollStudentInGradeClasses()`.

## Wizard editing

Enrolled students can be edited via the student wizard (Students table → Actions → Edit). All application data is pre-populated.

`updateStudentWizardStep` only updates `wizardStep` for draft students (non-null `wizardStep`) — prevents accidental reversion of enrolled students. Final button reads "Save", not "Create".

## Merit list

Rank shortlisted / selected / waitlisted applications by configurable weights (must sum to 100):

- Academic score — 40%
- Entrance exam — 35%
- Interview — 25%

## Payment recording

Admins record cash / bank transfer payments to mark application fee paid. Currently sets `Application.applicationFeePaid` only — full `Payment` model integration is tracked separately.

## Multi-tenant safety

All queries scoped by `schoolId`. Submit flow resolves school from subdomain via `getSchoolBySubdomain()`; session is verified to belong to that school.
