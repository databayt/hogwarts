## Pipeline Overview (Admin Side)

```
Admin                              System
─────                              ──────
Review application ──────────────── UNDER_REVIEW
Score & rank ────────────────────── Merit list
Select applicant ────────────────── SELECTED
                                    (auto: admissionOffered=true)
Confirm enrollment ──────────────── ADMITTED
                                    → Student record (ACTIVE)
                                    → User (role=STUDENT)
                                    → Guardian records
                                    → StudentYearLevel
                                    → AcademicGrade
                                    → FeeAssignments
                                    → UserInvoice(s)
                                    → StudentDocuments
Place in section ────────────────── Classroom roster
Edit in wizard ──────────────────── Update details
                                    (all data pre-populated)
```

## Admin Review Flow

### Step 1: Review Applications

The admin logs into the school dashboard and navigates to **Admission → Applications**.

The applications table shows all submissions with columns: Application #, Applicant, Campaign, Class, Status, Merit Rank, Submitted Date.

**Actions per application:**

- **View Details** — full application view (opens detail page at `/{lang}/admission/applications/{id}`)
- **Update Status** — UNDER_REVIEW → SHORTLISTED → SELECTED → WAITLISTED → REJECTED
- **Send Email** — opens mailto link

When status is updated to SELECTED, the system auto-sets `admissionOffered = true` with an offer expiry read from `AdmissionSettings.offerExpiryDays` (default 14 days).

**Route:** `/{lang}/admission/applications`

### Step 2: Enrollment Confirmation

The admin confirms enrollment for selected students. This triggers a single transaction (30-second timeout) that creates a fully integrated student record:

1. Updates application status to `ADMITTED` with enrollment number (`ENR-{timestamp}-{random}`)
2. Creates a `User` record if the applicant was a guest (with `role: STUDENT`, `schoolId`)
3. Creates a `Student` record with **all** application data pre-populated (see Data Transfer below)
4. Matches `applyingForClass` to the correct `YearLevel` (cascading: exact name → case-insensitive → grade number extraction → levelOrder fallback)
5. Creates `StudentYearLevel` linking student to grade + school year
6. Sets `academicGradeId` on the student from the matched `YearLevel` → `AcademicGrade` link
7. Updates user role from `USER` to `STUDENT` and assigns `schoolId`
8. Auto-assigns active `FeeStructures` for the campaign's academic year → creates `FeeAssignment` records (status: `PENDING`)
9. Creates `Guardian` + `StudentGuardian` + `GuardianPhoneNumber` records from father/mother data
10. Copies application documents to `StudentDocument` records
11. **Auto-generates `UserInvoice`(s)** from fee assignments (non-fatal — enrollment succeeds even if invoice generation fails):
    - Creates one `UserInvoice` per `FeeAssignment` (not one combined invoice)
    - Invoice number format: `ENR-{unique}` via `generateUniqueInvoiceNumber()`
    - Status: `UNPAID`, due date: 30 days from enrollment
    - School name/address as "from", student name/email as "to"
    - Each invoice has one `UserInvoiceItem` per fee structure
    - Visible in Finance → Invoices immediately after enrollment

**After transaction (non-blocking, best-effort):**

12. Sends enrollment confirmation notification to student (in-app + email)
13. Dispatches `fee_due` notification to student for auto-assigned fees
14. Notifies guardians about enrollment (in-app + email)
15. Returns `studentId` for immediate navigation to the student profile

The student is created with `wizardStep: null` and `status: ACTIVE`, meaning they appear as a fully complete active student — not a draft.

### Step 3: Section Placement

After enrollment, the admin can place the student in a specific section:

- The enrollment response includes a `suggestedSectionId` if exactly one section has capacity for the student's grade
- Lists sections with available capacity for the student's grade
- Assigns student to section
- Auto-enrolls in grade classes via `enrollStudentInGradeClasses()`

### Step 4: Student Wizard (Edit Mode)

After enrollment, the admin can open the student record in the student wizard to review or edit any information. Since `wizardStep` is `null`, the student appears as "Active" in the students list (not "Draft").

The admin can navigate to the wizard from:

- Students table → Actions → Edit (opens wizard at the personal step)
- Enrollment table → View Student (after enrollment confirmation)

All application data is pre-populated in the wizard forms since it was written to the Student record during enrollment. The wizard serves as an **editing interface** for enrollment-created students — the admin can update or add information that wasn't collected during the application (e.g., health records, detailed previous education, section assignment).

**Safety:** When editing an enrolled student, the `updateStudentWizardStep` action only updates `wizardStep` for draft students (where `wizardStep` is non-null). This prevents enrolled students from being accidentally reverted to draft status when navigating between wizard steps. The wizard final button shows "Save" (not "Create") for all students.

### Step 5: Classroom & Finance Integration

Once the student is enrolled and placed in a section:

- **Classroom**: Student appears in classroom roster, timetable, attendance
- **Finance**: `FeeAssignment` records show in Finance → Fees. The admin can record payments, apply discounts, or scholarships
- **Invoices**: Auto-generated `UserInvoice` records show in Finance → Invoices
- **Profile**: Student has a profile page at `/profile/{userId}` accessible from the students table
- **Attendance**: Student is available for attendance marking in their assigned classes

## Admin Dashboard

### Admission Tabs

| Tab          | Route                            | Purpose                                  |
| ------------ | -------------------------------- | ---------------------------------------- |
| Campaigns    | `/{lang}/admission`              | Create/manage admission campaigns        |
| Applications | `/{lang}/admission/applications` | Review submitted applications            |
| Merit List   | `/{lang}/admission/merit`        | Rank applications by scores              |
| Enrollment   | `/{lang}/admission/enrollment`   | Confirm enrollment for selected students |
| Settings     | `/{lang}/admission/settings`     | Configure admission portal settings      |

There is also a detail route at `/{lang}/admission/applications/{id}` for viewing a single application.

### Merit List

The admin can generate a merit list that ranks shortlisted/selected/waitlisted applications by configurable weights:

- Academic score (default weight: 40%)
- Entrance exam score (default weight: 35%)
- Interview score (default weight: 25%)

Weights are configurable in Admission Settings and must sum to 100.

### Payment Recording

Admins can manually record payments for cash/bank transfer methods, marking the application fee as paid.

**Note:** Application fee payments currently update the `Application.applicationFeePaid` flag only. A future enhancement will bridge these into the `Payment` model for full accounting integration (requires making `Payment.feeAssignmentId` optional or creating a dedicated "application fee" FeeStructure).

### Invoice Auto-Generation

When enrollment is confirmed, the system auto-generates `UserInvoice` records from the student's fee assignments:

1. Finds all `PENDING` fee assignments for the student's academic year
2. Creates `UserInvoiceAddress` records for school (from) and student (to)
3. Creates one `UserInvoice` per fee assignment with a `UserInvoiceItem`
4. Invoices appear in **Finance → Invoices** immediately
5. Invoice notes: `"Auto-generated from enrollment"`

This is **non-fatal** — enrollment succeeds even if invoice generation fails. If it fails, the admin sees a warning message suggesting manual invoice creation from Finance → Invoices.

The invoice uses the `createInvoiceFromEnrollment()` utility from `finance/invoice/actions.ts`.

## Guardian Auto-Creation

For each parent in the application (father, mother):

1. Name is split into `firstName` + `lastName` (split on last space)
2. `GuardianType` is upserted (e.g., "father", "mother")
3. `Guardian` is created (or found by email if provided)
4. `StudentGuardian` join record links student to guardian with occupation, isPrimary
5. `GuardianPhoneNumber` is created if phone is provided

Father is marked as primary guardian. If no father, mother becomes primary.

## Multi-Tenant Safety

All queries are scoped by `schoolId`. The submit flow resolves school from subdomain:

```typescript
const schoolResult = await getSchoolBySubdomain(subdomain)
const schoolId = schoolResult.data.id

await db.application.create({
  data: { ...formData, schoolId },
})
```

The session is also verified to belong to the correct school before accepting submissions.

## Configuration

### Admission Settings

```typescript
{
  // General
  allowMultipleApplications: false,   // Allow same email to apply multiple times
  requireDocuments: true,             // Require document uploads
  applicationFee: 0,                  // In campaign currency (0 = free)
  offerExpiryDays: 14,               // Days before offer expires (1-90)

  // Notifications
  autoEmailNotifications: true,       // Send emails on status changes

  // Payments
  enableOnlinePayment: false,         // Enable Stripe payments
  paymentMethods: ["stripe", "cash"], // Available methods
  bankDetails: {                      // For bank transfer method
    bankName: "",
    accountName: "",
    accountNumber: "",
    iban: "",
    swiftCode: ""
  },
  cashPaymentInstructions: "",        // Instructions for cash payments

  // Merit Criteria (must sum to 100)
  academicWeight: 40,                 // Academic score weight
  entranceWeight: 35,                 // Entrance exam weight
  interviewWeight: 25                 // Interview score weight
}
```

## Status Reference

| Status                | Trigger                      | What Happens                                                        |
| --------------------- | ---------------------------- | ------------------------------------------------------------------- |
| `DRAFT`               | Session created              | Form data being collected                                           |
| `SUBMITTED`           | Applicant submits final step | Application number generated, email sent                            |
| `UNDER_REVIEW`        | Admin changes status         | `reviewedAt` + `reviewedBy` set                                     |
| `SHORTLISTED`         | Admin changes status         | Applicant notified                                                  |
| `ENTRANCE_SCHEDULED`  | Admin schedules              | Entrance exam date set                                              |
| `INTERVIEW_SCHEDULED` | Admin schedules              | Interview date set                                                  |
| `SELECTED`            | Admin selects                | Offer created (configurable expiry via settings), payment link sent |
| `WAITLISTED`          | Admin waitlists              | Applicant notified                                                  |
| `REJECTED`            | Admin rejects                | Applicant notified                                                  |
| `ADMITTED`            | Admin confirms enrollment    | Student + Guardian + Fees + Invoice(s) created                      |
| `WITHDRAWN`           | Applicant or admin withdraws | Application closed                                                  |
