## Admin Review Flow

### Step 1: Review Applications

The admin logs into the school dashboard and navigates to **Admission ‚Üí Applications**.

The applications table shows all submissions with columns: Application #, Applicant, Campaign, Class, Status, Merit Rank, Submitted Date.

**Actions per application:**

- **View Details** ‚Äî full application view
- **Update Status** ‚Äî UNDER_REVIEW ‚Üí SHORTLISTED ‚Üí SELECTED ‚Üí WAITLISTED ‚Üí REJECTED
- **Send Email** ‚Äî opens mailto link

When status is updated to SELECTED, the system auto-sets `admissionOffered = true` with a 30-day offer expiry.

**Route:** `/{lang}/admission/applications`

### Step 2: Enrollment Confirmation

The admin confirms enrollment for selected students. This triggers a single transaction that creates a fully integrated student record:

1. Updates application status to `ADMITTED` with enrollment number (`ENR-{timestamp}-{random}`)
2. Creates a `User` record if the applicant was a guest (with `role: STUDENT`, `schoolId`)
3. Creates a `Student` record with **all** application data pre-populated (see Data Transfer below)
4. Matches `applyingForClass` to the correct `YearLevel` (cascading: exact ‚Üí case-insensitive ‚Üí AcademicGrade.gradeNumber ‚Üí levelOrder fallback)
5. Creates `StudentYearLevel` linking student to grade + school year
6. Sets `academicGradeId` on the student from the matched `YearLevel` ‚Üí `AcademicGrade` link
7. Updates user role from `USER` to `STUDENT` and assigns `schoolId`
8. Auto-assigns active `FeeStructures` for the campaign's academic year ‚Üí creates `FeeAssignment` records
9. **Auto-generates `UserInvoice`** from fee assignments (non-fatal ‚Äî enrollment succeeds even if invoice generation fails):
   - Queries the fee assignments created in step 8
   - Creates a `UserInvoice` with one `UserInvoiceItem` per fee structure
   - Invoice number format: `ENR{YY}{NNN}` (e.g., `ENR26001`)
   - Status: `UNPAID`, due date: 30 days from enrollment
   - School name/address as "from", student name/email as "to"
   - Visible in Finance ‚Üí Invoices immediately after enrollment
10. Creates `Guardian` + `StudentGuardian` + `GuardianPhoneNumber` records from father/mother/guardian data
11. Copies application documents to `StudentDocument` records
12. Sends enrollment confirmation notification (in-app + email)
13. Returns `studentId` for immediate navigation to the student profile

The student is created with `wizardStep: null` and `status: ACTIVE`, meaning they appear as a fully complete active student ‚Äî not a draft.

### Step 3: Section Placement

After enrollment, the admin can place the student in a specific section:

- The enrollment response includes a `suggestedSectionId` if exactly one section has capacity for the student's grade
- Lists sections with available capacity for the student's grade
- Assigns student to section
- Auto-enrolls in grade classes via `enrollStudentInGradeClasses()`

### Step 4: Student Wizard (Edit Mode)

After enrollment, the admin can open the student record in the student wizard to review or edit any information. Since `wizardStep` is `null`, the student appears as "Active" in the students list (not "Draft").

The admin can navigate to the wizard from:

- Students table ‚Üí Actions ‚Üí Edit (opens wizard at the personal step)
- Enrollment table ‚Üí View Student (after enrollment confirmation)

All application data is pre-populated in the wizard forms since it was written to the Student record during enrollment. The wizard serves as an **editing interface** for enrollment-created students ‚Äî the admin can update or add information that wasn't collected during the application (e.g., health records, detailed previous education, section assignment).

**Safety:** When editing an enrolled student, the `updateStudentWizardStep` action only updates `wizardStep` for draft students (where `wizardStep` is non-null). This prevents enrolled students from being accidentally reverted to draft status when navigating between wizard steps. The wizard final button shows "Save" (not "Create") for all students.

### Step 5: Classroom & Finance Integration

Once the student is enrolled and placed in a section:

- **Classroom**: Student appears in classroom roster, timetable, attendance
- **Finance**: `FeeAssignment` records show in Finance ‚Üí Fees. The admin can record payments, apply discounts, or scholarships
- **Profile**: Student has a profile page at `/profile/{userId}` accessible from the students table
- **Attendance**: Student is available for attendance marking in their assigned classes

### Merit List

The admin can generate a merit list that ranks shortlisted/selected/waitlisted applications by:

- Merit score
- Entrance exam score
- Interview score

### Payment Recording

Admins can manually record payments for cash/bank transfer methods, marking the application fee as paid.

**Note:** Application fee payments currently update the `Application.applicationFeePaid` flag only. A future enhancement will bridge these into the `Payment` model for full accounting integration (requires making `Payment.feeAssignmentId` optional or creating a dedicated "application fee" FeeStructure).

### Invoice Auto-Generation

When enrollment is confirmed, the system auto-generates a `UserInvoice` from the student's fee assignments:

1. Queries all `PENDING` fee assignments for the student's academic year
2. Creates `UserInvoiceAddress` records for school (from) and student (to)
3. Creates `UserInvoice` with one `UserInvoiceItem` per fee structure
4. Invoice appears in **Finance ‚Üí Invoices** immediately

This is **non-fatal** ‚Äî enrollment succeeds even if invoice generation fails. The invoice uses the `createInvoiceFromEnrollment()` utility from `finance/invoice/actions.ts`.

## Guardian Auto-Creation

For each parent/guardian in the application (father, mother, optional guardian):

1. Name is split into `givenName` + `surname` (split on last space)
2. `GuardianType` is upserted (e.g., "father", "mother")
3. `Guardian` is created (or found by email if provided)
4. `StudentGuardian` join record links student to guardian with occupation, isPrimary
5. `GuardianPhoneNumber` is created if phone is provided

Father is marked as primary guardian. If no father, mother becomes primary.

## Multi-Tenant Safety

All queries are scoped by `schoolId`. The submit flow resolves school from subdomain:

```typescript
const schoolResult = await getSchoolBySubdomain(subdomain)
const schoolId = schoolResult.data.id

await db.application.create({
  data: { ...formData, schoolId },
})
```

The session is also verified to belong to the correct school before accepting submissions.

## Status Reference

| Status                | Trigger                      | What Happens                                     |
| --------------------- | ---------------------------- | ------------------------------------------------ |
| `DRAFT`               | Session created              | Form data being collected                        |
| `SUBMITTED`           | Applicant submits final step | Application number generated, email sent         |
| `UNDER_REVIEW`        | Admin changes status         | `reviewedAt` + `reviewedBy` set                  |
| `SHORTLISTED`         | Admin changes status         | Applicant notified                               |
| `ENTRANCE_SCHEDULED`  | Admin schedules              | Entrance exam date set                           |
| `INTERVIEW_SCHEDULED` | Admin schedules              | Interview date set                               |
| `SELECTED`            | Admin selects                | Offer created (30-day expiry), payment link sent |
| `WAITLISTED`          | Admin waitlists              | Applicant notified                               |
| `REJECTED`            | Admin rejects                | Applicant notified                               |
| `ADMITTED`            | Admin confirms enrollment    | Student + Guardian + Fees + Invoice created      |
| `WITHDRAWN`           | Applicant or admin withdraws | Application closed                               |
