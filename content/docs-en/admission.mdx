## Pipeline Overview

```
Applicant                          Admin                              System
────────                          ─────                              ──────
Visit school site ─────────────────────────────────────────────────────────
Select campaign ──────────────────────────────────────────────────────────
Complete 6-step wizard ───────────────────────────────────────────────────
Submit application ───────────────── Application created (SUBMITTED) ─────
Pay fee (card/cash/bank) ─────────── Payment recorded ────────────────────
                                   Review application ── UNDER_REVIEW ────
                                   Score & rank ──────── Merit list ──────
                                   Select applicant ──── SELECTED ────────
                                                         (auto: admissionOffered=true)
                                   Confirm enrollment ── ADMITTED ─────────
                                                         → Student record (ACTIVE)
                                                         → User (role=STUDENT)
                                                         → Guardian records
                                                         → StudentYearLevel
                                                         → AcademicGrade
                                                         → FeeAssignments
                                                         → StudentDocuments
                                   Place in section ──── Classroom roster ─
                                   Edit in wizard ────── Update details ───
                                                         (all data pre-populated)
```

## Application Flow (A to Z)

This section walks through the entire admission flow from the applicant's first click to the admin confirming enrollment.

### Step 1: Applicant Visits the School Site

The applicant navigates to the school's public site at `school.databayt.org` and clicks **Admissions** in the navigation. This leads to the admissions info page with a hero section, process overview, and a **Start Application** CTA.

**Route:** `/{lang}/admissions`

### Step 2: Campaign Selection

The applicant is taken to the apply landing page where they see active admission campaigns (e.g., "Admission 2026-2027"). They can also see any saved draft applications if they're logged in.

Two options are presented:

- **Start from scratch** — begin a new application
- **Import from profile** — auto-fill from documents or LinkedIn (same flow, pre-populated)

**Route:** `/{lang}/apply`

### Step 3: Overview Page

After selecting a campaign, the applicant sees an overview of the 3 phases:

| Phase | Label              | Steps                 |
| ----- | ------------------ | --------------------- |
| 1     | Basic Information  | Attachments, Personal |
| 2     | Details            | Contact, Location     |
| 3     | Family & Education | Guardian, Academic    |

Clicking **Get Started** initializes a session and enters the wizard.

**Route:** `/{lang}/apply/overview?id={campaignId}`

### Step 4: The 6-Step Wizard

### Step 5: Submit

When the applicant clicks "Submit Application", the system:

1. Saves the academic step data
2. Checks completeness across all steps (firstName, lastName, email, phone, fatherName, motherName, applyingForClass)
3. Flattens all step data into a single object
4. Calls `submitApplicationAction()` which delegates to `submitApplication()`

**Server-side submit flow:**

1. Resolve school from subdomain
2. Validate with `createFullApplicationSchema()` (Zod)
3. Verify session belongs to this school
4. Check campaign is still OPEN and not expired
5. Check for duplicate email + campaign combination
6. Generate unique application number (`APP-{year}-{random6}`)
7. Generate access token (32 chars, 6-month expiry)
8. Create `Application` record with status `SUBMITTED`
9. Mark session as converted
10. Send confirmation email via Resend
11. Check if payment is required (`campaign.applicationFee > 0`)

### Step 6: Payment (If Required)

If the campaign has an application fee, the applicant is redirected to the payment page.

| Payment Method | Description                              |
| -------------- | ---------------------------------------- |
| Pay with Card  | Stripe Checkout redirect                 |
| Pay at School  | Cash payment — shows reference number    |
| Bank Transfer  | Shows bank details with reference number |

After selecting cash or bank transfer, a confirmation card with the reference number is shown.

**Route:** `/{lang}/apply/{applicationId}/payment?number={applicationNumber}`

### Step 7: Success Page

Displays:

- Application number in a card
- Payment status (paid/pending)
- Next steps timeline (email confirmation → review → interview → decision)
- Links to track status and schedule a school tour

**Route:** `/{lang}/apply/{campaignId}/success?number={applicationNumber}`

### Step 8: Admin Reviews Application

The admin logs into the school dashboard and navigates to **Admission → Applications**.

The applications table shows all submissions with columns: Application #, Applicant, Campaign, Class, Status, Merit Rank, Submitted Date.

**Actions per application:**

- **View Details** — full application view
- **Update Status** — UNDER_REVIEW → SHORTLISTED → SELECTED → WAITLISTED → REJECTED
- **Send Email** — opens mailto link

When status is updated to SELECTED, the system auto-sets `admissionOffered = true` with a 30-day offer expiry.

**Route:** `/{lang}/admission/applications`

### Step 9: Enrollment Confirmation

The admin confirms enrollment for selected students. This triggers a single transaction that creates a fully integrated student record:

1. Updates application status to `ADMITTED` with enrollment number (`ENR-{timestamp}-{random}`)
2. Creates a `User` record if the applicant was a guest (with `role: STUDENT`, `schoolId`)
3. Creates a `Student` record with **all** application data pre-populated (see Data Transfer below)
4. Matches `applyingForClass` to the correct `YearLevel` (cascading: exact → case-insensitive → AcademicGrade.gradeNumber → levelOrder fallback)
5. Creates `StudentYearLevel` linking student to grade + school year
6. Sets `academicGradeId` on the student from the matched `YearLevel` → `AcademicGrade` link
7. Updates user role from `USER` to `STUDENT` and assigns `schoolId`
8. Auto-assigns active `FeeStructures` for the campaign's academic year → creates `FeeAssignment` records
9. Creates `Guardian` + `StudentGuardian` + `GuardianPhoneNumber` records from father/mother/guardian data
10. Copies application documents to `StudentDocument` records
11. Sends enrollment confirmation notification (in-app + email)
12. Returns `studentId` for immediate navigation to the student profile

The student is created with `wizardStep: null` and `status: ACTIVE`, meaning they appear as a fully complete active student — not a draft.

#### Guardian Auto-Creation

For each parent/guardian in the application (father, mother, optional guardian):

1. Name is split into `givenName` + `surname` (split on last space)
2. `GuardianType` is upserted (e.g., "father", "mother")
3. `Guardian` is created (or found by email if provided)
4. `StudentGuardian` join record links student to guardian with occupation, isPrimary
5. `GuardianPhoneNumber` is created if phone is provided

Father is marked as primary guardian. If no father, mother becomes primary.

### Step 10: Section Placement

After enrollment, the admin can place the student in a specific section:

- The enrollment response includes a `suggestedSectionId` if exactly one section has capacity for the student's grade
- Lists sections with available capacity for the student's grade
- Assigns student to section
- Auto-enrolls in grade classes via `enrollStudentInGradeClasses()`

### Step 11: Student Wizard (Edit Mode)

After enrollment, the admin can open the student record in the student wizard to review or edit any information. Since `wizardStep` is `null`, the student appears as "Active" in the students list (not "Draft").

The admin can navigate to the wizard from:

- Students table → Actions → Edit (opens wizard at the personal step)
- Enrollment table → View Student (after enrollment confirmation)

All application data is pre-populated in the wizard forms since it was written to the Student record during enrollment. The wizard serves as an **editing interface** for enrollment-created students — the admin can update or add information that wasn't collected during the application (e.g., health records, detailed previous education, section assignment).

**Safety:** When editing an enrolled student, the `updateStudentWizardStep` action only updates `wizardStep` for draft students (where `wizardStep` is non-null). This prevents enrolled students from being accidentally reverted to draft status when navigating between wizard steps. The wizard final button shows "Save" (not "Create") for all students.

### Step 12: Classroom & Finance Integration

Once the student is enrolled and placed in a section:

- **Classroom**: Student appears in classroom roster, timetable, attendance
- **Finance**: `FeeAssignment` records show in Finance → Fees. The admin can record payments, apply discounts, or scholarships
- **Profile**: Student has a profile page at `/profile/{userId}` accessible from the students table
- **Attendance**: Student is available for attendance marking in their assigned classes

## Session Management

### How Sessions Work

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  localStorage │ ←──→ │   Context    │ ←──→ │  Database    │
│  (backup)     │     │  (in-memory) │     │  (source of  │
│               │     │              │     │   truth)     │
└──────────────┘     └──────────────┘     └──────────────┘
```

**Storage key:** `hogwarts_apply_session_{campaignId}`

**Session state:**

- `sessionToken` — server-generated nanoid(32), null until first save
- `campaignId` — which admission campaign
- `formData` — keyed by step name (attachments, personal, contact, location, guardian, academic)
- `currentStep` — current step name
- `lastSaved`, `isDirty`, `isLoading`, `isSaving`, `error`

**Auto-save:** Every 30 seconds when `isDirty` is true.

**Session expiry:** 7 days.

**Resume methods:**

1. **localStorage** — automatic on page load
2. **Email link** — `/{lang}/apply/continue?token=xxx`
3. **User account** — authenticated users see their drafts on the apply landing page

**Unsaved changes:** The browser warns before navigating away when `isDirty` is true.

### First Save Email

On the first successful save, the system sends a "resume application" email with a token link so the applicant can continue from any device.

## Status Tracking

### OTP Verification

Parents can track their application status without creating an account:

1. Navigate to `/{lang}/apply/status`
2. Enter application number + email
3. Receive 6-digit OTP via email (10-minute expiry)
4. Rate limited: max 3 OTPs per hour, max 5 verification attempts
5. On verification, view application timeline

### Status Timeline

```
DRAFT → SUBMITTED → UNDER_REVIEW → SHORTLISTED → SELECTED → ADMITTED
                                  ↘ WAITLISTED
                                  ↘ REJECTED
                                  ↘ WITHDRAWN
```

The timeline shows each completed stage with dates and next steps.

## Multi-Tenant Safety

All queries are scoped by `schoolId`. The submit flow resolves school from subdomain:

```typescript
const schoolResult = await getSchoolBySubdomain(subdomain)
const schoolId = schoolResult.data.id

await db.application.create({
  data: { ...formData, schoolId },
})
```

The session is also verified to belong to the correct school before accepting submissions.

## Troubleshooting

**Application Not Saving**

- Check browser console for network errors
- Verify session hasn't expired (7-day limit)
- Ensure the school's subdomain resolves correctly

**Submit Fails with Validation Error**

- The error message includes field names (e.g., "Validation failed: postalCode: ...")
- Check that all required fields are filled across all steps
- The completeness check requires: firstName, lastName, email, phone, fatherName, motherName, applyingForClass

**OTP Not Received**

- Check spam folder
- Verify email matches the application
- Rate limit: max 3 OTPs per hour

**Payment Page Shows No Methods**

- Check `AdmissionSettings.paymentMethods` for the school
- Ensure campaign has `applicationFee > 0`
- Stripe requires webhook configuration for card payments
