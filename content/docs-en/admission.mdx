The admin admission flow runs from application review through enrollment confirmation. Confirming enrollment fires a single transaction that creates the Student, Guardians, FeeAssignments, and UserInvoices in one atomic step.

**Applying is always free.** The application wizard collects information only — no payment is taken at the application stage. Payment is triggered at two later points: the registration fee when the applicant accepts an offer, and tuition invoices after enrollment is confirmed.

## Student intake — four channels, one pipeline

A student can enter the system four ways. **All four funnel through one shared core, `provisionStudent`**, so however a student is added they get the same complete result — and every student is born from a real `Application`, tagged with the channel it came through.

    C1["Apply — public wizard<br/>(review → offer → confirm)"] --> Core

| Channel         | How a student is added                                                             | `Application.channel` |
| --------------- | ---------------------------------------------------------------------------------- | --------------------- |
| Apply           | Family completes the public application, staff review → offer → confirm enrollment | `PORTAL`              |
| Admin single    | An admin creates one student in the `/students` wizard                             | `ADMIN_DIRECT`        |
| Onboarding bulk | CSV upload while setting up the school                                             | `ONBOARDING_IMPORT`   |
| `/school/bulk`  | CSV upload later, from the dashboard                                               | `BULK_IMPORT`         |

The non-`PORTAL` channels create a "direct-admit" `Application` in the `ADMITTED` state, hung off a hidden per-school system campaign, so they never appear in the review queue. The Applications and Enrollment tabs **default to `PORTAL`** — a channel filter reveals the rest. Everything below describes the `PORTAL` (public application → enrollment) path in detail; the other three channels reuse the same `provisionStudent` step from enrollment onward.

## Enrollment transaction

`confirmEnrollment()` keeps the application-specific work (status guard, `ENR-{yy}-{seq}` enrollment number, the registration-fee `Payment` + ledger entry) and delegates the actual student creation to the shared **`provisionStudent()`** core (`src/lib/student-provisioning.ts`) — the same core the admin wizard and bulk import call. Inside one `db.$transaction`:

1. Set application status `ADMITTED` with the enrollment number.
2. `provisionStudent()`:
   - Resolve `AcademicGrade`, generate the per-school student code (`YYGGNNNN`).
   - Create `User` (role `STUDENT`) if the applicant was a guest; promote `USER` → `STUDENT` otherwise. **User is created before fees** — invoices no-op without a `userId`.
   - Create the `Student` (or reuse a wizard draft), linking `userId` + `applicationId`.
   - Match `applyingForClass` → `YearLevel`; create `StudentYearLevel` + set `academicGradeId`.
   - Auto-assign active `FeeStructures` → `FeeAssignment` rows (`PENDING`) and one `UserInvoice` per assignment (via `ensureStudentFeeAssignments` → `ensureInvoicesForAssignment`).
   - Create `Guardian` + `StudentGuardian` + `GuardianPhoneNumber`; copy application documents to `StudentDocument`.
3. Materialize the registration-fee `Payment` + journal entry when the registration fee was paid.

Post-transaction (best-effort, owned by the caller): notify student + guardians, mint a password-reset link for a new guest user, dispatch `fee_due` notifications, compute a section suggestion. The student is created with `wizardStep: null` and `status: ACTIVE`. The admin wizard and CSV import call `provisionStudent()` the same way (with `origin` `ADMIN_DIRECT` / `BULK_IMPORT` / `ONBOARDING_IMPORT`), which is what gives every channel the same login + invoices.

## Guardian auto-creation

For each parent (father, mother):

1. Split name into `firstName` + `lastName` (last space).
2. Upsert `GuardianType` (`father`, `mother`).
3. Create `Guardian` (or find by email).
4. Create `StudentGuardian` join row with occupation, `isPrimary`.
5. Create `GuardianPhoneNumber` if phone provided.

Father is primary; if no father, mother becomes primary.

## Section placement

After enrollment, place the student in a section:

- Enrollment response includes `suggestedSectionId` if exactly one section has capacity for the student's grade.
- Lists sections with available capacity for that grade.
- `placeStudentInSection()` assigns + auto-enrols in grade classes via `enrollStudentInGradeClasses()`.

## Wizard editing

Enrolled students can be edited via the student wizard (Students table → Actions → Edit). All application data is pre-populated.

`updateStudentWizardStep` only updates `wizardStep` for draft students (non-null `wizardStep`) — prevents accidental reversion of enrolled students. Final button reads "Save", not "Create".

## Merit list

Admins enter entrance and interview scores directly on the merit table (0–100 per score). `generateMeritList()` computes a weighted `meritScore` for each application:

- Entrance exam — 60%
- Interview — 40%

Applications are ranked by `meritScore` descending; unscored applications sort last. Scores are batch-written via `updateApplicationScores()`. The merit table covers shortlisted, selected, and waitlisted applications.

## Offer → accept → pay → enroll flow

Once an applicant is `SELECTED`:

1. An offer is created with a configurable expiry (`offerExpiryDays`, default 14).
2. The applicant logs in via a token'd offer URL (login preserves the full callback path).
3. The applicant pays the **registration fee** through the school's gateway (Tap for UAE/Gulf, Stripe elsewhere). The application wizard shows an informational payment preview — no payment is collected at the application stage.
4. On payment success, the admin sees a registration-fee paid banner and can proceed.
5. Admin clicks "Confirm enrollment" → `confirmEnrollment()` transaction fires (see Enrollment transaction).
6. Post-enrollment the admin uses the **Placement dialog** on the enrollment table to pick a section (seat counts shown); `placeStudentInSection()` enrols the student in all grade classes.

Abandoned-checkout retry is unblocked — applicants who navigated away can return and complete payment.

## Leads tab

The **Leads** tab (`/{lang}/admission/leads`) surfaces:

- **Inquiries** — contact form submissions with name, email, phone, interest grade, source, and follow-up status.
- **Tour bookings** — scheduled visits with attendee count, status (`PENDING` / `CONFIRMED` / `CANCELLED`), and reschedule/cancel actions.

ADMIN and STAFF roles have full action access; ACCOUNTANT has read-only access. New inquiries and bookings trigger in-app notifications to ADMIN + STAFF. Tour bookings guard against overselling (TOCTOU-safe) and honour the `enableTourBooking` campaign flag; cancel/reschedule decrements the attendee count correctly.

## Document-AI processing

Uploaded applicant documents (ID scans, transcripts) are processed by an extraction queue. The cron `/api/cron/process-document-jobs` runs every 10 minutes and drains pending jobs. `classifyDocument()` and `getDocumentProcessingStatus()` are RBAC-gated. AI usage is budget-checked (`canUseAI`) before each call; the bank-receipt Zod schema is optional so partial extractions do not fail the whole job.
