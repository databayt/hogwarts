The attendance module tracks student attendance across multiple capture methods, with role-based access, parent notifications, US DoE risk levels, intervention tracking, and CSV / Excel / PDF export. Teachers see only their assigned classes; students and guardians see only their own records.

> **Ship tracker:** [databayt/hogwarts#1](https://github.com/databayt/hogwarts/issues/1) — consolidated attendance + compliance master tracker. See also [Aldar Epic 01](https://kun.databayt.org/en/docs/aldar#01--adek-attendance-submission).

## Routes

The attendance root is role-aware: TEACHER lands on **Quick Attendance** (mobile-first, absent-oriented — everyone defaults present, taps cycle absent → late → present, one save submits the roster and notifies guardians of absentees), ADMIN/STAFF land on the overview, students/guardians on their self-view. All under `src/app/[lang]/s/[subdomain]/(school-dashboard)/attendance/`. Tabs: `/`, `/manual`, `/qr-code`, `/records`, `/excuses`, `/early-warning`, `/interventions`, `/analytics`, `/reports`, `/settings`. The staff overview's **Quick-access grid** links `/recent`, `/bulk-upload`, `/barcode`, `/hall-pass`, `/interventions/tiers`, `/gamification`, `/ai` (+ `/kiosk`, `/letters` for admins). Deep-link only: `/geo`, `/intentions`, `/bulk`. `/analysis` permanently redirects to `/analytics`; `/parent/attendance` redirects to `/attendance/records`. See the Capture methods table below for per-route purpose; see RBAC for per-role access.

## Manual attendance

Flat inline toolbar (`PlatformToolbar`): class selector, date picker, bulk status buttons, Save. `getCurrentPeriod()` auto-selects the class matching the current timetable period; names translate on demand via `getText()`. Keyboard: `P` / `A` / `L` for present / absent / late, `Cmd/Ctrl+S` save. Absent-student guardians are auto-notified on save; an approved `AbsenceIntention` converts ABSENT → EXCUSED.

The **roster unit is the section** — `Section.students` defines who is in the class. Each section has exactly one main classroom (homeroom) via `Section.classroomId`; non-homeroom rooms (labs, halls, fields) are reached per-period through the timetable slot (`Timetable.classroomId`), which is also what `getCurrentPeriod()` resolves for period-aware attendance. See [Classrooms › Integration](/docs/classrooms#integration).

## Parent engagement

`triggerAbsenceNotification()` fires on absent: in-app (high priority), SMS via Twilio (if configured), email. Uses school `preferredLanguage`.

Excuse reasons: Medical, Family Emergency, Religious, School Activity, Transportation, Weather, Other. Approved → `EXCUSED`; rejected → unexcused.

Student / guardian view: percentage + trend, calendar history, status breakdown, excuse submission for unexcused absences.

## Early warning

US DoE risk levels:

| Risk               | Attendance rate | Absence rate |
| ------------------ | --------------- | ------------ |
| Satisfactory       | ≥95%            | `<5%`        |
| At-risk            | 90 – 94.9%      | 5 – 9.9%     |
| Moderately chronic | 80 – 89.9%      | 10 – 19.9%   |
| Severely chronic   | `<80%`          | ≥20%         |

Intervention tracking at `/attendance/interventions` — 14 intervention types (parent contact, home visit, counsellor / social worker / administrator referral, attendance contract, truancy referral, community resource, academic support, mentorship, incentive, other) with statuses Scheduled → In Progress → Completed / Cancelled / Escalated. Policy engine in `actions/policy.ts` evaluates triggers and manages exemptions.

## Schema

```prisma
model Attendance {
  id           String           @id @default(cuid())
  schoolId     String
  studentId    String
  classId      String
  date         DateTime         @db.Date
  status       AttendanceStatus
  method       AttendanceMethod @default(MANUAL)
  checkInTime  DateTime?
  checkOutTime DateTime?
  notes        String?
  markedBy     String?
  markedAt     DateTime         @default(now())
  deletedAt    DateTime?

  periodId    String?
  timetableId String?
  periodName  String?

  @@unique([schoolId, studentId, classId, date, periodId])
  @@index([schoolId, date, periodId])
}
```

**Soft-delete semantics.** `deletedAt` is a tombstone, not a hard delete. A
removed record is excluded from every analytics, dashboard, export and
regulator-compliance read, and from the student / guardian mobile views — yet the
row is retained for audit and an admin can restore it (`restoreAttendance`).
Because the `@@unique` tuple still reserves a deleted row's key, re-marking a
student for the same day/period **revives** the original record (clears
`deletedAt`) instead of creating a duplicate — this holds across manual, period,
QR, barcode, geofence and kiosk capture.

Supporting models: `AttendanceExcuse`, `AttendanceIntervention`, `QRCodeSession`, `StudentIdentifier`, `AbsenceIntention`, `PolicyTrigger`, `PolicyExemption`, `MasterAttendance`, `HallPass`, `KioskLog`, `AttendanceReward`, `StudentBadge`, `AttendanceStreak`.

## Regulator compliance (ADEK eSIS — UAE)

UAE schools (`School.country === "AE"`) can opt into a daily-CSV upload to the
ADEK eSIS portal at `esis.adek.gov.ae` plus the ADEK-mandated 2-hour parent
contact for unreported absences. The framework is generic — a new regulator
(SEC KSA, MoE Qatar, MoE Sudan) only adds a value to the
`ComplianceProvider` enum and a sibling directory under
`src/lib/compliance/providers/<authority>/`.

| Piece                | Path                                              | Purpose                                            |
| -------------------- | ------------------------------------------------- | -------------------------------------------------- |
| Settings UI          | `/{lang}/compliance`                              | Toggle, mode, SLA, daily-time, submissions list    |
| Cross-tenant admin   | `/{lang}/(saas-dashboard)/compliance` (DEVELOPER) | Shared credential groups for Aldar-style piggyback |
| Daily cron           | `/api/cron/esis-submit` (`0 10 * * *`)            | Enqueues + fans out per school                     |
| Followup cron        | `/api/cron/absence-followup` (`*/30 * * * *`)     | 2h SLA evidence trail                              |
| Webhook              | `/api/webhooks/adek`                              | Async `ACCEPTED`/`REJECTED` receipts               |
| Worker (out of repo) | `tools/adek-rpa-worker/` (Fly.io)                 | Playwright fallback when no API                    |

Connector modes — set per school × provider:

| Mode           | What it does                                                                                                            |
| -------------- | ----------------------------------------------------------------------------------------------------------------------- |
| `DRY_RUN`      | Generate CSV, persist for audit, registrar uploads to eSIS manually                                                     |
| `PIGGYBACK`    | Submit via shared-group eSIS credentials (Aldar pattern, 1 cred row × N schools, circuit breaker on cascading failures) |
| `OFFICIAL_API` | Documented vendor API (awaiting ADEK developer access)                                                                  |
| `RPA`          | External Playwright worker claims `QUEUED` rows and uploads via the eSIS web portal                                     |
| `DISABLED`     | Skip in cron                                                                                                            |

### 2-hour parent-contact SLA

When `SchoolComplianceConfig.enabled = true`,
`triggerAbsenceNotification` dispatches on `in_app + email + whatsapp`
channels (instead of `in_app` only) so the email + WhatsApp crons drain the
notification row. Every 30 minutes, the `absence-followup` cron looks for
ABSENT rows that are:

- `markedAt < now() - parentContactSlaMinutes` (default 120, computed in UTC)
- have no `AttendanceExcuse`
- have no approved `AbsenceIntention` covering today
- have no `AttendanceIntervention` of type `PARENT_PHONE_CALL` / `PARENT_EMAIL`
  in the past 24h

It dispatches a `absence_unreported_followup` notification to each guardian,
then writes an `AttendanceIntervention` row (`PARENT_EMAIL`, parentNotified=true,
contactMethod=`in_app+email+whatsapp`). That row is the regulator-facing
audit evidence — `getParentContactSlaReport()` queries it to produce
"contacted within SLA" counts for ADEK audits.

### Schema (compliance side)

Five new entities in `prisma/models/compliance.prisma` — none of them
ADEK-specific:

- `ComplianceProvider` enum (currently `ADEK_ESIS`, `CUSTOM`)
- `ConnectorMode` enum
- `ComplianceSubmissionStatus` enum + `CircuitBreakerState` enum
- `SchoolComplianceConfig` — one row per school × provider (opt-in)
- `SharedComplianceCredentialGroup` — group-shared creds (Aldar pattern)
- `SchoolComplianceCredential` — per-school creds (OFFICIAL_API mode)
- `ComplianceSubmission` — one row per attempt, indexed by `(schoolId,
provider, submissionDate, attemptNumber)`. Retries chain via `supersededBy`.
  The CSV is persisted inline in `csvArtifactContent` and downloaded via
  `/api/compliance/artifact/[submissionId]` (signed S3 URL planned).

`AuditLog.userId` was loosened to nullable in the same epic so cron / worker /
webhook system events can be recorded — calls pass `userId: null`.

A new `NotificationType.absence_unreported_followup` covers the 2h SLA path.
