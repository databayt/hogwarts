## Overview

The Document Intelligence Initiative is a platform-wide effort to bring AI-powered document processing to every module in Hogwarts. It transforms manual, error-prone document handling into automated extraction, classification, and validation workflows. The initiative spans 8 epics, 42 stories, and 233 story points, building on existing infrastructure rather than replacing it.

### Architecture Context

The plan extends the following existing infrastructure:

| Component           | Location                                     | Current State                                                                                            |
| ------------------- | -------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| Document Extraction | `src/lib/document-extraction/`               | Claude 3.5 Sonnet via Vercel AI SDK `generateObject()` with Zod schemas, hardcoded to 8 onboarding steps |
| AI Providers        | `src/lib/ai/providers.ts`                    | Multi-provider routing (Anthropic + Groq) with task-based selection and cost estimation                  |
| Rate Limiter        | `src/lib/ai/rate-limiter.ts`                 | Priority queue with 5 concurrent, exponential backoff                                                    |
| AI Config           | `src/lib/ai/config.ts`                       | `AIServiceConfig` with `monthlyCostLimit`, fallback behavior, confidence thresholds                      |
| AI Services         | `src/lib/ai/openai.ts`                       | Essay grading, OCR, question generation with `isAIServiceAvailable()` guard pattern                      |
| Receipt OCR         | `finance/receipt/ai/extract-receipt-data.ts` | Receipt OCR pattern using `generateObject()` with Zod schema + DB status update                          |
| QBank Engine        | `qbank-automation/engine/`                   | PDF parse + LangChain chunking + embeddings (placeholder `findRelevantChunks()`)                         |
| PDF Templates       | `src/components/file/generate/`              | `@react-pdf/renderer` templates for invoice, receipt, certificate, report card, ID card, transcript      |

### Key Schema Facts

- `Application.documents` is `Json?` -- untyped array of URLs, no AI processing
- `Application.entranceScore`, `interviewScore`, `meritScore` are manually entered `Decimal?`
- `SourceMaterial` + `SourceChunk` models exist but `findRelevantChunks()` returns empty
- `CatalogMaterial` and `CatalogBook` have `fileUrl`/`digitalFileUrl` but no processing pipeline
- `FinancialReport` model exists with `status: GENERATING|COMPLETED|FAILED` and `fileUrl` but generation is stub
- `Payment` model supports `BANK_TRANSFER` method with `receiptUrl` field
- `ExpenseReceipt` has full OCR pipeline already working
- `StudentAnswer` has `uploadUrl`, `ocrText`, `ocrConfidence` fields, and `processOCRWithAI()` action exists for single answers

### Patterns to Follow

- Server actions use `"use server"` with `auth()` + `schoolId` guard
- `getTenantContext()` for server components
- Error codes returned (not strings) per translation rules
- All queries scoped by `schoolId`
- Dictionary-based i18n for all UI text

### Story 0.1: Generalize the Document Extraction Service

**Points:** 8 | **Type:** API

The current `src/lib/document-extraction/` is hardcoded to 8 onboarding step IDs. Refactor it into a domain-agnostic extraction service that accepts any Zod schema and prompt string at call-time, rather than looking them up from a fixed `stepSchemaMap`. This allows admission documents, bank receipts, invoices, textbook metadata, and onboarding docs to all use the same extraction pipeline. The existing onboarding flow continues to work via a thin wrapper that passes the step-specific schema/prompt.

**Acceptance Criteria:**

- Given a caller provides a Zod schema, prompt string, and file URL or buffer, When `extractWithSchema()` is called, Then it returns structured data validated against that schema with confidence scoring
- Given the old onboarding API at `/api/onboarding/extract` is called, When it receives a file and stepId, Then it delegates to the new generalized extractor via a compatibility wrapper and returns identical results
- Given the AI provider key is missing, When extraction is attempted, Then it returns `{ success: false, errorCode: "AI_NOT_CONFIGURED" }` matching the existing `isAIServiceAvailable()` pattern
- Given a text-based document (PDF/Word/CSV/Excel), When processed, Then text is extracted first via existing file-handlers, then passed to Claude as text content (not image) for cost efficiency

**Key Files:**

- `src/lib/document-extraction/claude-extractor.ts` -- refactor `extractWithClaude()` to accept schema+prompt parameters
- `src/lib/document-extraction/schemas.ts` -- keep existing onboarding schemas, export `stepSchemaMap` unchanged
- `src/lib/document-extraction/types.ts` -- add `GenericExtractionOptions` type with schema, prompt, context fields
- `src/lib/document-extraction/index.ts` -- add `extractWithSchema()` public function; keep `extractFromDocument()` as backward-compatible wrapper
- `src/lib/document-extraction/prompts.ts` -- keep existing; callers now pass prompts directly

**Dependencies:** None

### Story 0.4: Build Document Processing Queue Runner

**Points:** 8 | **Type:** API

Implement the async job runner that processes `DocumentProcessingJob` records. This is a server-side function (callable from API routes or server actions) that picks up PENDING jobs, sets them to PROCESSING, executes the appropriate extraction/processing logic based on job type, stores results, and handles failures with retry. Uses the existing `aiRateLimiter` for concurrency control. Each job type has a registered handler function (strategy pattern). Include a `/api/document-processing/run` endpoint for cron-triggered or manual processing.

**Acceptance Criteria:**

- Given 3 PENDING jobs exist, When the runner executes, Then it processes them sequentially (respecting rate limits), updating status to PROCESSING then COMPLETED/FAILED
- Given a job fails with a retryable error (rate limit, timeout), When the retry count is below max, Then the job status resets to PENDING with incremented `retryCount`
- Given a job fails with a non-retryable error (invalid file, auth error), When recorded, Then status is FAILED and no retry is scheduled
- Given budget enforcement rejects an AI call, When the job cannot proceed, Then status is set to FAILED with `errorCode: "AI_BUDGET_EXCEEDED"`
- Given the runner is invoked via API endpoint, When authenticated as admin, Then it processes up to 10 pending jobs and returns summary stats

**Key Files:**

- `src/lib/document-extraction/queue-runner.ts` -- new file with `processNextJobs()`, `registerHandler()`, `processJob()`
- `src/lib/document-extraction/handlers/index.ts` -- new handler registry mapping job types to processing functions
- `src/app/api/document-processing/run/route.ts` -- new API endpoint for triggering processing

## EPIC-7: Book & Library Intelligence

### Story 7.1: Auto-Extract Book Metadata on Upload

**Points:** 5 | **Type:** API

When a digital book file is uploaded to `CatalogBook.digitalFileUrl` or `Book` (library), auto-extract metadata: title, author, ISBN, publisher, publication year, language, page count, genre/subject, and table of contents. Use the textbook processing pipeline from EPIC-2 but with a book-specific schema. For catalog books, also extract a cover image from the first page if no cover is provided.

**Acceptance Criteria:**

- Given a librarian uploads a PDF book, When processing completes, Then `CatalogBook.title`, `author`, `isbn`, `publisher`, `publicationYear`, `pageCount`, `language` are auto-populated
- Given the book has no ISBN in the metadata, When AI extracts from the copyright page, Then the ISBN is found and stored
- Given a cover image is not provided, When the first page is rendered, Then a thumbnail is generated and stored as `CatalogBook.coverUrl`
- Given the local `Book` model mirrors a `CatalogBook`, When the catalog book is processed, Then linked `Book` records inherit the metadata

**Key Files:**

- `src/lib/document-extraction/handlers/book-metadata.ts` -- new handler reusing textbook processor
- `src/components/school-dashboard/qbank-automation/engine/textbook-processor.ts` -- extend for book-specific schema

**Dependencies:** Story 2.1

### Story 7.2: Generate Book Summaries and Content Descriptions

**Points:** 5 | **Type:** API

For books with digital files, generate AI-powered summaries at two levels: a short description (1-2 sentences for search results, stored in `CatalogBook.description`) and a detailed summary (2-3 paragraphs for the book detail page, stored in `CatalogBook.summary`). For books with chapter structure, also generate per-chapter summaries. Leverage the summary generation infrastructure from EPIC-2.

**Acceptance Criteria:**

- Given a book is uploaded and text is extracted, When summary generation runs, Then `CatalogBook.description` gets a 1-2 sentence blurb and `CatalogBook.summary` gets a detailed summary
- Given a book has 12 chapters, When chapter summaries are generated, Then each chapter has a 100-word summary accessible from the book detail view
- Given the book is in Arabic, When summaries are generated in Arabic, Then the `lang` field is set to "ar" and English summaries are optionally generated for bilingual display
- Given AI budget is constrained, When summary requests are queued, Then short descriptions are prioritized over detailed summaries

**Key Files:**

- `src/components/school-dashboard/qbank-automation/engine/summary-generator.ts` -- reuse from EPIC-2

**Dependencies:** Story 7.1, Story 2.4

### Story 7.3: Build Searchable Content Index for Library

**Points:** 5 | **Type:** API

Create a content index for the library that enables full-text search across all book content. For each processed book, store chunked content with metadata (chapter, page number) that can be searched via Prisma text queries. When a teacher/student searches for "photosynthesis", return matching books with relevant page numbers and context snippets. If embeddings are available (from EPIC-2), use semantic search; otherwise fall back to text-based search.

**Acceptance Criteria:**

- Given a library has 50 books with processed content, When a teacher searches "photosynthesis", Then matching books are returned with chapter name and page number context
- Given semantic search is available (embeddings present), When searched, Then results are ranked by semantic similarity
- Given embeddings are not available, When searched, Then Prisma `contains` or `search` is used for text-based matching
- Given a search returns 5 books, When results are displayed, Then each shows: title, author, matching chapter, page number, and a content snippet with the search term highlighted

**Key Files:**

- `src/components/school-dashboard/library/search/content-search.ts` -- new search service

**Dependencies:** Story 7.1, Story 2.3
