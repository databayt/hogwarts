## Platform Accounts (No schoolId)

These accounts exist at the platform level and are **not tied to any school**.

### dev@databayt.org — DEVELOPER

- **Arabic Name**: المطور
- **Purpose**: Platform administrator with access to the SaaS dashboard
- **Access**: `ed.databayt.org/en/dashboard` (production) or `localhost:3000/en/dashboard` (local)
- **Capabilities**: View all schools, platform analytics, system configuration
- **Note**: Only the DEVELOPER role can access the SaaS dashboard

### user@databayt.org — USER

- **Arabic Name**: مستخدم جديد
- **Purpose**: Fresh user for testing the onboarding flow
- **Access**: Main domain only (no school subdomain)
- **Flow**: Login → "Get Started" → Onboarding wizard → Create new school
- **Reset**: `pnpm db:reset-test-user` (restores to fresh state with no schoolId)
- **Note**: This account should always remain in a "fresh" state for onboarding tests

## Known Gaps

- **No second school** for cross-tenant testing (only Demo Academy exists)
- **No OAuth test accounts** (GitHub, Google) — only credentials-based login
- **No suspended/inactive users** for testing disabled account flows
- **No multi-school user** — architecture limitation: User has a single `schoolId`
- **Name coincidence**: parent@ and student@ share surname "حسن" and father's given name matches student's name "محمد" — this is an index-based coincidence from the random name generator

## Reset Procedures

```bash
# Reset only user@databayt.org to fresh state (no schoolId, USER role)
pnpm db:reset-test-user

# Full re-seed (all data, additive/idempotent)
pnpm db:seed

# Nuclear option: drop all data and re-seed (MANUAL ONLY)
pnpm prisma migrate reset && pnpm db:seed
```

**Important**: `pnpm db:seed` takes 45-120 minutes. For individual modules, use `pnpm db:seed:single <name>` instead. Seeds use `upsert` operations and are idempotent.
