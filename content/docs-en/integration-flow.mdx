## The Dependency Chain

Every school on the platform progresses through 8 layers to reach a fully operational timetable. Each layer depends on the ones above it.

## Layer 1: Global Catalog

The catalog is the universal shared backbone. All curriculum content lives here once. Schools reference it via bridge tables -- they never duplicate it.

### Hierarchy

```
CatalogSubject (~220 grade-specific US K-12 + future curricula)
  > CatalogChapter (topic groups per grade-subject)
    > CatalogLesson (individual lessons)
```

### Grade-Specific Subjects

The catalog uses **grade-specific subjects** rather than level-based subjects. Each level-based entry (e.g., "Elementary Math") is expanded into one `CatalogSubject` per grade:

```
Level-based (old):  "Elementary Math"  levels: [ELEMENTARY]  grades: [1,2,3,4,5,6]
                                        |
Grade-specific:     "Math Grade 1"     grades: [1]   slug: us-g1-math
                    "Math Grade 2"     grades: [2]   slug: us-g2-math
                    "Math Grade 3"     grades: [3]   slug: us-g3-math
                    ...
Teacher creates content (video, question, material)
  > Saved as PENDING + PRIVATE
    > SaaS admin reviews
      > APPROVED: indexed in catalog with visibility tier
      > REJECTED: feedback sent to creator
description step --> School.schoolType (e.g., "public", "british", "ib")
location step    --> School.country, School.city, School.state
capacity step    --> School.sectionsPerGrade, School.studentsPerSection + computed maxClasses/maxStudents
schedule step    --> School.curriculum (structure slug) + Period records + Terms + SchoolWeekConfig
```

### Capacity Step Clarification

The capacity step collects section-based inputs that drive computed plan limits:

**Inputs (what the form collects):**

| Form Field           | Saved As                    | What It Means                            |
| -------------------- | --------------------------- | ---------------------------------------- |
| Teachers             | `School.maxTeachers`        | Total teacher headcount                  |
| Sections per Grade   | `School.sectionsPerGrade`   | How many sections (A, B, C...) per grade |
| Students per Section | `School.studentsPerSection` | Target class size                        |

**Computed (derived from inputs):**

| Computed Field | Formula                           | What It Means                               |
| -------------- | --------------------------------- | ------------------------------------------- |
| `maxClasses`   | `grades × sectionsPerGrade`       | Total `Class` records the school can create |
| `maxStudents`  | `maxClasses × studentsPerSection` | Total student records the plan allows       |

Key distinction: `maxClasses` limits `Class` records (course sections), NOT physical `Classroom` records. Classroom + Section records **are** auto-created during onboarding by `autoProvisionSections()`.

### The Onboarding Completion Action

When the user accepts terms on the legal step, `completeOnboarding()` in `legal/actions.ts` runs. It sets `isPublished: true` and `onboardingCompletedAt`, making the school live and accessible via its subdomain. It then triggers seven non-blocking provisioning steps via `provisionSchoolDefaults()` (run in background via `after()`):

1. **`setupDefaultsForSchool()`** -- creates YearLevels (filtered by schoolLevel), Departments (6 universal), and ScoreRanges (A+ through F)
2. **`setupCatalogForSchool()`** -- creates academic structure (Layer 3) and subject selections (Layer 4)
3. **`applyTimetableStructureForNewSchool()`** -- creates the school year, terms, periods, and week config
4. **Default `ClassroomType` upsert** -- creates a "Classroom" type so the Configure tab works immediately
5. **`autoProvisionSections()`** -- creates Classroom + Section records for every AcademicGrade (N per grade, where N = `School.sectionsPerGrade`, default 2). Naming: "Grade N-A", "Grade N-B" for both rooms and sections.
6. **`generateUniqueJoinCode()`** -- sets `School.joinCode` for the internal onboarding join page
7. **`syncStudentGrades()`** -- links any students with raw yearLevel strings to the newly created AcademicGrade records

Each step has its own `.catch()` block -- a failure in any step does NOT abort subsequent steps. The entire `provisionSchoolDefaults` is fire-and-forget (the user never sees these errors).

The `congratulations/actions.ts` file contains `publishSchool()` as an admin-only fallback for manual re-provisioning. It calls both `setupCatalogForSchool()` and `applyTimetableStructureForNewSchool()`, each wrapped in its own try/catch so that a failure in one does not block the other.

### `applyTimetableStructureForNewSchool()` -- What It Creates

This function (in `src/lib/catalog-setup.ts`) reads the selected timetable structure and provisions:

| Record             | Details                                                                               |
| ------------------ | ------------------------------------------------------------------------------------- |
| `SchoolYear`       | Default year (Sep 1 - Jun 30), created if none exists                                 |
| `Term` (x2)        | Term 1: Sep 1 - Jan 31 (`isActive: true`), Term 2: Feb 1 - Jun 30 (`isActive: false`) |
| `Period` (x N)     | All periods from the structure definition (class, break, lunch)                       |
| `SchoolWeekConfig` | Working days + `defaultLunchAfterPeriod` from the structure definition                |

### Timetable Structure Selection

Schools choose from country-specific timetable structures during the schedule step. The system now includes generic structures alongside country-specific ones:

| Slug             | Name                     | Periods | Days    | Country | Default |
| ---------------- | ------------------------ | ------- | ------- | ------- | ------- |
| `sd-gov-default` | Sudan Government         | 8 x 45m | Sun-Thu | SD      | Yes     |
| `sd-private`     | Sudan Private            | 7 x 50m | Sun-Thu | SD      | No      |
| `sd-british`     | British Curriculum       | 6 x 55m | Mon-Fri | SD      | No      |
| `sd-ib`          | IB / American            | 6 x 50m | Mon-Fri | SD      | No      |
| `sd-half-day`    | Half Day / KG            | 5 x 40m | Sun-Thu | SD      | No      |
| `gulf-standard`  | Gulf Government Standard | 7 x 45m | Sun-Thu | GULF    | Yes     |
| `gulf-private`   | Gulf Private / Intl      | 7 x 50m | Sun-Thu | GULF    | No      |
| `mena-standard`  | MENA Standard            | 7 x 45m | Mon-Fri | MENA    | Yes     |
| `us-standard`    | US Standard              | 7 x 50m | Mon-Fri | US      | Yes     |
| `intl-default`   | International Default    | 6 x 50m | Mon-Fri | `*`     | No      |

`getRecommendedStructures(country, schoolType, schoolLevel)` uses a **scoring system** to rank structures:

| Signal              | Score | Example                                          |
| ------------------- | ----- | ------------------------------------------------ |
| Country exact match | +40   | SD school → SD structures                        |
| Region match        | +25   | SA school → GULF structures                      |
| Wildcard (`*`)      | +5    | Fallback for unrecognized countries              |
| SchoolType match    | +20   | `"private"` school → structures with `"private"` |
| SchoolLevel match   | +10   | `"secondary"` → structures with `"secondary"`    |

**Thresholds:** score >= 30 = recommended, score >= 50 = auto-select. Country-to-region mapping: SA/AE/QA/KW/BH/OM → GULF; EG/JO/LB/TN/MA/DZ → MENA.

### What Happens Next

After provisioning, `setupCatalogForSchool()` reads the school's profile (`country`, `schoolLevel`) and auto-creates everything in Layer 3 and Layer 4.

If catalog setup was missed during onboarding, SaaS admins can trigger it manually from the tenants table via **Setup Catalog** in the row actions dropdown. The button shows current status ("X levels, Y grades" or "Not configured") and calls `tenantSetupCatalog(schoolId)` with `skipIfExists: true`.

**Depends on:** Layer 1 (catalog must exist to link subjects).

See: [Onboarding](/docs/onboarding)

## Layer 3: Academic Structure

`setupCatalogForSchool()` reads the school profile and creates a complete academic hierarchy inside a single database transaction.

### What Gets Created

### Level Filtering

The school's `schoolLevel` field determines which levels are created:

| `schoolLevel`   | Levels Created  | Grades |
| --------------- | --------------- | ------ |
| `"primary"`     | ELEMENTARY only | 1-6    |
| `"secondary"`   | MIDDLE + HIGH   | 7-12   |
| `"both"` / null | All three       | 1-12   |

### Key Models

| Model            | Scope  | Key Fields                                      |
| ---------------- | ------ | ----------------------------------------------- |
| `AcademicLevel`  | School | `level`, `levelOrder`, `startGrade`, `endGrade` |
| `AcademicGrade`  | School | `gradeNumber`, `levelId`, `yearLevelId`         |
| `AcademicStream` | School | `streamType` (SCIENCE, ARTS), `gradeId`         |

**Depends on:** Layer 2 (school profile must exist with `country` and `schoolLevel`).

## Layer 4: Subject Selection

The bridge layer connects the global catalog to school-specific grades. This is also created by `setupCatalogForSchool()` in the same transaction as Layer 3.

### Auto-Assignment

`setupCatalogForSchool()` checks each catalog subject's `grades` array first (for grade-specific subjects), falling back to level-based mapping only if the `grades` array is empty:

Result: ~99% of relevant subjects auto-assigned. Schools start with a working curriculum immediately.

### Progressive Fallback

`setupCatalogForSchool()` uses a 4-step progressive fallback to find catalog subjects, ensuring every school gets content:

| Step | Filter                              | Example                                   |
| ---- | ----------------------------------- | ----------------------------------------- |
| 1    | `country + curriculum + schoolType` | SD + national + private (exact match)     |
| 2    | `country + curriculum`              | SD + national (ignore schoolType)         |
| 3    | `country="*" + curriculum`          | IB worldwide, British worldwide           |
| 4    | US + us-k12                         | Baseline fallback for unrecognized combos |

The curriculum is inferred from `country + schoolType` if not explicitly set: international schools get `"us-k12"`, known countries get `"national"`, and unknown countries fall back to `"us-k12"`.

### Customization

After auto-assignment, schools can:

- **Browse the full catalog** and add subjects from other systems
- **Hide subjects** per grade by toggling `isActive`
- **Hide chapters/lessons** via `SchoolContentOverride`
- **Rename subjects** using `customName` on the selection record
- **Set period counts** via `weeklyPeriods` for timetable planning

### School-Initiated Content

Schools can request new subjects be added to the catalog:

```
School creates CRUD request --> SaaS reviews --> Approved --> Added to global catalog
Physical classroom side:          LMS / catalog side:
Subject (school-scoped)           CatalogSubject (global)
  |                                   |
  +--- catalogSubjectId ------------->+
  |                                   |
  +--- Class (timetable)             +--- CatalogChapter
  +--- Teacher                       +--- CatalogLesson
  +--- Exam                          +--- CatalogBook
```

Similarly, `CatalogBook.catalogSubjectId` links books to specific grade-specific catalog subjects, and `SchoolBookSelection.gradeId` allows grade-scoped book assignments within a school.

**Depends on:** Layer 1 (catalog subjects) + Layer 3 (grades and streams).

See: [Catalog - School Configuration](/docs/catalog#school-configuration-and-auto-assignment)

## Layer 5a: Teacher Assignment

Teachers are connected to subjects and classes through multiple relationship types.

### Teacher Roles

| Role                  | Model / Field                   | Description                                   |
| --------------------- | ------------------------------- | --------------------------------------------- |
| **Subject Teacher**   | `Class.teacherId` (primary)     | Primary teacher of a class section            |
| **Co-Teacher**        | `ClassTeacher` with `role`      | `PRIMARY`, `CO_TEACHER`, or `ASSISTANT`       |
| **Subject Expert**    | `TeacherSubjectExpertise`       | Qualification metadata (`expertiseLevel`)     |
| **Department Member** | `TeacherDepartment`             | Department membership with `isPrimary` flag   |
| **Department Head**   | `TeacherDepartment`             | `isDepartmentHead: true` (one per department) |
| **Homeroom Teacher**  | `Class` (pastoral, not subject) | Pastoral care, not tied to a specific subject |

### Assignment Flow

### Key Models

| Model                     | Scope  | Unique Constraint                     |
| ------------------------- | ------ | ------------------------------------- |
| `ClassTeacher`            | School | `[schoolId, classId, teacherId]`      |
| `TeacherSubjectExpertise` | School | `[schoolId, teacherId, subjectId]`    |
| `TeacherDepartment`       | School | `[schoolId, teacherId, departmentId]` |

### Cascade Validation on Delete

Both teachers and subjects enforce cascade validation before deletion -- preventing orphaned references in downstream layers.

**Teacher delete** (`deleteTeacher` in `teachers/actions.ts`):

| Dependency checked | Behavior                                                    |
| ------------------ | ----------------------------------------------------------- |
| `Class.teacherId`  | Blocks delete -- "Reassign classes first"                   |
| `Timetable`        | Blocks delete -- "Remove from timetable first"              |
| `ClassTeacher`     | Auto-cleaned in a transaction (co-teaching records removed) |

**Subject delete** (`deleteSubject` in `subjects/actions.ts`):

| Dependency checked        | Behavior                                            |
| ------------------------- | --------------------------------------------------- |
| `TeacherSubjectExpertise` | Blocks delete -- "Remove teacher expertise first"   |
| `Class.subjectId`         | Blocks delete -- "Delete or reassign classes first" |
| `Exam.subjectId`          | Blocks delete -- "Delete exams first"               |
| `QuestionBank.subjectId`  | Blocks delete -- "Delete question banks first"      |

This prevents data integrity issues where a deleted teacher/subject would leave dangling references in classes, timetable slots, or exams.

**Depends on:** Layer 4 (subjects must be selected before teachers can be assigned to them).

## Layer 5b: Student Enrollment

Students enter through the admission pipeline and are placed into classes.

### Application Flow

A 6-step wizard collects student information:

| Step      | Collects                                               |
| --------- | ------------------------------------------------------ |
| Personal  | Name, DOB, gender, nationality, religion               |
| Contact   | Email, phone, address                                  |
| Guardian  | Father, mother, guardian details                       |
| Academic  | `applyingForClass`, `preferredStream`, previous school |
| Documents | Uploaded files                                         |
| Review    | Summary and submission                                 |

### Academic Placement Fields

The academic step is critical for integration:

| Field              | Type       | Purpose                                    |
| ------------------ | ---------- | ------------------------------------------ |
| `applyingForClass` | `String`   | Target grade (e.g., "Grade 10")            |
| `preferredStream`  | `String?`  | Preferred stream (e.g., "science", "arts") |
| `dateOfBirth`      | `DateTime` | Used for DOB-based grade suggestion        |

### Status Progression

```
DRAFT --> SUBMITTED --> UNDER_REVIEW --> SHORTLISTED -->
ENTRANCE_SCHEDULED --> INTERVIEW_SCHEDULED --> SELECTED -->
WAITLISTED / REJECTED / ADMITTED / WITHDRAWN
```

### After ADMITTED

Once a student reaches `ADMITTED` status, they need to be connected to the school structure:

```
Student
  +--> StudentYearLevel  (which grade)
  +--> StudentClass      (which class section, e.g., "Grade 10-A")
  +--> StudentBatch      (which section/batch)
```

This connects students to subjects, teachers, and classrooms through the class:

```
Student --> StudentClass --> Class --> Classroom (home room)
Student --> StudentClass --> Class --> Teacher (primary)
Student --> StudentClass --> Class --> ClassTeacher (co-teachers)
Student --> StudentClass --> Class --> Timetable (schedule)
```

### Section Placement

Two UI components handle placing admitted students into sections:

- **Placement Dialog** (`admission/placement-dialog.tsx`) -- shown per-student after admission. Displays available sections for the student's grade with capacity badges (`enrolled/max`). Validates `maxCapacity` before placement.
- **Bulk Placement** (`admission/bulk-placement.tsx`) -- table of all `ADMITTED` students without section placement. Select multiple students, pick a target section, and batch-assign. Prevents placing into full sections.

Both call `placeStudentInSection(applicationId, sectionId)` which validates section capacity, updates `Student.sectionId`, and auto-creates `StudentClass` records for all classes in that grade. It also triggers `syncStudentClassToEnrollment()` to auto-create corresponding LMS `Enrollment` records.

### Grade Number Extraction

Admission placement needs to match the free-text `applyingForClass` field (e.g., "Grade 5", "الصف الخامس") to an `AcademicGrade`. The `extractGradeNumber()` utility in `src/lib/grade-utils.ts` handles this with a multi-format parser:

| Input Format     | Example                 | Result |
| ---------------- | ----------------------- | ------ |
| Direct number    | `"5"`                   | 5      |
| English pattern  | `"Grade 5"`, `"Year 5"` | 5      |
| Ordinal suffix   | `"5th grade"`           | 5      |
| Arabic ordinals  | `"الصف الخامس"`         | 5      |
| English ordinals | `"fifth grade"`         | 5      |
| Trailing number  | `"Section 5"`           | 5      |

This ensures reliable grade matching regardless of how the applicant or school entered the grade information, supporting both Arabic and English input.

**Depends on:** Layer 3 (grades must exist for placement) + Layer 4 (subjects must be selected).

See: [Admission](/docs/admission)

## Layer 6: Classroom Configuration

Physical rooms and sections are **auto-provisioned** during onboarding by `autoProvisionSections()`. The Configure tab provides additional manual actions for class generation and student enrollment.

### Auto-Provisioned vs Manual

| Record        | Auto-Provisioned? | How                                                    |
| ------------- | ----------------- | ------------------------------------------------------ |
| ClassroomType | Yes               | `completeOnboarding()` upserts "Classroom" type        |
| Classroom     | Yes               | `autoProvisionSections()` creates N rooms per grade    |
| Section       | Yes               | `autoProvisionSections()` creates N sections per grade |
| Class         | **No**            | Admin must run "Generate Classes" in Configure tab     |
| StudentClass  | **No**            | Admin must run "Enroll Students" in Configure tab      |

### Admin Post-Onboarding Steps (Configure Tab)

After onboarding completes and teachers/students are added, the admin uses the Configure tab (`/classrooms/configure`) to complete the pipeline:

```
Step 1: "Generate Sections" (optional) -- add more sections beyond auto-provisioned ones
Step 2: "Generate Classes"  -- creates 1 Class per subject-per-grade-per-term
Step 3: "Enroll Students"   -- creates StudentClass for every student x class per grade
AcademicGrade (e.g., Grade 7)
    |
    | gradeId (optional FK)
    |
    +--> Class
    |      name: "Math - Grade 7 A"
    |      gradeId     --> AcademicGrade (which grade)
    |      subjectId   --> CatalogSubject (what is taught)
    |      teacherId   --> Teacher (who teaches)
    |      classroomId --> Classroom (where it happens)
    |      termId      --> Term (when it runs)
    |      minCapacity: 10  (enrollment floor)
    |      maxCapacity: 50  (enrollment ceiling)
    |
    +--> Classroom (e.g., A101)
           capacity: 30  (physical seats)
Timetable {
  schoolId     -- tenant isolation
  termId       -- which academic term
  dayOfWeek    -- 0 (Sun) through 6 (Sat)
  periodId     -- FK to Period
  classId      -- FK to Class (which class section)
  teacherId    -- FK to Teacher (who teaches)
  classroomId  -- FK to Classroom (which room)
  weekOffset   -- 0 (current) or 1 (next week)
}
```

### Three Unique Constraints

These prevent double-booking at the database level:

```
@@unique([schoolId, termId, dayOfWeek, periodId, classId, weekOffset])
@@unique([schoolId, termId, dayOfWeek, periodId, teacherId, weekOffset])
@@unique([schoolId, termId, dayOfWeek, periodId, classroomId, weekOffset])
Exam > ExamSession > StudentAnswer > MarkingResult > ExamResult
QuestionBank > GeneratedExam > GeneratedPaper
Result (standalone grades for assignments/homework)
ExamResult + Result (Layer 8)
    |
    v
Report Card Generation (aggregate per student per term)
    |
    +----> Certificate PDF (composable template system)
    +----> Transcript (multi-year aggregation)
    +----> Notifications (grade_posted, report_ready, certificate_issued, promotion_decided)
    +----> Promotion/Retention (year-end evaluation → Student.academicGradeId update)
```

### Certificate Composition

Certificates use the same 5-layer composition architecture as exam papers:

```
Atom > Section > Variant > Composition > Document
```

7 slots: Header, Title, Recipient, Body, Scores, Signatures, Footer. 4 regional presets (SD, SA, US, MENA). Configuration stored in `ExamCertificateConfig.compositionConfig` (JSON).

### Promotion Pipeline

````
Admin triggers evaluation → PromotionBatch created
    → Per student: GPA + failed subjects + attendance → PromotionCandidate
    → Admin reviews and overrides → Batch approved
    → Execute: Student.academicGradeId updated, StudentYearLevel created
    → Notifications sent
No Catalog          --> No subject selection     --> No teacher assignment
No Onboarding       --> No academic structure    --> No grades or streams
No Academic Structure --> No subject selection   --> No class sections
No Terms             --> No timetable slots      (termId is required)
No Teachers          --> No Class records        (teacherId is required on Class)
No Class Records     --> No timetable slots      (classId is required on Timetable)
No StudentClass      --> Timetable works but no students see it
No Classrooms        --> No Class records        (classroomId is required on Class)
No Exams/Results     --> No report cards         --> No transcripts
No Report Cards      --> No promotion evaluation (GPA required)
No Attendance Data   --> Promotion runs without attendance check

### Check what a school has set up

```typescript
// Verify all layers are ready for timetable creation
const [
  levels,
  grades,
  selections,
  teachers,
  classrooms,
  periods,
  terms,
  weekConfig,
] = await Promise.all([
  db.academicLevel.count({ where: { schoolId } }),
  db.academicGrade.count({ where: { schoolId } }),
  db.schoolSubjectSelection.count({ where: { schoolId, isActive: true } }),
  db.teacher.count({ where: { schoolId } }),
  db.classroom.count({ where: { schoolId } }),
  db.period.count({ where: { schoolId } }),
  db.term.count({ where: { schoolId } }),
  db.schoolWeekConfig.findFirst({ where: { schoolId } }),
])

const ready =
  levels > 0 &&
  grades > 0 &&
  selections > 0 &&
  teachers > 0 &&
  classrooms > 0 &&
  periods > 0 &&
  terms > 0 &&
  weekConfig !== null
````
