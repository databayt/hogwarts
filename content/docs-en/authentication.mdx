## Overview

Our authentication system is built on Next.js App Router with Auth.js (NextAuth v5), designed for multi-tenant school management. It supports OAuth providers (Google, Facebook), email/password credentials, two-factor authentication, and cross-subdomain single sign-on.

**Sub-docs for detail:**

- [Credentials Authentication](/docs/credentials) -- registration, login with 2FA, email verification, password reset, token system, brute force protection
- [OAuth Authentication](/docs/oauth) -- provider config, account linking, callback handling, cross-subdomain SSO
- [Flow Diagrams](/docs/flow-diagrams) -- visual diagrams for all auth flows

## Entry Points

There are four ways a user enters the authentication flow:

| #   | Entry Point          | Trigger                         | Post-Login Redirect                                                                |
| --- | -------------------- | ------------------------------- | ---------------------------------------------------------------------------------- |
| 1   | **Marketing Header** | Login icon on `ed.databayt.org` | DEVELOPER: `/dashboard`, school user: `{school}.databayt.org/dashboard`, USER: `/` |
| 2   | **Get Started**      | Button on marketing page        | `/login?callbackUrl=/onboarding` then 15-step wizard                               |
| 3   | **School Subdomain** | `{school}.databayt.org/login`   | Same as #1 but within school context                                               |
| 4   | **Protected Route**  | Direct URL to protected page    | `/login?callbackUrl={original_url}`, returns after auth                            |

Users without a school (USER role) are redirected to `/{locale}` (homepage), not `/onboarding`. Onboarding is only reachable via explicit callbackUrl from the "Get Started" button.

## Redirect Logic

The redirect decision is a 5-step priority chain in `src/auth.ts` redirect callback and `src/components/auth/login/action.ts`:

1. **Callback URL** -- from cookie, URL params, or baseUrl params (preserved across OAuth via httpOnly cookie)
2. **Explicit relative path** -- from `signIn({ redirectTo })` (e.g., DEVELOPER gets `/en/dashboard`)
3. **Subdomain detection** -- if on a school subdomain, go to that school's dashboard
4. **Smart redirect** -- decode JWT, look up school domain, redirect to school dashboard
5. **Fallback** -- main domain dashboard

## Session & JWT Structure

**JWT Token** (populated in `src/auth.ts` jwt callback):

```typescript
token = {
  id: string              // User ID
  role: UserRole           // User role
  schoolId: string | null  // Multi-tenant key
  provider: string         // OAuth provider
  providerAccountId: string
  sessionToken: string     // Cache-busting
  updatedAt: number        // Forced refresh detection
  hash: string             // Cache-busting hash
  exp: number              // Expires at (+24h)
}
```

**Session** (populated in session callback):

```typescript
session.user = {
  id: string
  email: string
  role: UserRole           // From token, or preview role if active
  schoolId: string | null  // Multi-tenant key
  isPreviewMode: boolean   // True when preview role cookie is active
}
```

## Tenant Context

```typescript
import { getTenantContext } from "@/lib/tenant-context"

const { schoolId, requestId, role, isPlatformAdmin } = await getTenantContext()
// Priority: impersonation cookie > x-subdomain header > session
```

**3-tier caching** for subdomain-to-schoolId resolution: in-memory (1 min) -> Redis/Upstash (5 min) -> database (circuit breaker protected). Redis is optional; falls back to in-memory only.

## Multi-Tenant Prisma Adapter

The standard Prisma adapter fails with our composite `@@unique([email, schoolId])` constraint. Our `MultiTenantPrismaAdapter` overrides key methods:

| Method             | Standard Adapter                   | Multi-Tenant Adapter                              |
| ------------------ | ---------------------------------- | ------------------------------------------------- |
| `getUserByEmail`   | `findUnique({ where: { email } })` | `findFirst({ where: { email, schoolId: null } })` |
| `createUser`       | No school context                  | Creates with `schoolId: null`                     |
| `getUserByAccount` | Standard lookup                    | Handles composite constraints                     |

OAuth users start with `schoolId: null` until they complete onboarding or join a school.

## User Roles

```typescript
enum UserRole {
  DEVELOPER  // Level 8 - Platform admin (cross-school access)
  ADMIN      // Level 7 - School administrator
  TEACHER    // Level 6 - Teaching staff
  ACCOUNTANT // Level 5 - Finance staff
  STAFF      // Level 4 - General staff
  GUARDIAN   // Level 3 - Parent/guardian
  STUDENT    // Level 2 - Enrolled student
  USER       // Level 1 - Default role
}
```

Hierarchy defined in `src/lib/school-access.ts`, must stay in sync with `roleRoutes` in `src/routes.ts`.

## Route Protection

### Edge Middleware (`src/proxy.ts`)

```
Request -> Edge Middleware
  |-- Decode JWT (lightweight, no DB call)
  |-- Extract: role, schoolId
  |-- Check roleRoutes matrix for route access
  |-- Detect subdomain (custom domain -> Redis lookup)
  |-- Set x-subdomain header for downstream
  |-- PASS -> Next.js renders page
      FAIL -> Redirect to /login or /unauthorized
```

### Server-Side

```typescript
import { auth } from "@/auth"

export default async function ProtectedPage() {
  const session = await auth()
  if (!session) redirect("/login")
  return <Dashboard schoolId={session.user.schoolId} />
}
```

### Client-Side Hooks

```typescript
"use client"

import { useCurrentRole } from "@/components/auth/use-current-role"
import { useCurrentUser } from "@/components/auth/use-current-user"
```

### Role Gate

```typescript
import { RoleGate } from "@/components/auth/role-gate"

<RoleGate allowedRole={UserRole.ADMIN}>
  <AdminPanel />
</RoleGate>
```

## Cross-Subdomain SSO

Production cookies use `.databayt.org` domain, enabling seamless authentication across all school subdomains. Key cookies: `authjs.session-token`, `authjs.csrf-token`, `authjs.pkce.code_verifier`, `authjs.state`, `authjs.nonce` -- all httpOnly, sameSite lax, secure.

## User Registration Options

| Option                | Method                                                                  | Status  |
| --------------------- | ----------------------------------------------------------------------- | ------- |
| **Self-Registration** | OAuth (Google/Facebook) -- no admin work                                | Working |
| **Bulk Creation**     | Admin imports CSV with pre-assigned roles                               | Working |
| **Join Codes**        | 6-char codes (`ABCDEFGHJKMNPQRSTUVWXYZ2345679`), self-enrollment toggle | Working |
| **Invitations**       | Email-based invite links with pre-assigned roles                        | Partial |

Join codes: 481M possible codes (28^6). Self-enrollment toggle via `SchoolBranding.allowSelfEnrollment` -- enabled: instant link, disabled: pending `MembershipRequest`. See `src/lib/join-code.ts` and `src/lib/school-access.ts`.

## Preview Mode

Allows admins to see the UI as a different role without changing their actual role. Uses `preview-mode` and `preview-role` cookies. Session callback overrides role and sets `isPreviewMode: true`.

| Hook               | Returns             | Purpose                                |
| ------------------ | ------------------- | -------------------------------------- |
| `useEffectiveRole` | `UserRole`          | Preview role if active, else actual    |
| `useIsPreviewMode` | `boolean`           | Whether preview mode is active         |
| `useRoleInfo`      | Full context object | Actual, preview, effective role + flag |
