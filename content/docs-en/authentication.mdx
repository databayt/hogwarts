## Overview

Our authentication system is built on Next.js App Router with Auth.js (NextAuth v5), specifically designed for multi-tenant school management. It supports OAuth providers (Google, Facebook), email/password credentials, two-factor authentication, and seamless cross-subdomain single sign-on.

For detailed email/password credential flows (registration, login with 2FA, email verification, password reset, token system, and brute force protection), see [Credentials Authentication](/docs/credentials). For Google and Facebook OAuth flows (provider config, account linking, callback handling, cross-subdomain SSO), see [OAuth Authentication](/docs/oauth).

#### Logout Implementation Details

The logout flow uses **client-side redirect** to avoid middleware interference:

```
User clicks "Logout" in dropdown
         │
         ▼
┌─────────────────────────────────────────┐
│  LogoutButton determines return URL:    │
│  - Marketing/Operator: /{locale}        │
│  - School Platform: /{locale}/s/{sub}/  │
└──────────┬──────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────┐
│  logout(returnUrl) server action:       │
│  - Calls signOut({ redirect: false })   │
│  - Session cookie cleared server-side   │
│  - Returns returnUrl to client          │
└──────────┬──────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────────┐
│  Client-side redirect:                  │
│  window.location.href = returnUrl       │
│  (Hard navigation ensures fresh state)  │
└─────────────────────────────────────────┘
```

**Why client-side redirect?**

When using `signOut({ redirect: true })`, NextAuth's server-side redirect would:

1. Clear the session cookie
2. Send redirect response to homepage
3. **Problem**: Middleware sees unauthenticated request to current protected route
4. Middleware redirects to `/login?callbackUrl={current_route}` before homepage loads

By using `redirect: false` + client-side `window.location.href`:

1. Session cleared server-side
2. Client navigates directly to homepage
3. No middleware interference (homepage is public)

**Files:**

- `src/components/auth/logout-button.tsx` - Context-aware URL + client redirect
- `src/components/auth/logout-action.ts` - Server action with `redirect: false`

**Authorization Flow Verified:**

```
User Request → Edge Middleware (proxy.ts)
         ↓
    JWT Decode → Extract schoolId, role
         ↓
    Route Protection (roleRoutes matrix)
         ↓
    Platform Layout → getTenantContext() → Resolve subdomain
         ↓
    requireSchoolAccess() → user.schoolId === school.id OR isPlatformAdmin?
         ↓
    PASS → Dashboard  |  FAIL → /unauthorized or 404
```

**Key Findings:**

1. **Cross-Subdomain SSO** - Cookie domain `.databayt.org` enables seamless authentication across all subdomains
2. **DEVELOPER Bypass** - Platform admins (DEVELOPER role) can access any school's dashboard regardless of schoolId
3. **Unauthenticated Protection** - Protected routes redirect to `/login?callbackUrl={route}`
4. **Graceful 404** - Non-existent school subdomains return 404 (no data leakage)

### Implementation Details

**Files Modified:**

- `src/routes.ts` - `DEFAULT_LOGIN_REDIRECT = "/"`
- `src/components/auth/login/action.ts` - Role-based redirect logic
- `src/components/auth/user-button.tsx` - Avatar sizing to match header icons

**Redirect Logic (login/action.ts → auth.ts redirect callback):**

The redirect decision is a 5-step priority chain implemented across the credentials login action and the NextAuth redirect callback:

```typescript
// Step 1: Callback URL (from cookie → URL params → baseUrl params)
// Preserved across OAuth redirects via httpOnly cookie
const callbackUrl = await resolveCallbackUrl(url, baseUrl)
if (callbackUrl) return validateCallbackUrl(callbackUrl, baseUrl)

// Step 2: Explicit relative path from signIn({ redirectTo })
// e.g., DEVELOPER → "/en/dashboard", USER → "/en" (homepage)
if (url.startsWith("/") && !url.match(/^\/(en|ar)\/?$/)) {
  return `${baseUrl}${url}`
}

// Step 3: Subdomain detection → school dashboard
const subdomain = detectSubdomainFromHost(host)
if (subdomain) return constructSchoolUrl(subdomain, `/${locale}/dashboard`)

// Step 4: Smart redirect - decode JWT → look up school → redirect
const token = await getToken({ req, secret: process.env.AUTH_SECRET })
if (token?.role === "DEVELOPER") return `${baseUrl}/dashboard`
if (token?.schoolId) {
  const domain = await getSchoolDomain(token.schoolId)
  if (domain) return constructSchoolUrl(domain, `/${locale}/dashboard`)
}

// Step 5: Fallback → main domain dashboard
return `${baseUrl}/dashboard`
```

**Credentials login action** also includes brute force protection (5 failures / 15 min) and audit logging via `src/lib/audit-log.ts`.

> **Note:** Users without a school (USER role) are redirected to `/{locale}` (homepage), not `/onboarding`. Onboarding is only reachable via explicit callbackUrl from the "Get Started" button.

## Core Features

### Authentication Methods

- **Multi-Provider Auth**: Google, Facebook, Email/Password
- **JWT-Based Sessions**: 24-hour expiry with automatic refresh
- **Cross-Subdomain SSO**: Seamless login across `*.databayt.org`
- **Email Verification**: Required for credentials, automatic for OAuth
- **Password Reset**: Secure token-based recovery
- **Two-Factor Authentication**: Optional 2FA for enhanced security

### Multi-Tenant Features

- **School Isolation**: Every user is associated with a `schoolId`
- **Compound Uniqueness**: Same email can exist across different schools
- **Subdomain Routing**: `school.databayt.org` automatically resolves to school context
- **Smart Redirects**: After login, users go directly to their school's dashboard
- **Role-Based Access**: 8 roles with clear hierarchy

### User Roles

```typescript
enum UserRole {
  DEVELOPER  // Platform admin (cross-school access)
  ADMIN      // School administrator
  TEACHER    // Teaching staff
  STUDENT    // Enrolled student
  GUARDIAN   // Parent/guardian
  ACCOUNTANT // Finance staff
  STAFF      // General staff
  USER       // Default role
}
```

### Role Hierarchy

Numeric hierarchy from `src/lib/school-access.ts` - higher number = more permissions:

| Role       | Level | Description                          |
| ---------- | ----- | ------------------------------------ |
| USER       | 1     | Default authenticated user           |
| STUDENT    | 2     | Can view own data                    |
| GUARDIAN   | 3     | Can view linked student data         |
| STAFF      | 4     | General school staff                 |
| ACCOUNTANT | 5     | Finance access                       |
| TEACHER    | 6     | Academic data access                 |
| ADMIN      | 7     | Full school management               |
| DEVELOPER  | 8     | Platform admin (bypasses all checks) |

This hierarchy must stay in sync with the `roleRoutes` matrix in `src/routes.ts`.

## User Registration Options

Schools have **flexibility** in how they manage user accounts:

### Option 1: Self-Registration via OAuth (Recommended)

Students and teachers authenticate themselves:

```
Student → school.databayt.org/login → Google/Facebook → Profile Setup
```

**Benefits**:

- No admin work required
- Users verify their own email automatically
- Streamlined experience
- No password management overhead

### Option 2: Bulk Account Creation

School admin creates accounts in bulk:

```
Admin → Import CSV → Bulk creation with roles → Send credentials
```

**Benefits**:

- Full admin control over who joins
- Pre-assigned roles
- Works for users without email access

### Option 3: Invitation System (Partially Working)

Admin invites specific users via MembershipRequest:

```
Admin → Generate invite → User clicks → OAuth or Credentials → Linked to school
```

**Status**: MembershipRequest model exists. Join codes with `allowSelfEnrollment: false` create pending requests. Full invitation link flow is planned.

**Benefits**:

- Controlled onboarding
- Pre-assigned roles
- Combines admin control with user convenience

### Option 4: School Join Codes (Working)

6-character alphanumeric codes for self-enrollment:

```
Student → Enter code "ABC123" → Sign up → Auto-linked as STAFF
```

**Implementation details** (see `src/lib/join-code.ts` and `src/lib/school-access.ts`):

- **6-character codes** using unambiguous 28-char alphabet (`ABCDEFGHJKMNPQRSTUVWXYZ2345679` - no 0/O, 1/I/L)
- **481M possible codes** (28^6) with collision-resistant generation (5 retry attempts)
- **Self-enrollment toggle** via `SchoolBranding.allowSelfEnrollment`:
  - Enabled: user is linked directly (USER → STAFF promotion)
  - Disabled: creates a pending `MembershipRequest` for admin approval
- **Guards**: user must not already belong to a school, school must be active
- **Join flow**: `joinSchoolByCode()` in `src/lib/school-access.ts`

**Benefits**:

- Easy to share (print, announce, etc.)
- No individual invitations needed
- Admin controls self-enrollment via toggle

## Session Management

### Extended Session

**JWT Token Fields** (populated in `src/auth.ts` jwt callback):

```typescript
token = {
  id: string              // User ID
  role: UserRole           // User role
  schoolId: string | null  // Multi-tenant key
  provider: string         // OAuth provider (google, facebook, credentials)
  providerAccountId: string // OAuth provider account ID
  sessionToken: string     // Cache-busting session token
  updatedAt: number        // Timestamp for forced refresh detection
  hash: string             // Cache-busting hash
  iat: number              // Issued at (Unix timestamp)
  exp: number              // Expires at (Unix timestamp, +24h)
}
```

**Session Fields** (populated in session callback):

```typescript
session.user = {
  id: string
  email: string
  role: UserRole           // From token, or preview role if active
  schoolId: string | null  // Multi-tenant key
  isPreviewMode: boolean   // True when preview role cookie is active
}
```

**Preview Mode**: When `preview-mode` and `preview-role` cookies are set, the session callback overrides `role` with the preview role and sets `isPreviewMode: true`. This allows admins to test the UI as other roles without changing their actual role.

### Tenant Context

```typescript
import { getTenantContext } from "@/lib/tenant-context"

const { schoolId, requestId, role, isPlatformAdmin } = await getTenantContext()
// Priority: impersonation cookie → x-subdomain header → session
```

**`TenantContext` type:**

```typescript
type TenantContext = {
  schoolId: string | null
  requestId: string | null // x-vercel-id or x-request-id or crypto.randomUUID()
  role: UserRole | null
  isPlatformAdmin: boolean // true when role === "DEVELOPER"
}
```

**3-Tier Caching** for subdomain → schoolId resolution:

1. **In-memory Map** (per-instance, 1 min TTL) - fastest
2. **Redis** (shared across serverless instances, 5 min TTL) - via Upstash
3. **Database** (last resort, wrapped in circuit breaker) - writes back to both caches

Redis is lazily initialized and optional. If Upstash credentials are missing, falls back to in-memory only.

## Multi-Tenant Prisma Adapter

The standard Prisma adapter from `@auth/prisma-adapter` doesn't support multi-tenant schemas with composite unique constraints. Our User model uses:

```prisma
@@unique([email, schoolId])  // Same email allowed across different schools
```

This causes the standard adapter to fail because it queries `findUnique({ where: { email } })` which requires a simple `@unique email` constraint.

### Custom Adapter Solution

We use a custom `MultiTenantPrismaAdapter` that overrides key methods:

```typescript
import { MultiTenantPrismaAdapter } from "@/lib/multi-tenant-prisma-adapter"

export const { auth, signIn, signOut } = NextAuth({
  adapter: MultiTenantPrismaAdapter(db),
  // ...
})
```

### How It Works

| Method             | Standard Adapter                    | Multi-Tenant Adapter                              |
| ------------------ | ----------------------------------- | ------------------------------------------------- |
| `getUserByEmail`   | `findUnique({ where: { email } })`  | `findFirst({ where: { email, schoolId: null } })` |
| `createUser`       | Creates user without school context | Creates with `schoolId: null` explicitly          |
| `getUserByAccount` | Standard lookup                     | Handles composite constraints                     |
| `linkAccount`      | Standard linking                    | Logs for debugging                                |

### OAuth User Lifecycle

1. **OAuth Callback**: User authenticates with Google/Facebook
2. **Adapter Check**: `getUserByEmail` looks for existing user with `schoolId: null`
3. **Create/Link**: New users created with `schoolId: null`, existing users linked
4. **Onboarding**: User goes through onboarding to join/create a school
5. **School Assignment**: `schoolId` updated when user joins a school

This allows the same email to exist in multiple schools while OAuth users start in a "limbo" state until onboarded.

### Redirect Logic Implementation

The redirect logic is handled in `src/auth.ts` in the `redirect` callback:

```typescript
callbacks: {
  async redirect({ url, baseUrl }) {
    // 1. Check for stored callback URL (from OAuth flow)
    const callbackUrl = cookies.get("auth-callback-url")
    if (callbackUrl) {
      // Clear cookie and use stored URL
      return callbackUrl
    }

    // 2. If user has schoolId, redirect to their school
    const session = await auth()
    if (session?.user?.schoolId) {
      const schoolDomain = await getSchoolDomain(session.user.schoolId)
      if (schoolDomain) {
        return constructSchoolUrl(schoolDomain, "/dashboard")
      }
    }

    // 3. No school - send to homepage (onboarding via "Get Started" only)
    return `${baseUrl}/`
  }
}
```

### Cookie-Based Callback URL Preservation

OAuth flows redirect through external providers, losing the original URL. We preserve it using an httpOnly cookie:

```typescript
// Before OAuth redirect (in social.tsx)
document.cookie = `auth-callback-url=${window.location.href}; path=/; secure; samesite=lax`

// After OAuth callback (in redirect callback)
const callbackUrl = cookies.get("auth-callback-url")
if (callbackUrl) {
  cookies.delete("auth-callback-url")
  return callbackUrl
}
```

## Troubleshooting

### OAuth Issues

**redirect_uri_mismatch**: Callback URL in provider console doesn't match application

- Verify exact URL including protocol and port
- No trailing slashes

**access_denied**: User denied permission or app misconfigured

- Check OAuth consent screen
- Verify required scopes

**Configuration error**: Check the debug endpoint first

```bash
curl https://ed.databayt.org/api/debug-auth | jq '.config'
```

Common causes:

- Missing `AUTH_SECRET` or too short (< 32 chars)
- Missing OAuth credentials
- Invalid `NEXTAUTH_URL`

### Session Issues

**User not persisting**: Cookie not being set

- Check domain configuration
- Verify AUTH_SECRET is set

**Role not updating**: Session not refreshed

- Force session refresh after role change
- Check JWT callback

### Multi-Tenant Issues

**Wrong school context**: Query returning other school's data

- Always include schoolId in where clause
- Use getTenantContext()

## Join Code System

Schools can generate 6-character join codes for easy member enrollment.

### Code Generation

```typescript
import { generateUniqueJoinCode } from "@/lib/join-code"

const code = await generateUniqueJoinCode()
// e.g., "ABC123" - 6 chars from ABCDEFGHJKMNPQRSTUVWXYZ2345679
```

- **28-char unambiguous alphabet** (excludes 0/O, 1/I/L to avoid confusion)
- **481M possible codes** (28^6) with 5-retry collision resistance
- Codes are stored on `School.joinCode` (`@@unique`)

### Join Flow

```
User enters code → joinSchoolByCode(userId, code)
  │
  ├── Code valid + allowSelfEnrollment: true
  │     → User linked to school, USER → STAFF promotion
  │
  └── Code valid + allowSelfEnrollment: false
        → MembershipRequest created (PENDING status)
        → Admin must approve before user gains access
```

### Self-Enrollment Toggle

Controlled by `SchoolBranding.allowSelfEnrollment`:

- **Enabled**: Users with valid code are immediately linked to school
- **Disabled**: Valid code creates a pending membership request for admin review

## Brute Force Protection

Login attempts are tracked and rate-limited via `src/lib/audit-log.ts`.

### How It Works

```typescript
import { isBruteForceBlocked, logLoginAttempt } from "@/lib/audit-log"

// Before credential validation:
const blocked = await isBruteForceBlocked(email)
if (blocked) return { error: "Too many login attempts. Try again later." }

// After credential validation:
await logLoginAttempt({ email, success: true / false, failureReason, ip })
```

- **Threshold**: 5 failed attempts within 15 minutes → blocked
- **Scope**: Per-email (IP-based blocking is supported but not enforced by default)
- **Storage**: `LoginAttempt` model with `email`, `success`, `failureReason`, `ip`, `timestamp`
- **Recovery**: Block automatically lifts after 15 minutes of no failed attempts

Preview mode allows admins to see the UI as a different role without changing their actual role.

### Implementation

- **Storage**: `preview-mode` and `preview-role` cookies + localStorage
- **Session override**: The session callback in `src/auth.ts` checks for preview cookies and overrides the role in the session object
- **Flag**: `session.user.isPreviewMode = true` when active

### Hooks (from `src/components/auth/use-preview-role.ts`)

| Hook               | Returns                                                     | Purpose                             |
| ------------------ | ----------------------------------------------------------- | ----------------------------------- |
| `useEffectiveRole` | `UserRole`                                                  | Preview role if active, else actual |
| `useIsPreviewMode` | `boolean`                                                   | Whether preview mode is active      |
| `useRoleInfo`      | `{ actualRole, previewRole, effectiveRole, isPreviewMode }` | Full role context                   |

## Edge Middleware Auth

The edge middleware (`src/proxy.ts`) performs lightweight JWT decoding for route protection without hitting the database.

### Flow

```
Request → Edge Middleware
  │
  ├── Decode JWT (lightweight, no DB call)
  ├── Extract: role, schoolId
  ├── Check roleRoutes matrix for route access
  ├── Detect subdomain (custom domain → Redis lookup)
  ├── Set x-subdomain header for downstream
  │
  └── PASS → Next.js renders page
      FAIL → Redirect to /login or /unauthorized
```

### Custom Domain Support

Custom domains are resolved via Redis key `custom-domain:${host}` → subdomain, handled in `src/proxy.ts`.

### Known Issues (Documented in Tests)

The test suite documents multi-tenant considerations:

1. ~~**`getUserByEmail` not tenant-scoped**~~ - **FIXED**: Custom `MultiTenantPrismaAdapter` queries with `{ email, schoolId: null }` for OAuth users
2. **`getUserByEmail` uses `findMany` + prefers password user** - By design: Uses `findMany` (not `findUnique`) because of composite `@@unique([email, schoolId])`. Prefers the user with a password hash for credentials login. This is intentional to support multi-tenant OAuth where the same email can exist in multiple schools.
3. **OAuth users created without schoolId** - By design: New OAuth users exist in limbo until onboarding completes
4. **Registration lacks school context** - By design: Users join schools through onboarding or invitation flow

See [ISSUE.md](/src/components/auth/ISSUE.md) for current test status.
