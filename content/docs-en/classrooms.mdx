## Overview

The Classrooms module manages physical rooms across a school campus. Each classroom has a type, capacity, optional grade affinity, and integrates with sections for homeroom grouping. The module supports role-based access (DEVELOPER, ADMIN full CRUD; TEACHER, STAFF read-only), a configure tab for bulk section+room generation, room detail pages with weekly timetable grids, and capacity analytics.

**Current status:** Schema, seed data, CRUD, room detail, section generation, capacity overview, and timetable integration are production. The schedule tab shows class timetables (not room schedules), the subjects tab is a placeholder, and the facility page remains static.

### Architecture

```
School
 ├── classrooms[] ──→ Classroom
 │                     ├── gradeId? ──→ AcademicGrade (grade affinity)
 │                     ├── sections[] ──→ Section (homeroom groups)
 │                     ├── classes[] ──→ Class (course sections)
 │                     ├── timetables[] ──→ Timetable (per-slot assignment)
 │                     └── constraints[] ──→ RoomConstraint
 │
 ├── sections[] ──→ Section
 │                   ├── gradeId ──→ AcademicGrade (required)
 │                   ├── classroomId? ──→ Classroom (assigned room)
 │                   ├── homeroomTeacherId? ──→ Teacher
 │                   └── students[] ──→ Student (homeroom roster)
 │
 ├── teachers[] ──→ Teacher
 │                   ├── classes[] ──→ Class ──→ Classroom (home room)
 │                   ├── timetables[] ──→ Timetable ──→ Classroom (per-slot)
 │                   └── homeroomSections[] ──→ Section ──→ Classroom
 │
 ├── students[] ──→ Student
 │                   ├── studentClasses[] ──→ StudentClass ──→ Class ──→ Classroom
 │                   └── sectionId? ──→ Section ──→ Classroom (homeroom)
 │
 └── maxClasses: Int (plan limit for classroom count)
```

## Data Models

### Classroom

The core model representing a physical room in the school.

```prisma
model Classroom {
  id       String  @id @default(cuid())
  schoolId String
  typeId   String
  gradeId  String?
  roomName String
  capacity Int
  lang     String  @default("ar")

  school        School         @relation(fields: [schoolId], references: [id], onDelete: Cascade)
  classroomType ClassroomType  @relation(fields: [typeId], references: [id])
  grade         AcademicGrade? @relation(fields: [gradeId], references: [id], onDelete: SetNull)
  classes       Class[]
  sections      Section[]
  timetables    Timetable[]
  constraints   RoomConstraint[]

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@unique([schoolId, roomName])
  @@index([schoolId, gradeId])
  @@map("classrooms")
}
```

| Field      | Type      | Description                                         |
| ---------- | --------- | --------------------------------------------------- |
| `id`       | `String`  | CUID primary key                                    |
| `schoolId` | `String`  | Tenant isolation (multi-tenant safety)              |
| `typeId`   | `String`  | FK to ClassroomType                                 |
| `gradeId`  | `String?` | Optional FK to AcademicGrade (grade affinity)       |
| `roomName` | `String`  | Display name (e.g., "A101", "Physics Lab")          |
| `capacity` | `Int`     | Maximum student count                               |
| `lang`     | `String`  | Storage language for translation (`@default("ar")`) |

**Unique constraint:** `@@unique([schoolId, roomName])` -- no duplicate room names within a school.

**Index:** `@@index([schoolId, gradeId])` -- fast grade-filtered queries.

### Section

A homeroom group within a grade. Sections represent divisions like "Grade 7-A", "Grade 7-B" and link students to a specific classroom and homeroom teacher.

```prisma
model Section {
  id        String @id @default(cuid())
  schoolId  String
  gradeId   String
  name      String   // "Grade 7-A"
  letter    String   // "A"
  lang      String   @default("ar")

  homeroomTeacherId String?
  classroomId       String?
  maxCapacity       Int     @default(30)

  school          School         @relation(fields: [schoolId], references: [id], onDelete: Cascade)
  grade           AcademicGrade  @relation(fields: [gradeId], references: [id], onDelete: Cascade)
  homeroomTeacher Teacher?       @relation("HomeroomTeacher", fields: [homeroomTeacherId], references: [id], onDelete: SetNull)
  classroom       Classroom?     @relation(fields: [classroomId], references: [id], onDelete: SetNull)
  students        Student[]

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@unique([schoolId, gradeId, letter])
  @@unique([schoolId, name])
  @@index([schoolId])
  @@index([schoolId, gradeId])
  @@map("sections")
}
```

| Field               | Type      | Description                               |
| ------------------- | --------- | ----------------------------------------- |
| `gradeId`           | `String`  | Required FK to AcademicGrade              |
| `name`              | `String`  | Full name (e.g., "Grade 7-A")             |
| `letter`            | `String`  | Section letter (e.g., "A")                |
| `homeroomTeacherId` | `String?` | Optional FK to Teacher (homeroom teacher) |
| `classroomId`       | `String?` | Optional FK to Classroom (assigned room)  |
| `maxCapacity`       | `Int`     | Section student limit (`@default(30)`)    |

**Relationship chain:** AcademicGrade -> Section -> Student, Section -> Classroom, Section -> Teacher (homeroom).

**Unique constraints:** `[schoolId, gradeId, letter]` (one "A" per grade per school), `[schoolId, name]` (unique section name per school).

### ClassroomType

Categorizes rooms for filtering and scheduling.

```prisma
model ClassroomType {
  id       String @id @default(cuid())
  schoolId String
  name     String
  lang     String @default("ar")

  school     School      @relation(fields: [schoolId], references: [id], onDelete: Cascade)
  classrooms Classroom[]

  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt

  @@unique([schoolId, name])
  @@map("classroom_types")
}
```

**8 seeded types:** `classroom`, `lab`, `library`, `art`, `music`, `hall`, `sports`, `admin`

### RoomConstraint

Scheduling constraints and accessibility metadata for a room. Defined in `prisma/models/timetable.prisma`.

```prisma
model RoomConstraint {
  id          String  @id @default(cuid())
  schoolId    String
  classroomId String
  termId      String?

  allowedSubjectTypes  String[] @default([])
  strictCapacityLimit  Boolean  @default(true)
  capacityBuffer       Int      @default(0)

  wheelchairAccessible Boolean @default(false)
  hasElevatorAccess    Boolean @default(false)
  floorLevel           Int?

  reservedPeriods   Json @default("{}")
  maintenanceBlocks Json @default("[]")

  @@unique([schoolId, classroomId, termId])
  @@map("room_constraints")
}
```

| Field                  | Type       | Description                                                                               |
| ---------------------- | ---------- | ----------------------------------------------------------------------------------------- |
| `allowedSubjectTypes`  | `String[]` | Restrict room to specific subject types (loaded but not consumed by scheduling algorithm) |
| `strictCapacityLimit`  | `Boolean`  | Enforce hard capacity limit during scheduling                                             |
| `capacityBuffer`       | `Int`      | Allow N extra students beyond capacity                                                    |
| `wheelchairAccessible` | `Boolean`  | Accessibility flag                                                                        |
| `hasElevatorAccess`    | `Boolean`  | Elevator access flag                                                                      |
| `floorLevel`           | `Int?`     | Floor number for accessibility routing                                                    |
| `reservedPeriods`      | `Json`     | `{dayOfWeek: [periodId, periodId]}` -- blocked time slots                                 |
| `maintenanceBlocks`    | `Json`     | `[{dayOfWeek, periodId, reason, startDate?, endDate?}]`                                   |

## Role-Based Access

Authorization is implemented in `src/components/school-dashboard/listings/classrooms/authorization.ts` using the `checkClassroomPermission()` function.

### Permission Matrix

| Role       | create           | read             | update           | delete           |
| ---------- | ---------------- | ---------------- | ---------------- | ---------------- |
| DEVELOPER  | Yes              | Yes              | Yes              | Yes              |
| ADMIN      | Yes (own school) | Yes (own school) | Yes (own school) | Yes (own school) |
| TEACHER    | No               | Yes (own school) | No               | No               |
| STAFF      | No               | Yes (own school) | No               | No               |
| STUDENT    | No               | No               | No               | No               |
| GUARDIAN   | No               | No               | No               | No               |
| ACCOUNTANT | No               | No               | No               | No               |

### Data Paths per Role

How each role reaches classroom data:

**Admin** -- Direct access via classroom listing + room detail pages. Full CRUD through server actions with `assertClassroomPermission()`.

**Teacher** -- Read-only listing + room names from `Timetable` (filtered by `teacherId`) + dashboard today's classes. Also reaches classrooms via `Class.classroomId` for their assigned classes. Homeroom section via `Section.homeroomTeacherId`.

**Student** -- Room names from `Timetable` (filtered by `classIds` via `StudentClass`) + homeroom via `Student.sectionId -> Section.classroomId` (schema only, no UI). No direct classroom listing access.

**Guardian** -- Room names from child's timetable via parent portal (`getChildTimetable()`). No direct classroom access.

**Staff** -- Read-only classroom listing (same as teacher).

### RBAC Gaps

| Issue                                  | Description                                                                       |
| -------------------------------------- | --------------------------------------------------------------------------------- |
| Edit/Delete visible to read-only roles | Backend blocks TEACHER/STAFF mutations, but action buttons still render in the UI |
| Create button visible to all           | The "Add Room" button shows for read-only roles                                   |
| `getRoomTimetable()` skips permission  | Only checks session exists, no `assertClassroomPermission("read")`                |
| `getGrades()` no auth check            | Returns grades for any authenticated user with tenant context                     |
| `/api/classrooms` no auth check        | Only rate limiting + tenant context, no session/role verification                 |

## Grade-Based Features

### Grade Affinity

`Classroom.gradeId` optionally links a room to a specific `AcademicGrade`. When set, the room is considered "grade-assigned" (e.g., all Grade 7 sections use rooms in Building B). When `null`, the room is "Shared" -- available to any grade.

Grade affinity is displayed in:

- **Listing table**: Grade column shows grade name or "Shared" badge
- **Room detail**: Grade badge in room header
- **Capacity overview**: Grade column in utilization table

### Section Generation (Configure Tab)

The Configure tab (`/classrooms/configure`) bulk-creates Section + Classroom pairs per grade. This is the primary way to set up classroom-to-grade mappings.

**Flow:**

1. Admin selects grades and configures sections per grade, capacity per section, and room type
2. `generateSections()` runs in a `db.$transaction()`:
   - For each grade, determines how many new sections are needed (skips existing)
   - Assigns section letters (A, B, C...) skipping already-used letters
   - Creates (or upserts) a Classroom: `"Grade {N} - Room {letter}"`
   - Creates a Section: `"Grade {N}-{letter}"` linked to the classroom
   - Validates section capacity is less than or equal to room capacity
3. Enforces `School.maxClasses` plan limit before creating

**School defaults:** Uses `School.sectionsPerGrade` and `School.studentsPerSection` as form defaults.

**Naming convention:**

- Room: `"Grade {gradeNumber} - Room {letter}"` (e.g., "Grade 7 - Room A")
- Section: `"Grade {gradeNumber}-{letter}"` (e.g., "Grade 7-A")

### Grade-Based Gaps

| Gap                                   | Description                                                                |
| ------------------------------------- | -------------------------------------------------------------------------- |
| No grade-based filtering for teachers | Teachers can't filter to "my grades" rooms                                 |
| No "my rooms" teacher view            | No personalized room list based on timetable                               |
| No homeroom UI                        | Student -> Section -> Classroom path exists in schema but has no UI        |
| No section CRUD                       | Sections can only be bulk-created via Configure, no individual edit/delete |

## Tab Navigation

The classroom layout (`layout.tsx`) provides 5 tabs:

| Tab       | Route                   | Status          | Description                                                                                    |
| --------- | ----------------------- | --------------- | ---------------------------------------------------------------------------------------------- |
| Rooms     | `/classrooms`           | **Production**  | DataTable with CRUD, search, type filter                                                       |
| Configure | `/classrooms/configure` | **Production**  | Bulk section + room generation per grade                                                       |
| Capacity  | `/classrooms/capacity`  | **Production**  | Summary stats (total, capacity, utilization) + per-room table                                  |
| Subjects  | `/classrooms/subjects`  | **Placeholder** | Imports from `classes/subjects/content` (class-subject assignments, not room-subject mappings) |
| Schedule  | `/classrooms/schedule`  | **Partial**     | Shows class timetables grouped by grade, not room-specific schedules                           |

### Tab Details

**Rooms** -- Standard listing pattern with DataTable. Columns: room name, type, grade, capacity, class count, timetable count. Supports pagination, name search, type filter. Create/edit/delete via dialog form.

**Configure** -- Server component loads grade configuration via `getGradeConfiguration()`, renders `ConfigureForm` with per-grade section count, capacity, and room type. Submits to `generateSections()` which atomically creates sections and rooms.

**Capacity** -- Summary cards (total rooms, total capacity, avg capacity, avg utilization) + detailed table with per-room utilization. Utilization = `classCount / capacity` (note: this metric is classes-per-room, not time-based occupancy).

**Subjects** -- Placeholder page with TODO comment. Currently imports `ClassSubjectsContent` from the classes module, showing class-subject assignments rather than room-subject mappings.

**Schedule** -- Queries timetable slots for the active term, groups classes by grade, and renders `ScheduleGrid`. Shows class schedules, not room schedules (no room-centric weekly view here -- that exists in the timetable module's by-room view).

## Integrations

### Timetable

Classrooms are a core dimension of the timetable. Every timetable slot assigns a room.

**Double-booking prevention** -- The Timetable model has 3 unique constraints:

```
@@unique([schoolId, termId, dayOfWeek, periodId, classId, weekOffset])
@@unique([schoolId, termId, dayOfWeek, periodId, teacherId, weekOffset])
@@unique([schoolId, termId, dayOfWeek, periodId, classroomId, weekOffset])
```

The third constraint ensures no room is assigned to two classes in the same period.

**Room detail timetable** -- The room detail page (`/classrooms/[id]`) has its own weekly timetable grid showing all slots for that room, with class name, subject, teacher, and grade. Utilization is calculated as `usedSlots / (teachingPeriods * workingDays)`.

**Timetable room view** -- The timetable module has a dedicated "Room view" (`by-room/content.tsx`) with a full weekly schedule and utilization widget.

**Slot editor** -- `slot-editor-dialog.tsx` filters rooms by time-slot occupancy so only available rooms appear in the dropdown.

**Auto-generation** -- `algorithm.ts` has a `requiresLab` heuristic that routes science subjects to lab-type rooms. The `allowedSubjectTypes` field on `RoomConstraint` is loaded but not consumed by `isRoomAvailable()`.

### Section

Sections link students to classrooms through a homeroom model:

**Path:** AcademicGrade -> Section -> Classroom (via `classroomId`)

**Path:** Student -> Section -> Classroom (via `sectionId` -> `classroomId`)

Sections are created via the Configure tab's `generateSections()` action, which creates both the Section and its assigned Classroom atomically.

### Classes

Each `Class` model has a **required** `classroomId` field for its default room assignment:

```prisma
model Class {
  classroomId String
  classroom   Classroom @relation(fields: [classroomId], references: [id])
}
```

### Students

Students reach classrooms through two paths:

- **Course path:** Student -> StudentClass -> Class -> Classroom
- **Homeroom path:** Student -> Section -> Classroom (schema only, no UI)

### Teachers

Teachers reach classrooms through three paths:

- **Home room:** Teacher -> Class -> Classroom (assigned classes' default rooms)
- **Per-slot room:** Teacher -> Timetable -> Classroom (actual room per period)
- **Homeroom:** Teacher -> Section -> Classroom (via `homeroomTeacherId`)

### Parent Portal

Guardians see classroom information via their children's timetables:

**Path:** Guardian -> Student -> Timetable -> Classroom (via `getChildTimetable()`)

### Exams

No direct `classroomId` on the Exam model. Venue resolved via:

**Path:** Exam -> Class -> Classroom

Conflict detection accepts optional `classroomId` and checks timetable occupancy + exam-to-exam conflicts.

### Dashboard

The school dashboard's `UpcomingClassCard` shows a "Room" field with **hardcoded mock data** ("Room 101", "Hall A"). Does not query from `Timetable` or `Classroom` tables.

### Facility Page

The facility management page is **100% hardcoded** static data (42 facilities, static numbers). Zero connection to the `Classroom` model.

## API

### `GET /api/classrooms`

Returns a list of classrooms for dropdown selection (timetable slot assignment, event booking, exam venue allocation).

**Location:** `src/app/api/classrooms/route.ts`

**Multi-tenant safety:** Scoped by `schoolId` from tenant context. Returns empty array if no schoolId.

**Rate limiting:** API tier rate limits applied.

**Auth:** No `auth()` check -- only rate limiting + tenant context. Any request with valid tenant context can list rooms.

**Response:**

```json
{
  "classrooms": [
    { "id": "clx...", "roomName": "A101" },
    { "id": "clx...", "roomName": "Physics Lab" }
  ]
}
```
