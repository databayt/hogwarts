The timetable module handles school schedules end-to-end: choose a structure during onboarding, configure working days and periods, assign weekly slots, detect conflicts, generate via AI, export to PDF, and serve role-specific views. All operations are scoped by `schoolId`.

## Role-based views

`content-production.tsx` wraps `RoleRouter` in `SessionProvider`. `RoleRouter` calls `getPersonalizedTimetable()` to determine `viewType` + `editable` and renders the matching view.

| Role       | View           | Data                                           | Sees                                              |
| ---------- | -------------- | ---------------------------------------------- | ------------------------------------------------- |
| DEVELOPER  | `AdminView`    | `getTimetableByRoom` / `getTimetableByTeacher` | Room / teacher grid, editable slots, slot editor  |
| ADMIN      | `AdminView`    | Same as DEVELOPER                              | Same as DEVELOPER                                 |
| TEACHER    | `TeacherView`  | `getTimetableByTeacher`                        | Personal schedule, workload, current / next class |
| STUDENT    | `StudentView`  | `getTimetableByStudentGrade`                   | All enrolled subjects (grade-based), PDF export   |
| GUARDIAN   | `GuardianView` | `getChildTimetable`                            | Child selector + each child's enrolled classes    |
| ACCOUNTANT | `AdminView`    | Same as ADMIN                                  | Full admin view, read-only (`editable={false}`)   |
| STAFF      | `AdminView`    | Same as ADMIN                                  | Full admin view, read-only                        |

Routing: role → `viewType` switch + `editable` flag (default `true`); ACCOUNTANT / STAFF get `editable: false`. `AdminView` passes `editable` to `SimpleGrid` and conditionally disables `onSlotClick`.

Known bugs: admin views lack a class / grade filter mode; teacher view has no export button despite `canExport` permission.

## Student grade resolution

Student timetable resolves through enrollment:

```
User (session)
  → Student (db.student.findFirst where userId)
    → StudentClass (db.studentClass.findMany where studentId)
      → classIds[]
        → Timetable (db.timetable.findMany where classId IN classIds)
```

Grade name: `Student → StudentYearLevel → YearLevel.levelName`.

## Onboarding schedule step

Inserted between Capacity (order 5) and Branding (order 6).

```
1. Read school country, schoolType, schoolLevel
2. getRecommendedStructures(...) — show top recommendations + alternatives
3. User picks (auto-saves to school.system) or skips
4. "Next" always enabled — step is optional
```

Each structure card shows name, description, period count, time range, working days, color-coded timeline (blue = class, amber = break, green = lunch), "Recommended" badge.

## Applying a structure

`applyTimetableStructure()` creates Period records and updates `SchoolWeekConfig`.

```typescript
await applyTimetableStructure({
  yearId: "clx...",
  structureSlug: "sd-gov-default",
  replaceExisting: true,
})

// Legacy template names map automatically via LEGACY_TEMPLATE_MAP
await applyTimetableStructure({
  yearId: "clx...",
  structureSlug: "standard_8", // → sd-gov-default
})
```

Steps: validate admin + tenant, look up structure (legacy map), optionally delete existing periods (`replaceExisting`), create `Period` rows for each entry (class, break, lunch), upsert `SchoolWeekConfig` with working days + lunch position, log audit entry.
