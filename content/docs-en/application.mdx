## Pipeline Overview (Applicant Side)

```
Applicant                                         System
────────                                         ──────
Visit school site ──────────────────────────────────────
Select campaign ────────────────────────────────────────
Complete 6-step wizard ─────────────────────────────────
Submit application ──────────── Application created (SUBMITTED)
Pay fee (card/cash/bank) ────── Payment recorded
Track status (OTP) ──────────── Timeline view
Book school tour ────────────── Calendar invite sent
Submit inquiry ──────────────── Lead captured
```

## Application Flow (A to Z)

This section walks through the entire application flow from the applicant's first click to submission and payment.

### Step 1: Applicant Visits the School Site

The applicant navigates to the school's public site at `school.databayt.org` and clicks **Admissions** in the navigation. This leads to the admissions info page with a hero section, process overview, and a **Start Application** CTA.

**Route:** `/{lang}/admissions`

### Step 2: Campaign Selection

The applicant is taken to the application landing page where they see active admission campaigns (e.g., "Admission 2026-2027"). They can also see any saved draft applications if they're logged in.

Two options are presented:

- **Start from scratch** — begin a new application
- **Import from profile** — auto-fill from documents or LinkedIn (same flow, pre-populated)

**Route:** `/{lang}/application`

### Step 3: Overview Page

After selecting a campaign, the applicant sees an overview of the 3 phases:

| Phase | Label              | Steps                 |
| ----- | ------------------ | --------------------- |
| 1     | Basic Information  | Attachments, Personal |
| 2     | Details            | Contact, Location     |
| 3     | Family & Education | Guardian, Academic    |

Clicking **Get Started** initializes a session and enters the wizard.

**Route:** `/{lang}/application/overview?id={campaignId}`

### Step 4: The 6-Step Wizard

The wizard uses a shared `FormFooter` with 3 progress bars (one per phase). Each step auto-saves to the server every 30 seconds and persists to localStorage as a backup.

| Birth Certificate    | Document                 |
| Passport Photo       | Document                 |
| Signature            | Document                 |
| Medical Certificate  | Document                 |
| Transfer Certificate | Document                 |
| Other                | Document                 |

**Route:** `/{lang}/application/{campaignId}/attachments`

#### Step 2: Personal Information (Required)

| Field         | Required | Validation            |
| ------------- | -------- | --------------------- |
| First Name    | Yes      | 2-50 chars            |
| Last Name     | Yes      | 2-50 chars            |
| Date of Birth | Yes      | Date picker           |
| Gender        | Yes      | MALE / FEMALE / OTHER |
| Nationality   | Yes      | Country selector      |

| Category      | No       | Dropdown              |

**Next enabled when:** firstName, lastName, dateOfBirth, gender, nationality are all filled.

**Route:** `/{lang}/application/{campaignId}/personal`

#### Step 6: Academic Information (Final Step — Triggers Submit)

Organized in 3 sections: Previous Education, Applying For, Languages.

**Available Grades:** KG1 through Grade 12 (14 levels, Arabic labels).

The final button says **"Submit Application"** instead of "Next".

**Route:** `/{lang}/application/{campaignId}/academic`

### Step 5: Submit

When the applicant clicks "Submit Application", the system:

1. Saves the academic step data
2. Checks completeness across all steps (firstName, lastName, email, phone, fatherName, motherName, applyingForClass)
3. Flattens all step data into a single object
4. Calls `submitApplicationAction()` which delegates to `submitApplication()`

**Server-side submit flow:**

1. Resolve school from subdomain
2. Validate with `createFullApplicationSchema()` (Zod)
3. Verify session belongs to this school
4. Check campaign is still OPEN and not expired
5. Check for duplicate email + campaign combination
6. Generate unique application number (`APP-{year}-{random6}`)
7. Generate access token (32 chars, 6-month expiry)
8. Create `Application` record with status `SUBMITTED`
9. Mark session as converted
10. Send confirmation email via Resend
11. Check if payment is required (`campaign.applicationFee > 0`)

**Post-submit routing:** After successful submission, the academic step checks `result.requiresPayment`:

- **Payment required** → redirects to `/{lang}/application/{applicationId}/payment?number={applicationNumber}`
- **No payment** → redirects to `/{lang}/application/{applicationId}/success?number={applicationNumber}`

The layout's `FormFooter` no longer has a `finalDestination` — the academic step's `onNext` callback handles all post-submit navigation.

### Step 6: Payment (If Required)

If the campaign has an application fee, the applicant is redirected to the payment page.

| Payment Method | Description                              |
| -------------- | ---------------------------------------- |
| Pay with Card  | Stripe Checkout redirect                 |
| Pay at School  | Cash payment — shows reference number    |
| Bank Transfer  | Shows bank details with reference number |

After selecting cash or bank transfer, a confirmation card with the reference number is shown. A "Continue" button navigates to the success page.

**Route:** `/{lang}/application/{applicationId}/payment?number={applicationNumber}`

### Step 7: Success Page

Displays:

- Confetti celebration modal on first visit (animated via `success-modal.tsx`)
- Application number in a card
- **"Complete Payment" CTA** if payment is required but not yet received — links to the payment page
- Payment status (paid/pending) with method-specific details
- Next steps timeline (email confirmation → review → interview → decision)
- Links to explore school and schedule a school tour

**Route:** `/{lang}/application/{applicationId}/success?number={applicationNumber}`

## Application Status Banner

When a logged-in applicant visits the school site, a banner displays their application status at the top of the page. The banner is:

- **Server-rendered** — queries the most recent non-draft application for the current user
- **Client-interactive** — dismissible via localStorage per application
- **Status-aware** — shows different messages and CTAs by status

| Status          | Banner Message                                 | CTA                                   |
| --------------- | ---------------------------------------------- | ------------------------------------- |
| SUBMITTED       | "Application #\{number\} — Submitted"          | None                                  |
| UNDER_REVIEW    | "Application #\{number\} — Under Review"       | None                                  |
| SHORTLISTED     | "Application #\{number\} — Shortlisted"        | None                                  |
| SELECTED        | "Congratulations! You've been accepted"        | **"Complete Payment"** → payment page |
| SELECTED + paid | "Congratulations! Accepted — Payment received" | **"Go to Dashboard"**                 |
| ADMITTED        | "Welcome! Enrollment confirmed"                | **"Go to Dashboard"**                 |
| WAITLISTED      | "Application #\{number\} — Waitlisted"         | None                                  |
| REJECTED        | "Application #\{number\} — Not Selected"       | None                                  |

The banner includes a **timeline progress indicator** (4 dots: Submitted → Review → Decision → Enrolled) showing the applicant's position in the pipeline.

## Session Management

### How Sessions Work

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  localStorage │ ←──→ │   Context    │ ←──→ │  Database    │
│  (backup)     │     │  (in-memory) │     │  (source of  │
│               │     │              │     │   truth)     │
└──────────────┘     └──────────────┘     └──────────────┘
```

**Storage key:** `hogwarts_apply_session_{campaignId}`

**Session state:**

- `sessionToken` — server-generated nanoid(32), null until first save
- `campaignId` — which admission campaign
- `formData` — keyed by step name (attachments, personal, contact, location, guardian, academic)
- `currentStep` — current step name
- `lastSaved`, `isDirty`, `isLoading`, `isSaving`, `error`

**Auto-save:** Every 30 seconds when `isDirty` is true.

**Session expiry:** 7 days.

**Resume methods:**

1. **localStorage** — automatic on page load
2. **Email link** — `/{lang}/application/continue?token=xxx`
3. **User account** — authenticated users see their drafts on the application landing page

**Unsaved changes:** The browser warns before navigating away when `isDirty` is true.

### First Save Email

On the first successful save, the system sends a "resume application" email with a token link so the applicant can continue from any device.

## Status Tracking

### OTP Verification

Parents can track their application status without creating an account:

1. Navigate to `/{lang}/application/status`
2. Enter application number + email
3. Receive 6-digit OTP via email (10-minute expiry)
4. Rate limited: max 3 OTPs per hour, max 5 verification attempts
5. On verification, view application timeline

### Status Timeline

```
DRAFT → SUBMITTED → UNDER_REVIEW → SHORTLISTED → SELECTED → ADMITTED
                                  ↘ WAITLISTED
                                  ↘ REJECTED
                                  ↘ WITHDRAWN
```

The timeline shows each completed stage with dates and next steps.

3. **Booking Form** — collect visitor information (name, email, phone, student name, grade interest, number of attendees, special requests)
4. **Confirmation** — booking number, email with calendar invite

### Slot Types

| Type        | Purpose                 |
| ----------- | ----------------------- |

| INTERVIEW   | Admission interview     |
| ORIENTATION | New student orientation |

### Booking Management

Parents can view, reschedule, or cancel their bookings at `/{lang}/tour/{bookingId}`.

## Troubleshooting

**Application Not Saving**

- Check browser console for network errors
- Verify session hasn't expired (7-day limit)
- Ensure the school's subdomain resolves correctly

**Submit Fails with Validation Error**

- The error message includes field names (e.g., "Validation failed: postalCode: ...")
- Check that all required fields are filled across all steps
- The completeness check requires: firstName, lastName, email, phone, fatherName, motherName, applyingForClass

**OTP Not Received**

- Check spam folder
- Verify email matches the application
- Rate limit: max 3 OTPs per hour

**Payment Page Shows No Methods**

- Check `AdmissionSettings.paymentMethods` for the school
- Ensure campaign has `applicationFee > 0`
- Stripe requires webhook configuration for card payments
