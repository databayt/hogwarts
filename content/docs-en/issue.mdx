## The 6 Rules

### 1. One row per issue

Every issue is a single table row. No prose descriptions, no bullet-point lists.

```markdown
<!-- BAD: Prose description -->

## Missing schoolId in bulk delete

The bulk delete action in actions.ts doesn't include schoolId
in the where clause. This means a teacher could accidentally
delete records from another school...

<!-- GOOD: Table row -->

| 1 | P0 | Missing schoolId in bulk delete | Open | — |
```

### 2. Max 40 lines

If it exceeds 40 lines, archive completed issues or split into separate tracking.

```markdown
<!-- BAD: 500-line ISSUE.md with full descriptions for each issue -->

<!-- GOOD: ~40-line table with one row per issue -->

| ID  | P   | Title                           | Status      | Owner |
| --- | --- | ------------------------------- | ----------- | ----- |
| 1   | P0  | Missing schoolId in bulk delete | Open        | —     |
| 2   | P1  | Form resets on validation error | In Progress | @dev  |
```

### 3. Priority labels

Use consistent labels: P0 (blocking), P1 (next sprint), P2 (backlog), P3 (nice-to-have).

```markdown
<!-- BAD: Inconsistent priority -->

- CRITICAL: Fix login bug
- high: Add export feature
- Low priority: Update docs

<!-- GOOD: Consistent P0-P3 -->

| 1 | P0 | Fix login bug | Open | — |
| 2 | P1 | Add export feature | Open | — |
| 3 | P3 | Update docs | Open | — |
```

### 4. Auto-update on change

When fixing an issue, move the row to Completed with a date. When finding a new bug, add a row to Active.

```markdown
<!-- BAD: Issue fixed in code but ISSUE.md still shows it as Open -->

<!-- GOOD: Moved to Completed after fix -->

## Completed

| ID  | P   | Title               | Resolved   |
| --- | --- | ------------------- | ---------- |
| 0   | P0  | SSE on table render | 2026-02-15 |
```

### 5. Debt section separate

Technical debt (performance, accessibility, security, test gaps) is tracked separately from feature issues.

```markdown
<!-- BAD: Mixing debt with feature bugs -->

| 1 | P0 | Login broken | Open |
| 2 | P2 | No integration tests for actions.ts | Open |
| 3 | P1 | Missing aria-labels on buttons | Open |

<!-- GOOD: Separate sections -->

## Active

| 1 | P0 | Login broken | Open | — |

## Debt

| Test | No integration tests for actions.ts | Medium |
| A11y | Missing aria-labels on buttons | Low |
```

### 6. No duplicate tracking

If an issue is in GitHub Issues, link to it. Don't maintain two sources of truth.

```markdown
<!-- BAD: Full issue description duplicated from GitHub -->

## Authentication Bug

Same description as GitHub Issue #142...

<!-- GOOD: Link to GitHub -->

| 5 | P0 | Auth callback fails on redirect | Open | [#142](https://github.com/databayt/hogwarts/issues/142) |
```

## Canonical Template

The target ISSUE.md for every feature directory (~40 lines):

```markdown
# <Feature> Issues

> Health: green/yellow/red — One-line summary of current state.

## Active

| ID  | P   | Title                           | Status      | Owner |
| --- | --- | ------------------------------- | ----------- | ----- |
| 1   | P0  | Missing schoolId in bulk delete | Open        | —     |
| 2   | P1  | Form resets on validation error | In Progress | @dev  |
| 3   | P2  | Add export-to-CSV               | Open        | —     |

## Completed

| ID  | P   | Title               | Resolved   |
| --- | --- | ------------------- | ---------- |
| 0   | P0  | SSE on table render | 2026-02-15 |

## Debt

| Type | Description                           | Severity |
| ---- | ------------------------------------- | -------- |
| Test | No integration tests for actions.ts   | Medium   |
| A11y | Missing aria-labels on filter buttons | Low      |
| Perf | N+1 query in content.tsx line 45      | High     |

<!-- AUTO-UPDATE RULES
- After fixing an issue: move row from Active → Completed with date
- After finding a bug: add row to Active with next available ID
- After code review: update Debt section
- Health: green (0 P0), yellow (any P0), red (3+ P0 or blocking deploy)
-->
```

## Anti-Patterns

### 1. Prose descriptions of issues

Use table rows. The title column should be descriptive enough to understand the issue.

```markdown
<!-- BAD -->

## Form Validation Bug

When a user submits the form with invalid data, the form resets
instead of showing validation errors. This happens because the
React Hook Form state is being cleared...

<!-- GOOD -->

| 2 | P1 | Form resets on validation error instead of showing messages | In Progress | @dev |
```

### 2. Mixing feature issues with tech debt

Separate Active issues (bugs, features) from Debt (test gaps, a11y, performance).

```markdown
<!-- BAD: All in one table -->

| 1 | P0 | Login broken |
| 2 | P2 | Missing tests for actions.ts |
| 3 | P1 | N+1 query in content.tsx |

<!-- GOOD: Separate sections -->

## Active

| 1 | P0 | Login broken | Open | — |

## Debt

| Test | Missing tests for actions.ts | Medium |
| Perf | N+1 query in content.tsx | High |
```

### 3. ISSUE.md exceeding 40 lines

Archive completed issues aggressively. Only keep the last 5 completed items.

### 4. Stale issues that were fixed but never marked completed

The auto-update rule exists for this reason. After fixing a bug, move it to Completed immediately.

### 5. Duplicating GitHub Issues content

Link to the GitHub Issue instead. ISSUE.md is for local, co-located tracking of feature-specific issues.

```markdown
<!-- BAD: Full description copied from GitHub -->

## Authentication redirect fails on Safari

[300 words duplicated from GitHub Issue #142]

<!-- GOOD: One row with link -->

| 5 | P0 | Auth redirect fails on Safari | Open | [#142](url) |
```

## Quick Reference

| Section   | Content                            | Max Rows |
| --------- | ---------------------------------- | -------- |
| Active    | ID, priority, title, status, owner | 10       |
| Completed | ID, priority, title, resolved date | 5        |
| Debt      | Type, description, severity        | 5        |

### Priority Labels

| Label | Meaning                            |
| ----- | ---------------------------------- |
| P0    | Blocking — must fix before deploy  |
| P1    | Next sprint — fix within the week  |
| P2    | Backlog — fix when convenient      |
| P3    | Nice-to-have — fix if time permits |

### Health Indicator

| Color  | Criteria                            |
| ------ | ----------------------------------- |
| green  | 0 P0 issues, no blocking debt       |
| yellow | Any P0 issue, or high-severity debt |
| red    | 3+ P0 issues, or blocking deploy    |

### Debt Types

| Type     | Examples                               |
| -------- | -------------------------------------- |
| Test     | Missing unit/integration/e2e tests     |
| A11y     | Missing aria-labels, keyboard nav      |
| Perf     | N+1 queries, unnecessary re-renders    |
| Security | Missing input validation, XSS risks    |
| i18n     | Hardcoded strings, missing RTL support |

### What Belongs Where

| Content                | Belongs in    | Not in    |
| ---------------------- | ------------- | --------- |
| Bugs and features      | `ISSUE.md`    | README.md |
| Tech debt              | `ISSUE.md`    | Comments  |
| Architecture decisions | `README.md`   | ISSUE.md  |
| Feature purpose        | `README.md`   | ISSUE.md  |
| Detailed bug reports   | GitHub Issues | ISSUE.md  |
