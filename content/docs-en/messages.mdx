The messages module powers in-platform school communication: 5 conversation types, file attachments, emoji reactions, read receipts, full-text search, and full RBAC across all 8 user roles. Messages are dual-delivered to WhatsApp when the school has an active session, with bidirectional bridging for incoming replies.

## Routes

All under `src/app/[lang]/s/[subdomain]/(school-dashboard)/messages/`:

| Route                      | Roles | Purpose                              |
| -------------------------- | ----- | ------------------------------------ |
| `/messages`                | all   | Sidebar + chat interface             |
| `/messages?conversation=X` | all   | Deep link to a specific conversation |

## Conversation types

| Type           | Purpose                      | Auto-participants           | Key fields                        |
| -------------- | ---------------------------- | --------------------------- | --------------------------------- |
| `direct`       | 1-to-1 private               | both users                  | `directParticipant1Id`, `2Id`     |
| `group`        | Multi-user group             | creator as owner            | `title`, `description`, `avatar`  |
| `class`        | Class-bound (teacher-scoped) | teacher + enrolled students | `classId`                         |
| `department`   | Department-wide channel      | department members          | `departmentId`                    |
| `announcement` | Admin / DEVELOPER broadcast  | all school members          | `title`; read-only for non-admins |

Direct conversations dedup via `directParticipant1Id` / `directParticipant2Id` — `getOrCreateDirectConversation()` checks both orderings before creating.

System messages auto-generate for lifecycle events: `participant_joined`, `participant_left`, `participant_added`, `participant_removed`, `conversation_created`, `conversation_renamed`, `conversation_archived`, `message_pinned`, `message_unpinned`.

## Message lifecycle

| Status      | Meaning                                        |
| ----------- | ---------------------------------------------- |
| `sending`   | Client-side optimistic                         |
| `sent`      | Server received and persisted                  |
| `delivered` | Delivered to recipient's client                |
| `read`      | Recipient has viewed                           |
| `failed`    | Send failed (validation, network, permissions) |

| Content type | Rendering                      |
| ------------ | ------------------------------ |
| `text`       | Plain text with link detection |
| `image`      | Inline preview with lightbox   |
| `file`       | File card with download link   |
| `system`     | Centred, muted styling         |

Messages soft-delete via `isDeleted: true` + `deletedAt`. The UI shows "This message was deleted." Only the sender (or owner / admin) can delete. Edits set `isEdited: true` + `editedAt`; the UI shows an "edited" indicator.

## File attachments

Drag-and-drop uploads with previews, progress bars, S3 / Vercel Blob storage. Path: `messaging/{schoolId}/{conversationId}/{filename}`.

## WhatsApp integration

When the school has an active WhatsApp session, in-app messages auto-dispatch to WhatsApp via Evolution API. Bidirectional — incoming replies bridge back into conversations through a webhook.

Phone resolution priority: `Guardian.guardianPhoneNumbers` → `Teacher.teacherPhoneNumbers` → `StaffMember.staffPhoneNumbers`. Users without a domain record send in-app only.

UI surfaces:

| Element     | Location          | Purpose                                                     |
| ----------- | ----------------- | ----------------------------------------------------------- |
| W toggle    | Chat header       | Enable / disable WhatsApp per conversation                  |
| W badge     | Conversation card | Marks WhatsApp-enabled conversations in the sidebar         |
| Status icon | Message bubble    | WhatsApp delivery status (sent / delivered / read / failed) |

Rate limits: 1 message / second / school; 500 direct / day / school. Failed dispatches retry with exponential backoff (max 5 attempts) via `cron/process-whatsapp-notifications` (every 5 minutes).

Key files: `whatsapp-bridge.ts`, `src/lib/whatsapp/evolution-client.ts`, `src/app/api/webhooks/whatsapp/route.ts`, `src/app/api/cron/process-whatsapp-notifications/route.ts`, `src/components/school-dashboard/whatsapp/`.

## Contacts sidebar

WhatsApp-style contact list grouped by role. Click a contact → `createConversation({ type: "direct", participantIds: [contactId] })`; existing direct conversations are reused via dedup. Names translate on demand via `getDisplayText()` when the locale differs from the school language.

## Unread badge

`MessageMailIcon` in the platform header polls `/api/messages/unread-count` every 30 seconds and shows a count badge.

## Notifications

Triggers and recipients:

| Trigger               | Recipients                     | Priority |
| --------------------- | ------------------------------ | -------- |
| New message           | all participants except sender | normal   |
| `@mention` in message | mentioned users                | high     |
| Announcement          | all school members             | high     |
| Conversation invite   | invitee                        | normal   |

Dispatched via `notification-helpers.ts` (in-app + email + SMS).

## Search

`fullTextSearchMessages({ conversationId, query, limit, offset })` runs PostgreSQL `tsvector`/`tsquery` ranked search within a conversation. `globalSearch({ query, conversationTypes, limit })` searches across every conversation the user has access to.

## Schema

10 Prisma models: `Conversation`, `ConversationParticipant`, `Message`, `MessageAttachment`, `MessageReaction`, `MessageReadReceipt`, `TypingIndicator`, `MessageDraft`, `PinnedMessage`, `ConversationInvite`. Source of truth: `prisma/models/messages.prisma`.

Key indexes:

- `Conversation` — `[schoolId, type, lastMessageAt desc]`, `[classId]`, `[departmentId]`, `[directParticipant1Id, directParticipant2Id]`.
- `Message` — `[conversationId, createdAt desc]`, `[senderId]`, `[replyToId]`, `[conversationId, isSystem]`.
- `ConversationParticipant` — unique on `[conversationId, userId]`.

Enums: `ConversationType` (direct / group / class / department / announcement), `MessageStatus` (sending / sent / delivered / read / failed), `ParticipantRole` (owner / admin / member / read_only / guest).
