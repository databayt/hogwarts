## Overview

The Messages module provides a comprehensive in-platform messaging system for school communication. It supports 5 conversation types (direct, group, class, department, announcement), file attachments with drag-and-drop upload, emoji reactions, read receipts, message search, and full RBAC enforcement across all 8 user roles. Messages are automatically dual-delivered to WhatsApp when the school has an active WhatsApp session, with bidirectional bridging for incoming replies.

## Conversation Types

### Five Conversation Types

| Type           | Purpose                   | Auto-Participants           | Key Fields                        |
| -------------- | ------------------------- | --------------------------- | --------------------------------- |
| `direct`       | 1-to-1 private messaging  | Both users                  | `directParticipant1Id`, `2Id`     |
| `group`        | Multi-user group chat     | Creator as owner            | `title`, `description`, `avatar`  |
| `class`        | Class-bound communication | Teacher + enrolled students | `classId`                         |
| `department`   | Department-wide channel   | Department members          | `departmentId`                    |
| `announcement` | Admin/Developer broadcast | All school members          | `title`, read-only for non-admins |

### Direct Conversation Dedup

Direct conversations use `directParticipant1Id` and `directParticipant2Id` fields for quick lookup. The `getOrCreateDirectConversation()` query checks both orderings before creating a new conversation, preventing duplicates.

### System Message Templates

Automatic system messages are generated for lifecycle events:

| Template                | Trigger                     |
| ----------------------- | --------------------------- |
| `participant_joined`    | User joins conversation     |
| `participant_left`      | User leaves conversation    |
| `participant_added`     | User added by admin/owner   |
| `participant_removed`   | User removed by admin/owner |
| `conversation_created`  | New conversation created    |
| `conversation_renamed`  | Conversation title changed  |
| `conversation_archived` | Conversation archived       |
| `message_pinned`        | Message pinned              |
| `message_unpinned`      | Message unpinned            |

## Role-Based Access Control

All messaging server actions enforce RBAC via `authorization.ts`. Access is controlled at two levels: **platform role** (what conversation types a user can access) and **participant role** (what actions a user can take within a conversation).

### Platform Role Permissions

| Role       | Direct |   Group    |  Class  | Department | Announcement |
| ---------- | :----: | :--------: | :-----: | :--------: | :----------: |
| DEVELOPER  |   Y    |     Y      |    Y    |     Y      |      Y       |
| ADMIN      |   Y    |     Y      |    Y    |     Y      |      Y       |
| TEACHER    |   Y    |     Y      | Y (own) |     Y      |      -       |
| ACCOUNTANT |   Y    |     Y      |    -    |     -      |      -       |
| STAFF      |   Y    |     Y      |    -    |     -      |      -       |
| STUDENT    |   Y    | Y (create) |    -    |     -      |      -       |
| GUARDIAN   |   Y    |     -      |    -    |     -      |      -       |

| ---------------- | :--: | :------: | :--------: | :---------: | :------------: | :-: | :----: |
| owner            |  Y   |    Y     |     Y      |      Y      |       Y        |  Y  |   Y    |

### Teacher Scoping

Teachers can only access `class` conversations for classes they teach (via `Class.teacherId` + `ClassTeacher` bridge table). They cannot create or view class conversations for other teachers' classes.

## Message Lifecycle

### Message Status Flow

```
SENDING
   │
   ▼
  SENT ──────────────────► FAILED
   │                     (network/validation error)
   ▼
DELIVERED
   │
   ▼
  READ
```

### Statuses

| Status      | Description                                    |
| ----------- | ---------------------------------------------- |
| `sending`   | Client-side optimistic state                   |
| `sent`      | Server received and persisted                  |
| `delivered` | Delivered to recipient's client                |
| `read`      | Recipient has viewed the message               |
| `failed`    | Send failed (validation, network, permissions) |

### Content Types

| Content Type | Description                   | Rendering                    |
| ------------ | ----------------------------- | ---------------------------- |
| `text`       | Plain text message            | Text with link detection     |
| `image`      | Image attachment              | Inline preview with lightbox |
| `file`       | File attachment               | File card with download link |
| `system`     | Auto-generated system message | Centered, muted styling      |

### Soft Delete

Messages are soft-deleted via `isDeleted: true` and `deletedAt` timestamp. Deleted messages show a "This message was deleted" placeholder in the UI. Only the message sender (or conversation owner/admin) can delete messages.

### Message Editing

Users can edit their own messages. Edited messages display an "edited" indicator with the `editedAt` timestamp. The `isEdited` flag is set to `true` on update.

## WhatsApp Integration

Messages sent in-app are automatically dual-delivered to WhatsApp when the school has an active WhatsApp session. The integration is bidirectional -- incoming WhatsApp replies are bridged back into conversations via webhook.

### How It Works

```
┌──────────────────────────────────────────────────────────────────────┐
│                    WHATSAPP DUAL-DELIVERY FLOW                       │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  1. createConversation()                                             │
│     └─ Checks WhatsAppSession.status === "connected"                 │
│     └─ Auto-enables whatsappEnabled on conversation                  │
│     └─ Populates participant phone numbers (non-blocking)            │
│                                                                      │
│  2. sendMessage()                                                    │
│     └─ Persists message in DB                                        │
│     └─ If conversation.whatsappEnabled:                              │
│        └─ dispatchMessageToWhatsApp() (non-blocking)                 │
│        └─ Resolves phone: Guardian → Teacher → StaffMember           │
│        └─ Sends via Evolution API                                    │
│        └─ Stores whatsappMessageId + whatsappStatus on Message       │
│                                                                      │
│  3. Incoming WhatsApp reply                                          │
│     └─ Webhook: /api/webhooks/whatsapp/route.ts                      │
│     └─ Matches sender phone → conversation participant               │
│     └─ Creates Message in conversation                               │
│                                                                      │
│  4. Status updates (sent/delivered/read/failed)                      │
│     └─ Webhook receives delivery receipts                            │
│     └─ Updates Message.whatsappStatus                                │
│     └─ Message bubble shows status icon                              │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### Phone Resolution

Phone numbers are resolved from domain models in priority order:

1. **Guardian** -- `guardian.guardianPhoneNumbers` (primary)
2. **Teacher** -- `teacher.teacherPhoneNumbers` (primary)
3. **StaffMember** -- `staffMember.staffPhoneNumbers` (primary)

Users without a domain model record or phone number send in-app only (no WhatsApp dispatch).

## Contacts Sidebar

The contacts panel provides a WhatsApp-style interface for starting conversations. Contacts are grouped by role and clicking a contact immediately creates (or reuses) a direct conversation.

### How It Works

1. User clicks contacts icon in sidebar
2. Contacts loaded from `/api/contacts` with role-based grouping
3. Contact names translated on-demand via `getDisplayText()` when locale differs from school language
4. Click a contact → `createConversation({ type: "direct", participantIds: [contactId] })`
5. If a 1:1 conversation already exists, it is reused (dedup via `directParticipant1Id`/`directParticipant2Id`)

### Contact Groups

Contacts are grouped by platform role: Administrators, Teachers, Students, Guardians, Staff, Accountants. Each group is collapsible and searchable.

## Search

### Full-Text Search

The `fullTextSearchMessages()` query uses PostgreSQL's `tsvector` and `tsquery` for ranked full-text search within a conversation:

```typescript
fullTextSearchMessages(input: {
  conversationId: string
  query: string
  limit?: number
  offset?: number
})
// Returns: messages ranked by relevance (ts_rank)
```

### Global Search

The `globalSearch()` query searches across all conversations the user has access to:

```typescript
globalSearch(input: {
  query: string
  conversationTypes?: ConversationType[]
  limit?: number
})
// Returns: messages with conversation metadata
```

model Conversation {

  isArchived           Boolean            @default(false)
  lastMessageAt        DateTime?

  @@index([schoolId, type, lastMessageAt(sort: Desc)])
  @@index([classId])
  @@index([departmentId])
  @@index([directParticipant1Id, directParticipant2Id])

model ConversationParticipant {

  conversationId String
  userId         String
  role           ParticipantRole @default(member)

model Message {
  id          String      @id @default(cuid())

  whatsappMessageId String?                  // Evolution API message ID
  whatsappStatus    String?                  // sent/delivered/read/failed

  @@index([conversationId, createdAt(sort: Desc)])
  @@index([senderId])
  @@index([replyToId])
  @@index([conversationId, isSystem])

## Best Practices

### For Administrators

1. Use **announcement** conversations for school-wide broadcasts
2. Create **department** conversations for staff coordination
3. Monitor audit logs for messaging policy compliance
4. Archive inactive conversations to keep the sidebar clean
5. Assign appropriate participant roles (owner/admin/member/read_only)

### For Teachers

1. Use **class** conversations for class-specific communication
2. Pin important messages for student reference
3. Use **group** conversations for project-based collaboration
4. Respond to parent direct messages promptly
5. Use mentions to notify specific participants

### For Students

1. Use **direct** messages for private teacher communication
2. Create **group** conversations for study groups
3. Check pinned messages for important class updates
4. Mute conversations during study time using the mute toggle

### For Parents/Guardians

1. Use **direct** messages to communicate with teachers
2. Enable notifications to stay informed of school updates
3. Check announcement conversations for school-wide information
