The messages module powers in-platform communication: 5 conversation types, file attachments, emoji reactions, read receipts, full-text search, full RBAC across 8 roles. Messages dual-deliver to WhatsApp when the school has an active session, with bidirectional bridging for incoming replies.

> **Source directory is `messaging/`, not `messages/`.** All component code lives in `src/components/school-dashboard/messaging/` (~13k LOC, 80 files). The route, however, is `/messages`.

## Routes

The messaging UI lives in its own full-screen route group, `(school-messaging)`, with **no** dashboard header/sidebar/footer.

| Route                      | Purpose                              |
| -------------------------- | ------------------------------------ |
| `/messages`                | Sidebar + chat interface             |
| `/messages?conversation=X` | Deep link to a specific conversation |

File-system path: `src/app/[lang]/s/[subdomain]/(school-messaging)/messages/page.tsx`.

## Conversation types

| Type           | Purpose                      | Auto-participants           | Key fields                        |
| -------------- | ---------------------------- | --------------------------- | --------------------------------- |
| `direct`       | 1-to-1 private               | both users                  | `directParticipant1Id`, `2Id`     |
| `group`        | Multi-user group             | creator as owner            | `title`, `description`, `avatar`  |
| `class`        | Class-bound (teacher-scoped) | teacher + enrolled students | `classId`                         |
| `department`   | Department-wide channel      | department members          | `departmentId`                    |
| `announcement` | Admin / DEVELOPER broadcast  | all school members          | `title`, read-only for non-admins |

> **Creatable today: only `direct` and `group`.** The create dialog (`new-conversation-dialog.tsx`) exposes just those two tabs. `class`, `department`, and `announcement` exist in the `ConversationType` enum and the RBAC matrix below, but have **no creation entry point yet** — treat them as schema-level scaffolding, not shipped features.

Direct conversations dedup on `directParticipant1Id` / `directParticipant2Id` — `getOrCreateDirectConversation()` checks both orderings. (Dedup is enforced in application logic only; there is no DB unique constraint, so concurrent creates can still race a duplicate.)

System messages auto-generate for lifecycle events: `participant_joined`, `participant_left`, `participant_added`, `participant_removed`, `conversation_created`, `conversation_renamed`, `conversation_archived`, `message_pinned`, `message_unpinned`.

## Message lifecycle

| Status      | Meaning                                        |
| ----------- | ---------------------------------------------- |
| `sending`   | Client-side optimistic                         |
| `sent`      | Server received and persisted                  |
| `delivered` | Delivered to recipient's client                |
| `read`      | Recipient has viewed                           |
| `failed`    | Send failed (validation, network, permissions) |

| Content type | Rendering                      |
| ------------ | ------------------------------ |
| `text`       | Plain text with link detection |
| `image`      | Inline preview with lightbox   |
| `file`       | File card with download        |
| `system`     | Centred, muted styling         |

Messages soft-delete via `isDeleted: true` + `deletedAt` ("This message was deleted."). Only sender (or owner / admin) can delete. Edits set `isEdited: true` + `editedAt`; UI shows "edited".

## File attachments

Drag-and-drop with previews and progress bars. S3 / Vercel Blob storage. Path: `messaging/{schoolId}/{conversationId}/{filename}`.

## Real-time delivery

Live updates use **Socket.IO** via a standalone server in `socket-server/` — 15 events (`message:new/updated/deleted/read/reaction`, `conversation:updated/new`, `conversation:participant_added/removed`, `typing:start/stop`). Presence is Redis-backed (green dots on cards, last-seen in header). Server→client pushes go through `POST /api/emit` and `/api/emit-to-users`, protected by an `x-emit-secret` header.

> **Ops status:** the Socket.IO server is **not yet deployed** — tracked in [#262](https://github.com/databayt/hogwarts/issues/262) (migration to Oracle Cloud). Until it ships, the client falls back to **polling**: 15 s for the conversation list and 10 s for the active conversation (`messaging-client.tsx`). Realtime works end-to-end once the server is live and `NEXT_PUBLIC_SOCKET_URL` / `SOCKET_SECRET` / `EMIT_SECRET` are set on Vercel (currently `NEXT_PUBLIC_SOCKET_URL` still points at `localhost`).

## WhatsApp integration

When the school has an active WhatsApp session, in-app messages auto-dispatch via Evolution API. Bidirectional — incoming replies bridge back via webhook.

Phone resolution: `Guardian.guardianPhoneNumbers` → `Teacher.teacherPhoneNumbers` → `StaffMember.staffPhoneNumbers`. Users without a domain record send in-app only.

| Element     | Location          | Purpose                                                     |
| ----------- | ----------------- | ----------------------------------------------------------- |
| W toggle    | Chat header       | Enable / disable WhatsApp per conversation                  |
| W badge     | Conversation card | Marks WhatsApp-enabled conversations                        |
| Status icon | Message bubble    | WhatsApp delivery status (sent / delivered / read / failed) |

Rate limits: 1 msg / sec / school, 500 direct / day / school. Failed dispatches retry with exponential backoff (max 5 attempts) via `cron/process-whatsapp-notifications` (every 5 minutes).

> **Ops caveats:** the retry cron **refuses to run in production until `CRON_SECRET` is set**, and the webhook fails closed until `WHATSAPP_WEBHOOK_SECRET` is set (the per-instance webhook URL carries `?secret=` automatically as of 2026-06-12 — schools connected before the secret existed must reconnect once). Group WhatsApp is fully supported: the scalar `whatsapp*` columns on `Message` mirror **1:1** state, while groups write one `MessageWhatsappDelivery` row per recipient — retried independently with backoff, and advanced to delivered/read by webhook status events. Phones are normalized to a single digits-only form (strips `+`/`00`) across outbound, storage, and inbound sender lookup.

Key files: `whatsapp-bridge.ts`, `src/lib/whatsapp/evolution-client.ts`, `src/app/api/webhooks/whatsapp/route.ts`, `src/app/api/cron/process-whatsapp-notifications/route.ts`, `src/components/school-dashboard/whatsapp/`.

## Contacts sidebar

WhatsApp-style contact list grouped by role (`contacts/`). Click → `createConversation({ type: "direct", participantIds: [contactId] })`; existing direct conversations are reused via dedup (app check + a DB unique index on the direct pair, so concurrent creates can't race a duplicate). Names translate on demand via one batched `getNames()` call. The WhatsApp connect dialog (QR) is also embedded here, alongside the per-chat `W` toggle.

## Unread badge

`MessageMailIcon` in the platform header fetches `/api/messages/unread-count` on mount and on tab focus, and increments live via the Socket.IO `message:new` event.

## Notifications

| Trigger           | Recipients                     | Priority |
| ----------------- | ------------------------------ | -------- |
| New message       | all participants except sender | normal   |
| `@mention`        | mentioned users                | high     |
| Participant added | the added user                 | normal   |

Dispatched via `notification-helpers.ts` (in-app + email; mentions and participant-added text branch on the school's `preferredLanguage`). Notifications can also fan out to WhatsApp when the school has it enabled.

## Search

`fullTextSearchMessages({ conversationId, query, limit, offset })` — PostgreSQL `tsvector` / `tsquery` ranked search, backed by a GIN expression index. The last term gets `:*` prefix matching (search-as-you-type), Arabic text is searchable, and `/api/messages/search` uses this path (not ILIKE).

`globalSearch({ query, conversationTypes, limit })` — across every conversation the user can access.

## Schema

11 Prisma models in `prisma/models/messages.prisma`: `Conversation`, `ConversationParticipant`, `Message`, `MessageAttachment`, `MessageReaction`, `MessageReadReceipt`, `TypingIndicator`, `MessageDraft`, `PinnedMessage`, `ConversationInvite`, `StarredMessage`. WhatsApp models (`WhatsAppSession`, `WhatsAppMessage`, `WhatsAppGroup`, …) live separately in `prisma/models/whatsapp.prisma`.

`Message` carries no `lang` field — message bodies are user-to-user content, translated on demand at render time (via `getText()` for contact names), not stored bilingually.

Indexes:

- `Conversation` — `[schoolId, type, lastMessageAt desc]`, `[schoolId, classId]`, `[schoolId, departmentId]`, `[directParticipant1Id, directParticipant2Id, type]`
- `Message` — `[conversationId, createdAt desc]`, `[senderId, createdAt]`, `[replyToId]`, `[conversationId, isSystem]`, `[whatsappMessageId]`, `[forwardedFromId]`
- `ConversationParticipant` — `@@unique([conversationId, userId])`, `[userId, isActive]`, `[conversationId, role]`
- `TypingIndicator` — `[schoolId]`, `[conversationId, expiresAt]`

Enums: `ConversationType` (5), `MessageStatus` (5), `ParticipantRole` (5).
