title: Translation
description: Two-tier i18n — dictionary JSON for static UI, batched localize() for dynamic content. Coverage gaps and adoption patterns.
---

Translation runs on two tiers: dictionary JSON files for static UI (parity-enforced across `ar`/`en` by a CI test) and the batched `localize()` engine for dynamic database content via a three-tier cache (in-memory LRU → `Translation` table → Google Translate). Drift can no longer ship: en/ar key parity, untranslated placeholders, hardcoded-string counts, and RTL physical classes are all gated in `pnpm test` and CI. The remaining work is the hardcoded-string sweep tracked by the ratchet baselines — run `npx tsx scripts/i18n-hardcoded-ratchet.ts --by-dir` for the live count.

## Architecture

| Tier          | What                                          | How                                                                                    |
| ------------- | --------------------------------------------- | -------------------------------------------------------------------------------------- |
| 1 — Static UI | Sidebar, table headers, status badges, labels | Dictionary JSON → `getDictionary()` → `dictionary.key`                                 |
| 2 — Dynamic   | Announcements, subjects, departments, events  | Single-language storage → batched `localize()`/`getLabels()` → LRU → DB cache → Google |

## Infrastructure files

| File                                                   | Purpose                                             |
| ------------------------------------------------------ | --------------------------------------------------- |
| `src/components/translation/localize.ts`               | Batched `localize()`/`localizeOne()` — preferred    |
| `src/components/translation/registry.ts`               | TRANSLATABLE map (model → fields)                   |
| `src/components/translation/prewarm.ts`                | Cache-fill on write via `after()`                   |
| `src/components/translation/person.ts`                 | `getNames()`/`getLabels()` — batched names/labels   |
| `src/components/translation/display.ts`                | `getText()`, `getFields()` — single values only     |
| `src/components/translation/actions.ts`                | `translate()` core (LRU → DB → Google)              |
| `src/components/translation/google.ts`                 | Google client: timeout, chunking, retry policy      |
| `src/components/translation/util.ts`                   | `withLang`, `detectScript`                          |
| `src/components/internationalization/helpers/index.ts` | `ValidationHelper`, `ToastHelper`, `ErrorHelper`    |
| `src/components/internationalization/namespaces.ts`    | Single namespace registry (server + client)         |
| `src/components/internationalization/dictionaries.ts`  | Route-scoped dictionary loaders (`cache()`-wrapped) |

## Coverage gaps

| Gap                                       | Status                              |
| ----------------------------------------- | ----------------------------------- |
| Button text (`Save`, `Cancel`, `Delete`…) | Keys exist in `dictionary.common.*` |
| Validation (Zod messages)                 | Needs schema-factory refactor       |
| Config / select options                   | Needs option factories              |
| Form labels and placeholders              | Partial adoption                    |
| Server action errors                      | Needs error-code pattern            |
| Wizard tab / group labels                 | Pending                             |
| Empty states                              | Pending                             |
| Toast messages                            | Needs `ToastHelper` plumbing        |

`Cards` and `Lab` sidebar items also lack dictionary keys. Run `/i18n-check` for the current per-area count.

## Priority order

1. Button text — keys already exist in `dictionary.common.*`.
2. Wizard group labels — high visibility.
3. Form labels — largest UX impact.
4. Select options — needs config factory.
5. Validation — needs schema factory refactor.
6. Server action errors — needs error code pattern.
7. Empty states.
8. Toast messages — needs dictionary plumbing.
9. `getText` expansion to remaining content files.

## Form pattern

Pass dictionary slices into client forms.

```tsx
// content.tsx
const dictionary = await getDictionary(lang)
return <StudentForm dictionary={dictionary.school.students} />

// form.tsx
<FormLabel>{dictionary.form.name}</FormLabel>
<Input placeholder={dictionary.form.namePlaceholder} {...field} />
```

## Select pattern

Export option factories, not static arrays.

```ts
export const getGenderOptions = (d: Dictionary["school"]) => [
  { value: "male", label: d.students.gender.male },
  { value: "female", label: d.students.gender.female },
]
```

## Validation pattern

Schemas become factories that take a `ValidationHelper`. Already shipping in onboarding and auth.

```ts
import { getValidationMessages } from "@/components/internationalization/helpers"

export const createStudentSchema = (
  v: ReturnType<typeof getValidationMessages>
) =>
  z.object({
    name: z.string().min(1, v.required()),
    email: z.string().email(v.email()),
    phone: z.string().min(10, v.minLength(10)),
  })
```

Blocker: validation files across the dashboard need to convert from module-level schemas to factories.

## Toast pattern

```ts
import { getToastMessages } from "@/components/internationalization/helpers"

const t = useMemo(() => getToastMessages(dictionary), [dictionary])

if (result.success) toast.success(t.success.student.created())
else toast.error(t.error.student.createFailed())
```

Blocker: toasts run client-side; `ToastHelper` needs dictionary in scope.

## Server action error pattern

Return error codes; translate on the client.

```ts
// actions.ts
if (!session?.user) return { success: false, errorCode: "NOT_AUTHENTICATED" }
if (!schoolId) return { success: false, errorCode: "MISSING_SCHOOL_CONTEXT" }

// form.tsx
const ERROR_MAP: Record<string, string> = {
  NOT_AUTHENTICATED: dictionary.messages.errors.auth.notAuthenticated,
  MISSING_SCHOOL_CONTEXT:
    dictionary.messages.errors.tenant.missingSchoolContext,
}
toast.error(
  ERROR_MAP[result.errorCode] ?? dictionary.messages.errors.server.internalError
)
```

## Per-area status

All 21 areas (Dashboard, Students, Teachers, Classes, Subjects, Announcements, Events, Assignments, Parents, Grades, Classrooms, Exams, Finance, Attendance, Timetable, Profile, Settings, Notifications, Admission, School Config, Reports) use the dictionary for table headers and sidebar but are partial on `getText`, forms, selects, validation, toasts, and actions. Students/Teachers/Classes/Subjects/Announcements/Parents/Classrooms/Exams/Attendance/Profile use `getText` for some surfaces; the rest are display-text-pending.

Wizard routes (`/students/add`, `/teachers/add`, `/exams/manage`, `/exams/qbank`, `/finance/invoice`) have hardcoded forms, selects, validation, toasts, and action errors.
