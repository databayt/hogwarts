Translation runs on two tiers: dictionary JSON files for static UI (synced across `ar.json`/`en.json`) and `getDisplayText()` for dynamic database content via Google Translate + DB cache. Infrastructure is solid; adoption is partial â€” thousands of hardcoded English strings remain across hundreds of school-dashboard files. Run `/i18n-check` for the live count.

## Architecture

| Tier          | What                                          | How                                                                    |
| ------------- | --------------------------------------------- | ---------------------------------------------------------------------- |
| 1 â€” Static UI | Sidebar, table headers, status badges, labels | Dictionary JSON â†’ `getDictionary()` â†’ `dictionary.key`                 |
| 2 â€” Dynamic   | Announcements, subjects, departments, events  | Single-language storage â†’ `getDisplayText()` â†’ cached Google Translate |

## Form pattern

Pass dictionary slices into client forms.

```tsx
// content.tsx
const dictionary = await getDictionary(lang)
return <StudentForm dictionary={dictionary.school.students} />

// form.tsx
<FormLabel>{dictionary.form.name}</FormLabel>
<Input placeholder={dictionary.form.namePlaceholder} {...field} />
```

## Select pattern

Export option factories, not static arrays.

```ts
export const getGenderOptions = (d: Dictionary["school"]) => [
  { value: "male", label: d.students.gender.male },
  { value: "female", label: d.students.gender.female },
]
```

## Validation pattern

Schemas become factories that take a `ValidationHelper`. Already shipping in onboarding and auth.

```ts
import { getValidationMessages } from "@/components/internationalization/helpers"

export const createStudentSchema = (
  v: ReturnType<typeof getValidationMessages>
) =>
  z.object({
    name: z.string().min(1, v.required()),
    email: z.string().email(v.email()),
    phone: z.string().min(10, v.minLength(10)),
  })
```

Blocker: validation files across the dashboard need to convert from module-level schemas to factories.

## Toast pattern

```ts
import { getToastMessages } from "@/components/internationalization/helpers"

const t = useMemo(() => getToastMessages(dictionary), [dictionary])

if (result.success) toast.success(t.success.student.created())
else toast.error(t.error.student.createFailed())
```

Blocker: toasts run client-side; `ToastHelper` needs dictionary in scope.

## Server action error pattern

Return error codes; translate on the client.

```ts
// actions.ts
if (!session?.user) return { success: false, errorCode: "NOT_AUTHENTICATED" }
if (!schoolId) return { success: false, errorCode: "MISSING_SCHOOL_CONTEXT" }

// form.tsx
const ERROR_MAP: Record<string, string> = {
  NOT_AUTHENTICATED: dictionary.messages.errors.auth.notAuthenticated,
  MISSING_SCHOOL_CONTEXT:
    dictionary.messages.errors.tenant.missingSchoolContext,
}
toast.error(
  ERROR_MAP[result.errorCode] ?? dictionary.messages.errors.server.internalError
)
```

## Per-area status

All 21 areas (Dashboard, Students, Teachers, Classes, Subjects, Announcements, Events, Assignments, Parents, Grades, Classrooms, Exams, Finance, Attendance, Timetable, Profile, Settings, Notifications, Admission, School Config, Reports) use the dictionary for table headers and sidebar but are partial on `getDisplayText`, forms, selects, validation, toasts, and actions. Students/Teachers/Classes/Subjects/Announcements/Parents/Classrooms/Exams/Attendance/Profile use `getDisplayText` for some surfaces; the rest are display-text-pending.

Wizard routes (`/students/add`, `/teachers/add`, `/exams/manage`, `/exams/qbank`, `/finance/invoice`) have hardcoded forms, selects, validation, toasts, and action errors.
