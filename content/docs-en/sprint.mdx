The [MVP](/docs/mvp) doc records what is shipped; the [Roadmap](/docs/roadmap) records the quarter direction. This doc is narrower and time-boxed: the **single sprint that ends with launch**. Anchor date: 2026-06-20, HEAD `febb7da3b`.

The thesis is simple. After the June deploys ‚Äî transportation Phase 2‚Äì5, the conference notification/attendance integration, the public features + community hubs, exam-and-grade automation, the stream/LMS optimize pass, the saas-dashboard de-fabrication, timetable production-readiness, and demo-seed auto-provision ‚Äî almost every epic is **code-complete**. What stands between here and launch is no longer feature-building. It is four things: making the last few correctness fixes, confirming the database, turning on the ops and credentials, and running a QA pass that actually proves it. This sprint does exactly that, for all three launch audiences at once: the **King Fahad** pilot (Sudan, Sept 2026), the **public SaaS** at `ed.databayt.org`, and the **Aldar** enterprise pilot (UAE).

For requirements see [PRD](/docs/prd). For the pilot context see [Pilot](/docs/pilot).

## The 4-week sprint

Five tracks run in parallel across all four weeks: **A** Code, **B** Data, **C** Ops/Infra, **D** QA/Proof, **E** Polish. Each week has a theme and an exit gate.

| Week | Theme                | Exit gate                                                                                      |
| ---- | -------------------- | ---------------------------------------------------------------------------------------------- |
| 1    | Reachable & Truthful | Every shipped feature has a URL, CI runs e2e, prod has error visibility, DB confirmed          |
| 2    | Money & Messaging    | Every financial event posts a balanced ledger entry; real-time + WhatsApp live for all tenants |
| 3    | Conference & Proof   | LiveKit live (network-test scheduled), e2e green in CI, Arabic RTL clean                       |
| 4    | Harden & Launch      | Launch gate green for all three audiences                                                      |

**Week 1 ‚Äî Reachable & Truthful**

- A ¬∑ Fix the Playwright CI command (`tests` ‚Üí `test`) so e2e runs ¬∑ make `STRIPE_API_KEY` optional in `/api/health` ¬∑ add the `manage/` + `paper/` exam index pages and confirm sidebar surfacing.
- B ¬∑ Audit Neon prod vs Prisma; apply any additive drift on a Neon branch, then promote.
- C ¬∑ Set Sentry + Upstash env in Vercel (immediate observability + correct rate limiting).

**Week 2 ‚Äî Money & Messaging**

- A ¬∑ ‚úÖ Wired `postFeeAssignment` (assignFee + bulkAssignFees) ‚Äî the accrual fix that stops the receivable going negative. Investigation found the other three blocked _beyond_ wiring (no invoice/expense payment-recording flow; salary rule unbalanced) and **fixed** a pre-existing **`toCents` √ó100 ledger-inflation P0** ‚Äî rules now store whole units, `posting-rules.test.ts` green (15/15). The debit=credit invariant test already existed, and the `finance/content.tsx` + `salary/content.tsx` 1/100 display bug is fixed. ‚úÖ payroll progressive tax (replaces flat 15% in payroll + salary, unit-tested). The √∑100 ledger migration is a **verified no-op safety net** ‚Äî a read-only prod check found zero inflated rows (all 200 entries are seeded whole-units), so the toCents fix deploys safely alone; the script is now self-verifying/fees-scoped, run post-deploy only as a sweep. ‚úÖ invoice PDF + branded RTL email (logo into the PDF; email rebuilt on react-email ‚Äî RTL, branding, itemized, signature). ‚úÖ invoice + expense payment flows (`markInvoicePaid` + `markExpensePaid`). ‚úÖ salary poster ‚Äî rebalanced the salary posting rule + wired `postSalaryPayment` in `processPayments` ‚Üí **6 of 6 posters wired; the double-entry ledger is fully wired for every finance money event.** **Week-2 finance code is complete** ‚Äî and an adversarial multi-agent review of the whole finance diff found + fixed **12 money-path defects** (multi-tenant `schoolId` on updates, concurrent double-post races via conditional updates + a `JournalEntry` `@@unique` idempotency constraint, salary tax base, invoice partial-payment, an `expenses/approve` permission gate, 4 dashboards' √∑100 display, RTL discount sign). ‚úÖ the expense list now surfaces Approve/Reject + Mark-paid (`ExpenseRowActions`), closing the expense flow end-to-end. What remains is ops (Track C: Oracle VM, Tap/FCM env, ADEK) and **two additive deploy-pending migrations** (the toCents √∑100 sweep ‚Äî a verified no-op ‚Äî and the JournalEntry `@@unique`).
- C ¬∑ Stand up Oracle VM + Evolution API (6 env vars; per-school QR for King Fahad + Albayan) ¬∑ set Tap + FCM env ¬∑ **file the ADEK inquiry now** (longest external lead time).

**Week 3 ‚Äî Conference & Proof**

- C ¬∑ LiveKit 6-gate provisioning; schedule gate-6 in-school network test to a King Fahad / Aldar site visit.
- D ¬∑ Real e2e pass now that CI runs: triage the `test.skip` backlog, un-skip what runs headless, add a named section-grid attendance case, give the assertion-less specs real `expect()`s ¬∑ i18n RTL Arabic pass on the swept surfaces + parent-portal dictionary-key migration.

**Week 4 ‚Äî Harden & Launch**

- E ¬∑ Drain the highest-value SaaS-dashboard backlog: ‚úÖ invoice state-machine guard + atomic CAS (`3be3506ed` + 2026-06-21 ‚Äî no voiding a paid invoice, no concurrent double-transition; buttons gated) ¬∑ ‚úÖ exams RBAC + calculator test coverage + AI generation/grading cost cap (2026-06-21) ¬∑ ‚úÖ catalog TOCTOU closed (proposal reject + 4 exam-approval actions now atomic; chapter/lesson count denormalization wrapped in transactions, 2026-06-21). A **two-pass adversarial review** (find ‚Üí verify, then re-verify the fixes) confirmed + closed 8 further correctness defects across finance/catalog/exams/billing incl. a P0 ledger-reversal no-op. Remaining: i18n sweep ¬∑ the Kanban demo route (still linked from analytics nav + command menu ‚Äî full removal is a product call, not dead code).
- D ¬∑ Per-pilot smoke on King Fahad + Albayan ¬∑ load + rate-limit verification (now Redis-backed) ¬∑ final `secure` pass ¬∑ wire the hardcoded-string ratchet into the PR gate.
- Run the launch gate.
