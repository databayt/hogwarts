---
title: Client
description: Canonical patterns for writing client.tsx boundary components in feature directories.
---

`client.tsx` is the single `"use client"` boundary per feature directory. It sits between the server component (`content.tsx`) and interactive leaf components (`form.tsx`, `table.tsx`, `detail.tsx`), keeping `content.tsx` as a pure server component.

```
page.tsx → content.tsx (server) → client.tsx ("use client") → form.tsx, table.tsx
```

`content.tsx` fetches data on the server and passes it to `client.tsx`. `client.tsx` manages interactive state.

## When to use it

Not every feature needs `client.tsx`. Decision tree:

| Scenario                                                   | Use `client.tsx`?                      |
| ---------------------------------------------------------- | -------------------------------------- |
| `content.tsx` only renders a table with server data        | No — pass data directly to `table.tsx` |
| `content.tsx` needs modal state for create / edit          | Yes                                    |
| `content.tsx` needs view toggles, selections, or filters   | Yes                                    |
| `content.tsx` is pure server composition (dashboard stats) | No                                     |
| Both server data and interactivity                         | Yes — split them                       |

## Categories

| Category              | Share | What it manages                                             |
| --------------------- | ----: | ----------------------------------------------------------- |
| CRUD feature client   |   50% | Modal state for create / edit / view; composes table + form |
| Interactive dashboard |   30% | Tab state, view toggles, selection, drill-down              |
| Wizard / flow         |   20% | Step state, validation context, step navigation             |

## Five rules

1. **`"use client"` at the top.** This IS the boundary.
2. **Receive data as props.** Don't fetch in `client.tsx`. The server `content.tsx` fetches and passes.
3. **Stay under ~150 lines.** Beyond that, extract sub-components.
4. **No DB imports.** Never `import { db }` in a client component. Fetch via server actions.
5. **Forward props to leaves.** Pass `dictionary`, `lang`, and feature-specific data through to `form.tsx` / `table.tsx`.

## Templates

### CRUD feature

```tsx
"use client"

import { useState } from "react"

import { Button } from "@/components/ui/button"
import type { Dictionary } from "@/components/internationalization/types"

import { TeacherForm } from "./form"
import { TeachersTable } from "./table"
import type { TeacherListItem } from "./types"

interface Props {
  data: TeacherListItem[]
  dictionary: Dictionary["school"]["teachers"]
  lang: Locale
}

export function TeachersClient({ data, dictionary, lang }: Props) {
  const [showForm, setShowForm] = useState(false)
  return (
    <>
      <Button onClick={() => setShowForm(true)}>{dictionary.add}</Button>
      <TeachersTable data={data} dictionary={dictionary} lang={lang} />
      {showForm && (
        <TeacherForm
          dictionary={dictionary}
          onClose={() => setShowForm(false)}
        />
      )}
    </>
  )
}
```

The matching `content.tsx` is server-side and trivial:

```tsx
import { TeachersClient } from "./client"
import { getTeacherList } from "./queries"

export default async function TeachersContent({ dictionary, lang }: Props) {
  const data = await getTeacherList()
  return <TeachersClient data={data} dictionary={dictionary} lang={lang} />
}
```

### Interactive dashboard

```tsx
"use client"

import { useState } from "react"

import { Tabs, TabsContent, TabsList, TabsTrigger } from "@/components/ui/tabs"

export function DashboardClient({ stats, recent, dictionary }: Props) {
  const [tab, setTab] = useState<"overview" | "recent">("overview")
  return (
    <Tabs value={tab} onValueChange={(v) => setTab(v as typeof tab)}>
      <TabsList>
        <TabsTrigger value="overview">{dictionary.overview}</TabsTrigger>
        <TabsTrigger value="recent">{dictionary.recent}</TabsTrigger>
      </TabsList>
      <TabsContent value="overview">{/* stats */}</TabsContent>
      <TabsContent value="recent">{/* recent */}</TabsContent>
    </Tabs>
  )
}
```

### Wizard / flow

```tsx
"use client"

import { useState } from "react"

import { WizardLayout } from "@/components/wizard/layout"

export function ApplicationClient({ initialData, dictionary, lang }: Props) {
  const [step, setStep] = useState(0)
  return (
    <WizardLayout step={step} onStepChange={setStep}>
      {/* step components */}
    </WizardLayout>
  )
}
```

## Anti-patterns

- **Putting `"use client"` on `content.tsx`** — that's exactly what `client.tsx` exists to avoid. Extract the interactive bit.
- **Fetching inside `client.tsx`** — server fetches at the boundary, client receives.
- **Bundling unrelated client logic** — one boundary per feature, not one boundary per page.
- **Large `client.tsx`** — extract to sub-components (`<TeacherFormDialog />`, `<TeacherFilters />`) when over 150 lines.
- **Importing `db`** — TS will block it; the rule is about discipline before the error.

## Relationship to siblings

| File          | Role                                                   |
| ------------- | ------------------------------------------------------ |
| `page.tsx`    | Auth + tenant + render `<XxxContent />`. Thin.         |
| `content.tsx` | Server. Fetches data. Renders `<XxxClient />`.         |
| `client.tsx`  | Client. Manages state. Composes leaves.                |
| `form.tsx`    | Client leaf. RHF + Zod. Calls server action on submit. |
| `table.tsx`   | Client leaf. DataTable wrapper.                        |
| `actions.ts`  | Server actions called from `form.tsx` / `client.tsx`.  |
| `queries.ts`  | Server-side reads called from `content.tsx`.           |

## Naming

- Filename: `client.tsx`. Always.
- Function: `<Feature>Client` in PascalCase.
- Named export.

The current codebase has only a handful of `client.tsx` files; the pattern is being introduced. Target: 90%+ of `content.tsx` end up as server components by extracting client logic to `client.tsx`.
