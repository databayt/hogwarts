The exams module covers the full lifecycle: question authoring, paper printing, digital taking, AI-assisted marking, analytics, certificates, and progress tracking. 18 sub-modules, 40+ routes, ~80 component files.

## Create an exam — two modes

The front door (`/exams/new`) is a single screen offering two creation modes. Both converge on the same downstream — a `GeneratedExam` with questions → paper preview → schedule — and differ only in how the blueprint and questions are seeded.

| Mode                 | Route                      | Seeds from                                                                       | Speed   |
| -------------------- | -------------------------- | -------------------------------------------------------------------------------- | ------- |
| **Adopt a template** | `/exams/generate/catalog`  | A platform-curated, curriculum-aligned `ExamTemplate` (browse → preview → Adopt) | Fastest |
| **Generate with AI** | `/exams/qbank/ai-generate` | A topic prompt → AI-drafted questions into your bank (review before saving)      | Fast    |

The in-app "build a template from primitives" wizard was removed — a paper's **layout** is now an uploaded `.docx` (see [Document templates](#document-templates) below), while its **question distribution** lives on a `SchoolExamTemplate` blueprint seeded by adopting a catalog blueprint.

## Document templates

Schools print on their own mandated stationery, so the paper's **layout** is a `.docx` the school uploads — not something built in-app. `/exams/templates` (staff only: DEVELOPER / ADMIN / TEACHER) manages two categories, **Exam papers** and **Certificates**; report-card layouts live at `/grades/templates`. There is no standalone `/documents` page.

Author the Word file with `{{tag}}` placeholders and `{#loop}…{/loop}` sections, upload it, and the merge fields are detected automatically and shown back to you. The full vocabulary per category is listed in the upload dialog.

### Filling an exam paper

An uploaded exam-paper template is bound to real exam data through **Use template**, which offers two sources:

| Source           | What you pick                                        | What happens                                                                                              |
| ---------------- | ---------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| An existing exam | A `GeneratedExam` that already has questions         | Fills your layout directly                                                                                |
| Build one now    | A blueprint + class + paper title + date + exam type | Creates the `SchoolExam` + `GeneratedExam`, auto-selects questions from the bank per the blueprint, fills |

Before generating, the dialog shows **coverage** — every tag detected in your template, marked as either fillable or unsupported — so a mismatch is visible before you download rather than as a blank in the printed paper. Output is `.docx`; PDF conversion is deferred.

## Management

| Route              | Purpose                                         |
| ------------------ | ----------------------------------------------- |
| `/exams/new`       | Create-exam chooser (adopt · AI · from scratch) |
| `/exams/[id]`      | Details + result summary                        |
| `/exams/[id]/edit` | Modify before administration                    |

`manage/calendar.tsx` — visual schedule. `manage/conflict-display.tsx` — overlap warnings. `manage/analytics-dashboard.tsx` — school-wide stats, standards coverage, quick paper-gen, export.

The `/exams/manage` list (`manage/columns.tsx`) shows seven columns: **Title** (row anchor), **Grade** (the exam's `class.grade`, not the course-offering name), **Subject**, **Type**, **When** (date · start time · duration, merged into one), **Marks**, and **Status**. Exams are scoped to a `Class` (course offering) which carries `gradeId` and `classroomId` — there is no direct `Section` link, so the list cannot show a section without a schema change.

## Paper generation

`@react-pdf/renderer`-based system with a four-layer composable template:

1. **Atoms** (13 in `templates/atom/`) — `alignment-marker`, `answer-line`, `blank-slot`, `bubble`, `checkbox`, `divider`, `field-line`, `logo-block`, `option-label`, `points-badge`, `question-number`, `type-label`, `watermark`.
2. **Question sections** — one per question type.
3. **Page sections** — 6 slots × variants:

   | Slot         | Variants                                         |
   | ------------ | ------------------------------------------------ |
   | Header       | standard, ministry, minimal, bilingual, centered |
   | Footer       | standard, disclaimer, minimal, grading           |
   | Student info | standard, bubble-id, table, photo                |
   | Instructions | standard, compact, rules, sectioned              |
   | Answer sheet | standard, omr, grid                              |
   | Cover        | standard, toc, ministry                          |

4. **Composition** — `ComposableDocument` reads `CompositionConfig` and assembles slot variants.

```typescript
interface CompositionConfig {
  slots: Record<
    | "header"
    | "footer"
    | "studentInfo"
    | "instructions"
    | "answerSheet"
    | "cover",
    string
  >
  decorations: {
    accentBar: {
      enabled: boolean
      height?: number
      colorKey?: "accent" | "primary"
    }
    watermark: { enabled: boolean; text?: string; opacity?: number }
    frame: { enabled: boolean; outerWidth?: number; innerWidth?: number }
  }
  slotProps: Record<string, unknown>
}
```

Resolution: `blockConfig` JSON → regional preset overrides → `DEFAULT_COMPOSITION`.

`paper/composition-panel.tsx` provides the variant picker. The four legacy templates (Classic, Modern, Formal, Custom) are deprecated wrappers around `ComposableDocument`.

| Preset         | Highlights                                       |
| -------------- | ------------------------------------------------ |
| `sd-national`  | Sudan — ministry header, disclaimer, watermark   |
| `sa-national`  | Saudi Arabia — ministry, bubble-id, OMR sheets   |
| `us-standard`  | USA standard layout                              |
| `mena-private` | MENA private — bilingual header, custom branding |

Each style has a `PaperTheme` with locale-aware colour, typography, spacing, RTL/LTR direction, and number style (Arabic / Latin).

## Digital taking

`take/exam-player.tsx` is the full-screen runner: per-question nav, countdown timer, auto-save, validated submit. `take/proctor-guard.tsx` adds device fingerprint, security flag reporting (tab switch, copy / paste, screen capture), retake-penalty calculation.

```typescript
startExamSession(input: StartExamSessionInput)
autoSaveAnswers(input: AutoSaveAnswersInput)
reportSecurityFlag(input: ReportSecurityFlagInput)
submitExamAnswers(input: SubmitExamAnswersInput)
```

## Marking

`NOT_STARTED` → `AUTO_GRADED` | `AI_GRADED` | `IN_PROGRESS` → `REVIEWED` → `COMPLETED`. Override supported at any point.

```typescript
autoGradeAnswer(studentAnswerId: string)
aiGradeAnswer(studentAnswerId: string)
manualGrade(answerId: string, points: number, feedback?: string)
bulkGradeExam(examId: string, method: GradingMethod)
overrideGrade(resultId: string, newScore: number, reason: string)
```

OCR uses GPT-4 Vision for handwritten answers. AI grading uses essay rubrics from the question bank.

## Results & analytics

`/exams/result` lists exams with class average, pass rate, grade distribution. Per-exam at `/exams/result/[id]/analytics`:

- Class — average, median, mode, std dev, top / bottom performers, distribution chart, trend.
- Question — discrimination index, distractor analysis, most missed, average time per question.

Batch PDF at `/exams/result/[id]/pdf/batch` — background processing, multiple result-card templates, ZIP. CSV export for raw data. `results/cache-manager.ts` caches computed results.

## Troubleshooting

| Symptom                     | Likely cause                                                                                           |
| --------------------------- | ------------------------------------------------------------------------------------------------------ |
| Exam generation fails       | Pool too small for distribution; fix template or add questions                                         |
| AI grading errors           | `OPENAI_API_KEY` missing, rate limit, missing rubric                                                   |
| PDF paper crashes           | `@react-pdf/renderer` missing; missing logo URL; Rubik font unreachable; insufficient memory for batch |
| Digital exam won't start    | Exam status not `IN_PROGRESS`; student not enrolled                                                    |
| Proctor guard inactive      | Production HTTPS required                                                                              |
| Certificate QR doesn't scan | `QR_CODE_SECRET` missing; certificate revoked; verify route blocked                                    |

## Automation pipeline

The scoring loop is largely hands-off once questions are in the bank. Objective exams need zero manual marking; exams with subjective questions take one click.

```
1. Auto-generate from the bank  (exam-wizard-v2 → "Auto-generate from template")
   └─ autoGenerateExamQuestions() selects questions per the template distribution,
      assigns per-question points, writes GeneratedExamQuestion rows, drops the
      stale answer key. (Adopting a catalog exam does the same in one transaction.)

2. Exam opens on schedule  (/api/cron/exam-lifecycle, */15)
   └─ PLANNED → IN_PROGRESS at the scheduled start; IN_PROGRESS → COMPLETED after
      the end window (examDate + endTime + lateSubmitMinutes).

3. Students take + submit  (take route → ExamPlayer)
   └─ getExamForPlayer() loads the proctored runner, resolving the real Student.id
      and stripping isCorrect from options. submitExamSession() instant-grades a
      fully-objective paper on submit → finalizeStudentExam() → ExamResult + Result.

4. Finalize exams with subjective questions  (one click)
   └─ "Auto-mark & publish" → finalizeExamResults({ aiGradeSubjective: true,
      publish: true }): objective auto-mark + AI-grade essay/short-answer, aggregate
      MarkingResult → ExamResult (analytics, certificates) + Result (gradebook),
      notify students.

5. Report cards auto-generate at term end  (/api/cron/term-end-report-cards, daily)
   └─ generateReportCardsCore() aggregates each term's results into ReportCard +
      ReportCardGrade (dedup by examId). Drafts only — does not publish.

6. Admin reviews + publishes
   └─ publishReportCards() → notifies guardians/students; process-report-card-pdfs
      cron renders the PDF out-of-band → S3 → pdfUrl.

7. Official documents  (documents block)
   └─ Upload a .docx template (EXAM_PAPER / CERTIFICATE / REPORT_CARD) and fill it
      with real data via "Generate (my template)" (generateFromDefaultTemplate).

8. Reminders  (/api/cron/exam-reminders)
   └─ sweeps upcoming exams, dispatches advance notifications to students + teachers.
```

Key invariant: `grades/lib/gradebook.ts` is the single write path for ALL automated scoring surfaces (exams, quick assessments, stream lesson quizzes). It is not `"use server"` — import it from server actions.

Deferred: a fully **session-less** AI-finalize-at-close cron. The marking stack (`getOrCreateAnswerKey`, `batchAutoGradeWithKey`, `batchAIGrade`) derives `schoolId` from `auth()`, so a timer can't drive it until cron-callable cores are extracted — until then, subjective exams finalize with the one-click "Auto-mark & publish" button.
