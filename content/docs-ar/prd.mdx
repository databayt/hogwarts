### نموذج تعدد المستأجرين

تعدد المستأجرين قائم على النطاقات الفرعية ويتم حله على الحافة (edge).

```text
الإنتاج:        school.databayt.org             →  /[lang]/s/school/...
التجريبي:       kingfahad.databayt.org          →  /[lang]/s/kingfahad/...
المعاينة:       tenant---branch.vercel.app      →  /[lang]/s/tenant/...
التطوير:        subdomain.localhost:3000        →  /[lang]/s/subdomain/...
المخصص:         www.theirschool.com (CNAME)     →  منصة هوجوورتس
```

تدفق الحل:

1. `proxy.ts` على الحافة يستخرج النطاق الفرعي من ترويسة المضيف.
2. النطاقات المخصصة تُربَط بالنطاقات الفرعية المعتمدة عبر Upstash Redis (`custom-domain:${host}`).
3. إعادة كتابة داخلية إلى `/[lang]/s/[subdomain]/...`؛ بادئة `/s/[subdomain]` **داخلية فقط** — لا تُستخدم أبداً في `redirect()` / `Link href` / `router.push`.
4. ترويسة `x-subdomain` تُعيَّن للمعالجات اللاحقة.
5. `getTenantContext()` يقرأ الترويسة ويحل `schoolId` لاستخدامه في كل استعلام Prisma.

### تخزين المحتوى أحادي اللغة

هوجوورتس لا يستخدم أعمدة ثنائية اللغة (`titleEn`/`titleAr`). كل المحتوى المُنشأ من المستخدم يُخزَّن بلغة المدرسة المختارة مع حقل `lang`. القراءة باللغة الأخرى تمر عبر Google Translate، مع تخزين النتائج في `TranslationCache` للقراءات المتكررة. نصوص الواجهة تعيش في القواميس (`src/components/internationalization/dictionaries/`)؛ `getDisplayText()` (`@/lib/content-display`) هو الوصول المعتمد.

ثمانية أدوار، كلها معرَّفة في `prisma/models/auth.prisma:UserRole`. التفويض يُفرَض عبر قدرات CASL (`@/lib/abilities`) وبوابات الأدوار على مستوى المسار.

| `ADMIN`      | مدير المدرسة                                           | لوحة التحكم الكاملة لكل مستأجر `/[lang]/s/[subdomain]/*`             | `requireRole("ADMIN")` |

| `STUDENT`    | الطالب المُسجَّل                                       | `school-dashboard/*` (الدرجات الخاصة، الواجبات، الحضور)              | بوابة الدور            |

## النطاق الوظيفي

الحالة المُنفَّذة من 14 ملحمة موضحة في [المنتج الأدنى](/ar/docs/mvp). خارطة الطريق التالية + الرؤية تعيش في [خارطة الطريق](/ar/docs/roadmap). وثائق التفصيل لكل ملحمة تعيش بجوارها في `content/docs-ar/`. ملفات CLAUDE.md على مستوى الكتل (`src/components/<feature>/CLAUDE.md`) توثق عقد التنفيذ لكل ميزة.

يمكن للمدرسة: قبول المتقدمين ← تسجيل الطلاب ← إدارة الفصول / الأقسام / المواد / الجدول الزمني ← تسجيل الحضور (11 وضع التقاط) ← تقييم الامتحانات + توليد بطاقات التقارير ← تحصيل الرسوم (6 بوابات دفع) ← التواصل مع أولياء الأمور (داخل التطبيق + بريد + واتساب) ← تشغيل LMS ← إدارة المكتبة + النقل — كل ذلك على منصة واحدة بعزل قائم على `schoolId`.

## المتطلبات غير الوظيفية

### الأداء

| المقياس                              | الهدف                            |
| ------------------------------------ | -------------------------------- |
| تحميل الصفحة الأولي (3G)             | أقل من 2 ثانية                   |
| التنقل اللاحق                        | أقل من 500 مللي ثانية            |
| استجابة API (المئوي 95)              | أقل من 500 مللي ثانية            |
| استعلامات قاعدة البيانات (المئوي 95) | أقل من 100 مللي ثانية            |
| TTFB                                 | أقل من 200 مللي ثانية عبر الحافة |
| نقاط Lighthouse                      | ≥ 90                             |

### الأمان

- AES-256 في الراحة، TLS 1.3 في النقل.
- Bcrypt (تكلفة 12) لكلمات المرور؛ جلسات JWT؛ مصادقة ثنائية عبر `TwoFactorToken` + `TwoFactorConfirmation`.
- حماية من القوة الغاشمة عبر تدقيق `LoginAttempt` + حدود المعدل.
- RBAC على كل مسار؛ عزل متعدد المستأجرين مفروض من جانب الخادم.
- التحقق من صحة Zod على جانب الخادم؛ منع حقن SQL وXSS.
- الاحتفاظ بـ `AuditLog` مع أنواع أحداث منظمة لكل التغييرات الحساسة.
- 11 مهمة cron يومية للصيانة (الإشعارات، التنظيف، التذكيرات، الرسوم المتأخرة).

### قابلية التوسع

- تجميع اتصالات Prisma على Neon serverless PostgreSQL.
- وظائف Vercel serverless عديمة الحالة؛ `outputFileTracingExcludes` العدوانية تُبقي حزم الوظائف تحت حد 300 MB.
- S3 + CloudFront (`cdn.databayt.org`) لتسليم الأصول.
- `experimental.serverActions.bodySizeLimit: "10mb"` لتحميل الملفات.

### إمكانية الوصول

- WCAG 2.1 AA مستهدف.
- التنقل بلوحة المفاتيح، دعم قارئ الشاشة، تباين الألوان ≥ 4.5:1.
- خصائص CSS منطقية (لا `ml-*` / `mr-*`) ليعمل RTL بدون تجاوزات.
- HTML دلالي — لا فئات Tailwind مكتوبة بشكل ثابت `text-*` / `font-*`.

## القيود

هذه القواعد تنطبق على كل مساهمة. تُفرَض بواسطة الخطافات والوكلاء وبوابات المراجعة.

- **الترخيص — SSPL-1.0.** الاستضافة الذاتية مجانية. إعادة بيع هوجوورتس كخدمة مستضافة تتطلب ترخيصاً تجارياً.
- **سلامة تعدد المستأجرين.** كل استعلام Prisma يجب أن يتضمن `schoolId` (أو يكون على نموذج عالمي معفى صراحة). اختبارات العزل في `tests/multi-tenant/` تحرس الانحدارات.
- **تخزين أحادي اللغة.** لا تُضِف أعمدة ثنائية اللغة. استخدم حقل `lang` واعتمد على `TranslationCache` للقراءات عبر اللغات.
- **CSS منطقي فقط.** استخدم `ms-*` / `me-*` / `ps-*` / `pe-*` (start / end) — أبداً `ml-*` / `mr-*` / `pl-*` / `pr-*`. RTL يجب أن يعمل بدون تجاوزات.
- **لا عناصر HTML خام في المكونات.** استخدم primitives من `ui/` shadcn. لا `<button>` / `<input>` / `<select>` خام في كود الميزة.
- **نمط المرآة.** المسارات تستورد من `src/components/<feature>/content.tsx`. أسماء ملفات معتمدة لكل ميزة: `content.tsx`, `actions.ts`, `queries.ts`, `authorization.ts`, `validation.ts`, `form.tsx`, `table.tsx`, `columns.tsx`.
- **ملف `.env` مركزي واحد.** أبداً `.env.local`, `.env.development`، إلخ.
- **المنفذ 3000 في التطوير — دائماً.**
- **`pnpm tsc --noEmit` قبل أي بناء.** بناء Vercel فيه `typescript.ignoreBuildErrors: true` لتجنب OOM، لذا فحص النوع بوابة منفصلة.
- **سلامة إعادة الكتابة الداخلية.** `/s/[subdomain]` مسار داخلي فقط. الواجهة يجب أن تستخدم المسار النسبي للمستأجر النظيف.
- **كل نص الواجهة عبر مفاتيح القاموس.** لا نصوص إنجليزية أو عربية مكتوبة بشكل ثابت في المكونات.

## API والتكاملات

### REST API v1

- عقد OpenAPI 3.0.
- مصادقة بمفتاح API مع تحديد معدل متدرج.
- أحداث Webhook: `student.enrolled`, `payment.received`, `grade.updated`، إضافة إلى سجل `webhooks.prisma:ProcessedWebhookEvent` الكامل.
- توقيع طلب HMAC-SHA256؛ إعادة المحاولة بتراجع أُسي.

### SSO (Enterprise)

- SAML 2.0 — Okta, Azure AD, OneLogin.
- OAuth 2.0 — Google Workspace, Microsoft 365.
- SCIM لتزويد المستخدمين.

### اتصالات الجهات الخارجية (مُنفَّذة أو موصولة)

Stripe (مدفوعات + اشتراكات), bankak / tap / mobile-money / bank-transfer / cash (بوابات دفع إضافية), Plaid + Dwolla (مصرفية), Resend (بريد), Twilio (SMS), WhatsApp Evolution API (مراسلة), Google Translate (ترجمة عند الطلب), AWS S3 + CloudFront (أصول), Upstash Redis (ربط النطاقات + حدود المعدل), Sentry (أخطاء), Mapbox + Leaflet (توجيه النقل), ClickView (CDN محتوى الكتالوج), Anthropic + OpenAI + Groq (ميزات الذكاء الاصطناعي).

## نهج التطوير

- إيقاع أسبوعي للسبرنت.
- رسائل commit اصطلاحية وموقعة.
- مراجعة الكود مع موافق واحد على الأقل قبل الدمج إلى `main`.
- هدف تغطية 80%+ على المسارات الحرجة.
- ميزانيات الأداء (Lighthouse ≥ 90).
- 20% من سعة السبرنت محجوزة للديون التقنية.

### هرم الاختبار

| الطبقة          | الحصة | الأداة                                                      |
| --------------- | ----- | ----------------------------------------------------------- |
| الوحدة          | 70%   | Vitest                                                      |
| التكامل         | 20%   | Vitest + Supertest                                          |
| نهاية إلى نهاية | 10%   | Playwright (المصادقة، تعدد المستأجرين، RBAC، اختبار الدخان) |

### تدفق عمل قاعدة البيانات المحلية

عمليات Prisma المُدمِّرة (`db execute`, `db push --accept-data-loss`, `migrate reset`, `DROP TABLE`, `TRUNCATE`) محظورة بواسطة الخطافات. استخدم بروتوكول **Branch-Before-Touch** على Neon لأي عملية DB محفوفة بالمخاطر. أبداً لا تشغل `pnpm db:seed` مباشرة — دائماً `pnpm db:seed:single <name>`.
